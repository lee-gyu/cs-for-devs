# 챕터 1 기획 — 자료구조와 알고리즘

[ROADMAP.md](../ROADMAP.md)의 챕터 1(`docs/ch-1/`, 문서 4편)을 집필하기 위한 상세 기획이다.
범위·경로가 ROADMAP과 어긋나면 ROADMAP을 우선한다.

## 1. 챕터의 관점

독자는 5년차 이상 개발자다. Big-O 표기법의 정의, 기본 자료구조의 개념, 정렬 알고리즘의 존재는 이미 안다.
이 챕터가 그 위에 더하는 것은 세 가지다.

1. **점근 표기와 실측 사이의 간극** — 상수 인자와 캐시 효과가 지배하는 구간을 실측으로 확인하고, 벤치마크를 신뢰할 수 있게 만드는 방법론을 세운다.
2. **메모리 배치를 근거로 한 선택** — "리스트냐 배열이냐"를 API가 아니라 메모리에 놓이는 모양과 접근 패턴으로 판단한다. 언어 런타임(V8·JVM·CPython)이 같은 이름의 자료구조를 다르게 구현하는 지점을 포함한다.
3. **문제를 알고리즘으로 환원하는 능력** — 실무 문제에서 문제 구조(최적 부분 구조, 중복 부분 문제, 교환 논증)를 읽어내 패러다임을 고르고, 그래프로 모델링해 표준 알고리즘에 환원한다.

교과서식 증명·유도는 최소화하고, 각 주제를 콘텐츠 집필 지침의 세 층위(동작 모델 → 설계 배경 → 경계 조건)로 구성한다.

## 2. 범위 결정

### 다루는 것

- 점근 표기의 정확한 의미(무엇을 보장하고 무엇을 보장하지 않는가), amortized 분석, 벤치마크 방법론
- 배열·연결 리스트·해시 테이블·트리·힙의 메모리 배치와 런타임별 구현 차이
- 분할 정복·동적 계획법·그리디·백트래킹의 선택 기준
- 그래프 표현, BFS/DFS, 최단 경로(다익스트라 중심), 위상 정렬, 사이클 감지

### 다루지 않기로 결정한 것

- **정렬 전용 문서는 두지 않는다.** ROADMAP의 4편 구성에 없다. 병합·퀵 정렬은 03(분할 정복 사례)에서, Timsort의 적응성은 01(점근 표기가 놓치는 것의 사례)에서 흡수한다.
- **균형 트리의 회전 연산 상세**(AVL·레드-블랙의 케이스별 회전)는 다루지 않는다. "왜 균형이 필요한가"와 불변식 수준까지만. 회전 자체는 실무 판단에 기여하는 바가 적다.
- **확률적 자료구조**(Bloom filter, HyperLogLog)는 02에서 "정확성을 공간과 바꾸는 선택지"로 짧게 소개하고 상세는 다루지 않는다.

## 3. 문서별 상세 기획

각 문서는 콘텐츠 집필 지침의 기본 구조(학습 목표 → 배경 → 핵심 개념 → 실무 관점 → 더 깊이 → 정리 → 확인 문제 → 참고 자료)를 따르고, 45~90분 분량으로 작성한다.

### `01-complexity-analysis.md` — 복잡도 분석과 벤치마크 방법론

챕터 전체의 기준 문서다. 여기서 세운 벤치마크 방법론을 02~04와 실습이 재사용한다.

- **핵심 질문**: O(n)이 O(n²)보다 항상 빠른가? 아니라면 언제, 왜 역전되는가? 그 역전을 어떻게 신뢰할 수 있게 측정하는가?
- **핵심 개념 뼈대**
  - 점근 표기가 보장하는 것: 입력 크기에 대한 성장률의 상한·하한. 보장하지 않는 것: 상수 인자, 낮은 차수 항, 메모리 접근 비용의 불균일성
  - 최악·평균·amortized의 구분 — 동적 배열의 push가 amortized O(1)인 이유(재할당 비용의 분산)를 계산으로 보인다
  - 상수 인자가 지배하는 구간: 같은 O(n log n)이라도 캐시 지역성에 따라 수 배 차이가 나는 사례
  - 벤치마크 방법론: JIT 워밍업, dead code elimination 회피, GC 노이즈 격리, 반복 측정과 중앙값, 실행 환경 명시. Node.js에서의 함정(V8 최적화 티어 전환)을 구체적으로 다룬다
- **관찰 예제**: 연결 리스트 순회 vs 배열 순회를 같은 N으로 실측해 점근 표기가 같아도(O(n)) 실측이 크게 갈리는 것을 보인다. "왜"는 02와 ch-7로 연결
- **경계 조건**: 마이크로벤치마크가 거짓말하는 상황(최적화 제거, 비현실적 데이터 분포), Timsort처럼 입력의 기존 순서에 적응하는 알고리즘은 최악 복잡도만으로 예측할 수 없다는 사례

### `02-data-structures-in-memory.md` — 자료구조의 메모리 배치

- **핵심 질문**: `Array`, `list`, `ArrayList`는 이름이 같거나 비슷한데 실제 메모리에는 어떻게 놓이는가? 그 배치가 언제 성능을 가르는가?
- **핵심 개념 뼈대**
  - 연속 배치 vs 포인터 추적: 배열의 임의 접근이 O(1)인 진짜 이유(주소 계산)와 연결 리스트가 실측에서 불리한 이유(포인터 역참조마다 캐시 미스 가능성)
  - 동적 배열: 성장 계수(1.5 vs 2)와 재할당, capacity와 length의 구분
  - 해시 테이블: 체이닝 vs 오픈 어드레싱, load factor와 리사이징, 해시 함수의 품질과 HashDoS, 그리고 SipHash 채택 배경(Python·Rust)
  - 트리: BST의 퇴화와 균형의 필요성, 힙의 배열 표현(포인터 없는 트리)
  - 런타임별 구현 차이: V8의 elements kinds(packed/holey, SMI/double 전환)와 hidden class, CPython의 list(포인터 배열)와 compact dict(3.6+), JVM의 HashMap(체이닝 + 트리화 임계값 8)과 객체 헤더 오버헤드
- **관찰 예제**: V8에서 배열에 구멍을 만들거나 타입을 섞을 때 elements kind가 전환되는 것을 `%DebugPrint`(또는 `d8`) 혹은 벤치마크 차이로 관찰
- **경계 조건**: "JS 배열은 배열이 아니게 되는 순간"(holey·dictionary mode 전환), 해시맵이 O(1)이길 멈추는 순간(리사이징 스파이크, 최악 입력), 객체 수백만 개를 다룰 때 포인터 배열 vs TypedArray/SoA 배치의 차이

### `03-algorithm-design-paradigms.md` — 알고리즘 설계 패러다임

- **핵심 질문**: 새 문제를 만났을 때 어떤 패러다임을 시도할지 무엇을 보고 결정하는가?
- **핵심 개념 뼈대** — 각 패러다임을 "문제 구조의 신호 → 대표 사례 → 실패하는 조건" 순서로 통일해 서술한다
  - 분할 정복: 부분 문제가 독립일 때. 병합·퀵 정렬, 이진 탐색. 마스터 정리는 결과 위주로. (분산 처리의 분할-집계로 이어지는 발상은 ch-17로 연결)
  - 동적 계획법: 최적 부분 구조 + 중복 부분 문제. LCS/편집 거리(diff 도구의 계보), 메모이제이션 vs 테이블. "DP = 어려운 것"이 아니라 "지수적 탐색 공간에 중복이 있으면 다항으로 접힌다"는 관점
  - 그리디: 교환 논증으로 국소 선택의 안전성이 증명될 때. 인터벌 스케줄링, 다익스트라(04와 연결), 허프만(ch-3로 연결). 그리디가 틀리는 사례(동전 교환의 반례)를 반드시 짝으로
  - 백트래킹: 제약 충족과 가지치기. 정규식 백트래킹 엔진이 지수 시간으로 폭발하는 구조(ch-2 ReDoS의 예고편)
- **실무 관점**: 실무 문제를 패러다임으로 환원하는 연습 — 재시도 정책, 자원 할당, diff, 스케줄링 같은 문제에서 구조 신호를 읽는 사례 2~3개
- **경계 조건**: 최적 부분 구조가 없는 문제에 DP를 강요할 때, 탐색 공간에 중복이 없어 메모이제이션이 무력한 경우, NP-난해 문제의 냄새(ch-2 04로 예고)

### `04-graph-theory-and-algorithms.md` — 그래프 모델링과 알고리즘

- **핵심 질문**: 눈앞의 실무 문제가 그래프 문제임을 어떻게 알아보고, 어떤 표준 알고리즘으로 환원하는가?
- **핵심 개념 뼈대**
  - 그래프 모델링: 정점·간선으로 무엇을 삼을지가 문제 해결의 절반. 의존성 그래프, 상태 전이, wait-for 그래프 같은 사례로 모델링 감각을 먼저 세운다
  - 표현: 인접 리스트 vs 인접 행렬 — 02의 메모리 배치 논의를 그대로 적용(희소 그래프에서 행렬이 낭비인 이유, CSR 같은 압축 표현은 이름만)
  - 탐색: BFS(최단 홉, 레벨 구조)와 DFS(경로·사이클·연결 요소), 반복 vs 재귀(스택 깊이 한계)
  - 위상 정렬과 사이클 감지: 패키지 매니저의 의존성 해석, 빌드 순서, 순환 의존 오류의 정체
  - 최단 경로: 다익스트라(그리디 + 우선순위 큐, 03·02와 연결)와 음수 가중치에서 무너지는 조건, 벨만-포드는 존재와 용도만
- **실무 환원 사례** (각 사례를 "문제 → 그래프 모델 → 표준 알고리즘" 3단으로): pnpm 의존성 해석(위상 정렬), 데드락 감지(wait-for 그래프의 사이클), GC mark 단계(도달성 = 그래프 탐색, ch-6로 연결)
- **경계 조건**: V·E가 커질 때 표현 선택이 갈리는 지점, 다익스트라가 성립하지 않는 조건(음수 간선), NP-난해 그래프 문제(TSP, 그래프 색칠)는 이름과 냄새만 — 상세는 ch-2로

## 4. 문서 간 의존 관계

```
01 벤치마크 방법론 ──▶ 02 실측 예제 ──▶ 04 표현 선택 근거
        │                    │
        └──▶ 03 패러다임 비교 ┴──▶ 04 다익스트라(그리디+힙)
```

- 01에서 확립한 벤치마크 규칙(워밍업·반복·중앙값·환경 명시)은 02~04의 모든 실측 예제와 실습 과제가 참조한다. 각 문서에서 방법론을 재설명하지 않고 01로 링크한다.
- 02의 힙(배열 표현)은 04의 다익스트라 우선순위 큐가 사용한다.
- 03의 그리디·백트래킹은 04(다익스트라)와 ch-2(정규식 엔진)의 선수 지식이다.
- 챕터 밖 연결은 §2의 위임 표를 따른다. ch-7·ch-11 등 아직 없는 문서로의 링크는 집필 시점에 존재하지 않으므로, 본문에는 "챕터 N에서 다룬다"라고 쓰고 링크는 해당 챕터 집필 시 걸도록 TODO 주석을 남기지 않는다(문장으로만 위임).

## 5. 실습 과제 기획 (`exercises/ch-1/`)

ROADMAP 산출물: "핵심 자료구조를 직접 구현하고, 점근 복잡도 예측과 실측(캐시 효과 포함)의 차이를 벤치마크 리포트로 정리한다."

- **환경**: TypeScript + Node.js 24 (저장소 도구 체인과 동일, `mise.toml` 기준). 실습 코드는 `exercises/ch-1/` 아래 pnpm 워크스페이스 패키지로 둔다.
- **구현 대상**
  1. 동적 배열 — 성장 계수 1.5 vs 2를 바꿔가며 amortized 비용 실측
  2. 단일 연결 리스트 — 배열과의 순회·삽입 비교 대상
  3. 해시 맵 — 체이닝과 오픈 어드레싱 두 방식, load factor 조절 가능하게
  4. 이진 힙 — 우선순위 큐로서 정렬 배열과 비교
  5. (선택) 인접 리스트 그래프 + 다익스트라 + 위상 정렬
- **측정 시나리오** (예측 복잡도 표를 먼저 쓰고 실측과 대조)
  - 순차 순회: 배열 vs 연결 리스트 (같은 O(n), 캐시 효과로 갈리는 실측)
  - 임의 위치 삽입: 배열 vs 연결 리스트 (Stroustrup의 "연결 리스트를 피하라" 실험 재현)
  - 해시 맵: load factor에 따른 조회·삽입 곡선, 리사이징 순간의 지연 스파이크
  - 동적 배열: 성장 계수별 재할당 횟수와 총 복사량
- **완료 기준 초안**: 각 구조의 단위 테스트 통과, 한 명령으로 재현되는 벤치 스크립트, 리포트에 "예측과 실측이 갈린 지점과 그 원인" 서술 포함. 상세 기준은 [실습 문서](../exercises/ch-1/README.md)에서 확정했다.
- **도구 확정** (실습 문서 작성 시 결정): 테스트 러너는 `node:test`(내장, 의존성 추가 없음), 벤치마크 러너는 mitata(워밍업·통계·환경 출력이 01 방법론과 부합). 집계 러너가 뭉개는 연산 단위 스파이크(리사이징 등)는 자체 계측으로 병행한다. 근거는 [실습 문서](../exercises/ch-1/README.md)의 도구 선정 절에 있다.

## 6. 조사 노트 — 1차 자료 후보

집필 시 아래를 우선 확인하고, 수치·버전·스펙 표현은 인용 시점에 재검증한다.

- **알고리즘 일반**: CLRS 4판(점근 표기·amortized·그래프의 표준 서술), Sedgewick & Wayne *Algorithms* (실측 감각)
- **V8 / JS**: v8.dev 블로그 "Elements kinds in V8", "Getting garbage collection for free", "Array sorting"(Timsort 채택), ECMA-262의 Map/Set 복잡도 요구("average access sublinear")
- **CPython**: `listobject.c`·`dictobject.c` 소스와 dict 재설계 논의(compact dict, Python 3.6), `sys.getsizeof` 관찰
- **JVM**: OpenJDK `HashMap.java` 주석(트리화 임계값 8의 근거 — 포아송 분포 계산이 주석에 있음)
- **해시와 HashDoS**: SipHash 논문(Aumasson & Bernstein), Python·Rust의 기본 해시 채택 배경
- **캐시 효과 수치**: "Latency Numbers Every Programmer Should Know", Drepper *What Every Programmer Should Know About Memory* (ch-7의 주 자료이므로 챕터 1에서는 수치 인용 수준으로만)
- **연결 리스트 vs 배열 실측**: Stroustrup GoingNative 2012 "Why you should avoid Linked Lists"
- **벤치마크 방법론**: Brendan Gregg *Systems Performance*의 방법론 장, V8의 마이크로벤치마크 함정 관련 공식 문서

### 통념 검증 목록 (본문에서 정면으로 다룰 것)

- "해시 테이블은 O(1)이니 항상 Map을 쓰면 된다" → 리사이징 스파이크, 순회 비용, 작은 N에서 배열 선형 탐색이 더 빠른 구간
- "연결 리스트 중간 삽입은 O(1)이라 배열보다 빠르다" → 삽입 위치 탐색 O(n) + 캐시 미스로 실측 역전
- "O(n log n) 정렬이니 어떤 정렬이든 같다" → Timsort의 적응성, 기수 정렬처럼 비교 기반 하한 밖의 선택지
- "재귀 DFS면 충분하다" → 스택 깊이 한계(런타임별 기본 스택 크기)
