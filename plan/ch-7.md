# 챕터 7 기획 — 컴퓨터 구조

[ROADMAP.md](../ROADMAP.md)의 챕터 7(`docs/ch-7/`, 인트로 1편과 본문 2편)을 집필하기 위한 상세 기획이다.
범위·경로가 ROADMAP과 어긋나면 ROADMAP을 우선한다.

## 1. 챕터의 관점

독자는 5년차 이상 개발자다. 어셈블리를 매일 쓰지는 않지만, "캐시 친화적으로 짜라", "분기를 줄여라" 같은 조언을 들어 봤고 ch-1에서 점근 표기가 같은 코드의 실측이 크게 갈리는 것을 관찰했다.
이 챕터는 ISA와 마이크로아키텍처를 백과사전식으로 나열하지 않고, **"측정 가능한 증거로 코드의 성능 저하 원인을 진단하는 하드웨어 모델"**을 세우는 데 집중한다. 더하는 것은 세 가지다.

1. **순차 실행이라는 착시 걷어내기** — 코드는 한 줄씩 실행되는 것처럼 보이지만 실제로는 파이프라인·슈퍼스칼라·비순차 실행·분기 예측 위에서 흐른다. 이 간극이 같은 명령어 수의 코드를 수 배 차이로 가른다.
2. **메모리가 병목이라는 사실을 수치로 갖기** — 캐시 계층·캐시 라인·프리페처를 근거로 데이터 배치와 접근 패턴을 판단한다. ch-1에서 현상으로 관찰한 캐시 효과의 원인을 여기서 규명한다.
3. **멀티코어의 숨은 비용** — 캐시 일관성 프로토콜이 코어 간에 만드는 트래픽과 false sharing을 이해하고, "스레드를 늘렸는데 느려지는" 사례를 진단한다.

두 본문은 다음 네 질문을 공통 진단 프레임으로 사용한다.

1. **명령이 충분히 공급되는가** — 프론트엔드와 분기 예측이 실행 파이프라인을 비우는지 확인한다.
2. **명령이 서로 기다리는가** — 데이터 의존성과 부족한 ILP가 실행을 직렬화하는지 확인한다.
3. **CPU가 데이터를 기다리는가** — 캐시·DRAM·프리페처·메모리 수준 병렬성(memory-level parallelism, MLP)이 병목인지 확인한다.
4. **코어끼리 방해하는가** — 공유 캐시 라인과 일관성 트래픽이 병렬 확장을 막는지 확인한다.

모든 성능 서사는 **현상 → 경쟁 가설 → 최소 대조 실험 → 시간·카운터·생성 코드 관찰 → 개선 → 경계 조건** 순으로 전개한다. IPC나 단일 하드웨어 카운터는 결론이 아니라 다음 실험을 고르는 출발점으로 사용한다. 관찰한 상관관계와 원인을 입증하는 증거를 구분한다.

## 2. 범위 결정

### 다루는 것

- ISA와 마이크로아키텍처의 구분(계약 vs 구현), 명령어 실행의 실제 경로
- 파이프라인과 해저드(데이터·제어), 분기 예측, 슈퍼스칼라와 ILP, 비순차 실행, 투기 실행(Spectre는 경계 조건 사례로 구조만)
- cycles·instructions·IPC·branch-misses를 함께 읽는 병목 판별의 기초. IPC는 분기 실패·의존 체인·메모리 대기 중 원인을 좁히기 위한 출발점으로 취급
- 캐시 계층(L1/L2/L3)과 대표 지연 수치, 캐시 라인, 공간·시간 지역성, 연관도(associativity)와 conflict miss, 하드웨어 프리페처
- 메모리 지연 시간과 대역폭의 구분, 여러 독립 미스를 겹쳐 숨기는 MLP와 그 한계
- 캐시 미스 분류(compulsory·capacity·conflict·coherence)와 각 미스를 줄이는 데이터 배치(행/열 순회, AoS vs SoA)
- 멀티코어 캐시 일관성(MESI 수준의 상태 모델), false sharing, 일관성 트래픽이 확장성을 깎는 구조와 이를 입증하는 증거의 한계

### 다루지 않기로 결정한 것

- **어셈블리 작성 교육은 하지 않는다.** 컴파일러 출력(Compiler Explorer 수준)을 읽어 분기·cmov·벡터화 여부를 확인하는 정도만 요구한다.
- **SIMD·벡터화 상세는 다루지 않는다.** ILP의 연장선에서 존재와 효과만 소개하고, intrinsics·자동 벡터화 조건은 범위 밖이다.
- **GPU·가속기 아키텍처는 다루지 않는다.** 처리량 지향 설계라는 대비만 한 문단으로 언급한다.
- **NUMA는 이름과 증상 수준까지만.** 서버급 다중 소켓 튜닝은 커리큘럼 목표를 넘는다.
- **전력·클럭(터보 부스트, 스로틀링)은 측정 노이즈 요인으로만 언급한다.** 벤치마크 시 주의사항으로 흡수한다.
- **Spectre·Meltdown의 공격 기법 상세는 다루지 않는다.** 투기 실행이 마이크로아키텍처 상태에 흔적을 남긴다는 구조적 원인까지만.

## 3. 문서별 상세 기획

본문 2편은 콘텐츠 집필 지침의 기본 구조(학습 목표 → 배경 → 핵심 개념 → 실무 관점 → 더 깊이 → 정리 → 확인 문제 → 참고 자료)를 따르고, 45~90분 분량으로 작성한다. 00 인트로는 10~15분 분량의 오리엔테이션으로 구성한다.

### `00-introduction.md` — 같은 코드가 같은 속도로 실행되지 않는 이유

- **핵심 질문**: 알고리즘과 명령어 수가 비슷한 코드의 성능이 CPU와 데이터 배치에 따라 수 배씩 달라지는 이유는 무엇인가?
- **도입 시나리오**: 같은 알고리즘에서 분기를 제거했지만 빨라지지 않고, 스레드를 늘리자 오히려 느려진 상황을 제시한다. 처음부터 원인을 단정하지 않고 명령 공급·데이터 의존·메모리 대기·코어 간 간섭이라는 네 경쟁 가설을 세운다.
- 소스 코드의 순차 실행이라는 착시와 실제 하드웨어의 파이프라인·캐시·멀티코어 비용 사이의 간극을 문제의식으로 세운다.
- 빠른 코드를 만드는 팁보다 측정 가능한 가설을 세우고 반증하는 태도를 챕터의 핵심 관점으로 제시한다.
- 01은 명령 공급과 데이터 의존, 02는 메모리 대기와 코어 간 간섭을 진단한다는 학습 지도를 제시하고 상세 원리는 본문에 위임한다.

### `01-cpu-and-pipelines.md` — CPU는 코드를 순서대로 실행하지 않는다

- **핵심 질문**: 명령어 수가 거의 같은 두 코드의 실행 시간이 왜 수 배 차이 나는가?
- **핵심 개념 뼈대**
  - ISA vs 마이크로아키텍처: 같은 x86-64 코드가 세대별 CPU에서 다르게 수행되는 이유. ISA는 계약, 마이크로아키텍처는 구현
  - 파이프라인: 명령어를 단계로 나눠 겹치는 구조, 처리량과 지연의 구분, 해저드(앞 결과를 기다리는 데이터 해저드, 분기의 제어 해저드)
  - 분기 예측: 파이프라인을 채우기 위한 투기, 예측 실패 시 파이프라인 플러시 비용. "정렬된 배열의 조건 분기가 빠른" 고전 사례를 여기서 설명
  - 슈퍼스칼라와 비순차 실행: 독립 명령어의 병렬 발행, 의존 체인이 ILP를 제한하는 구조, 레지스터 리네이밍은 개념 수준
  - 낮은 IPC의 경쟁 원인: 프론트엔드 정체, 분기 예측 실패, 긴 의존 체인을 구분하고 다음에 볼 증거를 선택하는 법
  - 투기 실행의 경계: 예측 실행이 캐시에 남기는 흔적 — Spectre가 드러낸 "성능 최적화 = 관찰 가능한 부작용"이라는 구조를 짧은 경계 사례로만 취급
  - 측정 어휘: cycles, instructions, IPC, branch-misses — 각 카운터가 말하는 것과 말하지 못하는 것
- **실무 연결**: 예측 가능한 분기 vs 예측 불가능한 분기의 구분(데이터 의존 분기), branchless 변환(조건을 산술·cmov로)이 이기는 조건과 지는 조건, 긴 포인터 추적 의존 체인(연결 리스트 순회)이 비순차 실행으로도 숨겨지지 않는 이유(ch-1 실측의 원인 절반)
- **관찰 예제**: (1) 정렬 vs 무작위 배열의 조건 합산과 branchless 버전을 비교해 분기 실패를 진단하고, (2) 독립 누산기와 단일 누산기의 의존 체인을 비교해 ILP 제한을 분리한다. 시간과 카운터를 보기 전에 컴파일러가 분기를 `cmov`나 벡터 명령으로 바꾸지 않았는지 생성 코드를 확인한다.
- **경계 조건**: 마이크로벤치마크가 실서비스를 대표하지 않는 문제, 분기 예측기가 학습하는 패턴 길이의 한계, 예측이 잘 되는 분기와 양쪽 계산 비용 때문에 branchless가 지는 경우, 컴파일러 최적화가 실험의 독립 변수를 없애는 경우

### `02-memory-hierarchy.md` — 메모리 계층과 캐시 일관성

ch-1이 예고한 "캐시 효과의 원인"을 회수하는 문서이자 챕터 실습의 직접 기반이다.

- **핵심 질문**: 같은 O(n) 순회 두 개의 실측이 왜 10배 갈리는가? 스레드를 늘렸는데 왜 더 느려지는가?
- **핵심 개념 뼈대**
  - 왜 계층인가: 빠른 메모리는 작고 비싸다 — 지역성이라는 베팅 위에 세운 구조. 대표 지연 수치(L1/L2/L3/DRAM)를 자릿수 감각으로 제시
  - 지연 시간 vs 대역폭: 단일 의존 로드가 기다리는 시간과 많은 데이터를 지속해서 옮기는 처리량을 구분
  - 캐시 라인: 전송 단위가 만드는 효과 — 인접 데이터의 공짜 로드, 낭비되는 로드. 행/열 순회 차이의 직접 원인
  - 매핑과 연관도: set-associative 구조, conflict miss가 특정 stride에서 튀는 이유(2의 거듭제곱 stride 함정)
  - 하드웨어 프리페처: 순차·stride 패턴 감지, 포인터 추적이 프리페처를 무력화하는 구조(ch-1 연결 리스트 실측의 나머지 원인)
  - MLP: 비순차 실행이 여러 독립 캐시 미스를 겹쳐 숨기는 구조와 의존적인 포인터 추적에서는 이것이 불가능한 이유
  - 캐시 미스 분류: compulsory·capacity·conflict, 그리고 멀티코어의 coherence miss. TLB miss는 ch-8로 연결
  - 데이터 배치 설계: AoS vs SoA, 핫/콜드 필드 분리, 패딩과 정렬
  - 캐시 일관성: MESI의 상태 암기보다 쓰기가 다른 코어의 캐시 라인을 무효화하는 구조와 비용에 집중
  - false sharing: 서로 다른 변수가 같은 라인에 있을 때 생기는 무효화 핑퐁 — 논리적 공유가 없어도 물리적 공유가 생기는 함정
- **실무 연결**: "메모리를 더 쓰더라도 접근 패턴을 연속으로"가 이기는 조건, 멀티스레드 카운터·통계 수집에서 false sharing이 생기는 전형적 코드, 패딩·per-core 집계로 고치는 패턴
- **관찰 예제**: (1) working set 크기를 늘리며 원소당 접근 시간 곡선에서 캐시 계층 경계를 관찰하고, (2) worker 수를 늘리며 인접 카운터와 패딩된 카운터의 처리량·확장성 곡선을 비교한다. stride 기반 캐시 라인 추정은 선택 확장으로 둔다.
- **증거 해석**: 일반 `cache-misses`는 false sharing의 직접 증거가 아닐 수 있다. 시간과 코어 수별 확장성 곡선을 기본 증거로 삼고, 지원되는 CPU에서 coherence 관련 이벤트를 추가 증거로 사용한다.
- **경계 조건**: 데이터가 L1에 다 들어가는 크기에서는 배치 최적화가 무의미한 지점, 프리페처가 오히려 대역폭을 낭비하는 패턴, 독립 미스로 지연을 숨길 수 있는 구간, 일관성 트래픽 때문에 코어 수 증가가 역효과가 되는 공유 쓰기 패턴

- 01의 파이프라인·비순차 실행 모델은 02에서 "메모리 지연을 왜 다 숨길 수 없는가"(의존 체인, MLP 한계)의 전제가 된다.
- 02는 ch-1 01·02가 현상으로 남겨 둔 캐시 효과의 원인을 회수한다. 집필 시 ch-1 해당 문서에 상대 링크를 건다(ch-1은 이미 존재하므로 양방향 링크 가능).
- 02의 캐시 오염·TLB 개념은 ch-8(컨텍스트 스위치 간접 비용, 가상 메모리)이 위임받는다. ch-8 문서가 이미 존재하므로 집필 시 상대 링크로 연결한다.

## 5. 실습 과제 기획 (`exercises/ch-7/`)

ROADMAP 산출물: "행/열 순회, 분기 예측 실패, false sharing을 재현하는 마이크로벤치마크를 작성하고 하드웨어 카운터로 원인을 확인한다."

### 환경 결정

- **필수 트랙**: TypeScript + Node.js 24(`TypedArray`, `worker_threads` + `SharedArrayBuffer`)로 시간과 확장성 곡선을 관찰한다. `exercises/ch-7/` 아래 pnpm 패키지로 구성하고 ch-1의 벤치마크 방법론(워밍업·반복·중앙값·환경 명시)을 재사용한다. macOS와 Linux에서 모두 완료할 수 있어야 한다.
- **심화 트랙**: 각 시나리오의 최소 C11 대조 구현을 `gcc -O2`와 단일 `Makefile`로 빌드한다. `perf stat`은 PMU 접근이 가능한 Linux 실기 또는 VM에서 실행한다. macOS 위 일반 Docker VM은 PMU 접근을 보장하지 않으므로 기준 환경으로 삼지 않는다.
- **카운터 구성**: cycles·instructions·branches·branch-misses처럼 널리 제공되는 이벤트를 공통 집합으로 두고, 캐시·일관성 이벤트는 CPU가 지원할 때만 추가하는 선택 집합으로 분리한다. 특정 이벤트가 없다는 이유로 실습 전체가 실패하지 않게 한다.
- **트랙의 관계**: Node 트랙은 현상을 재현하는 필수 과제이고 C+`perf` 트랙은 JIT·GC·컴파일러 변환을 통제하며 원인을 더 엄밀히 확인하는 심화 과제다. 두 구현의 절대 수치 일치를 요구하지 않고 결과의 방향과 차이의 원인을 비교한다.

### 측정 시나리오

ROADMAP에 명시된 세 현상을 필수 산출물로 삼는다. 각 시나리오는 **현상 → 경쟁 가설 → 최소 대조 실험 → 증거 해석 → 개선 → 반례** 순으로 작성한다.

1. **행/열 순회**: 2차원 배열(1차원 `TypedArray`)을 행 우선·열 우선으로 합산한다. 같은 O(n²)에서 캐시 라인 활용률이 만드는 차이를 측정하되 프리페처·working set 크기를 경쟁 원인으로 함께 검토한다.
2. **분기 예측**: 예측 가능한 데이터와 무작위 데이터의 조건 합산, branchless 버전을 비교한다. branch-misses와 IPC를 보기 전에 JIT 또는 컴파일러가 분기를 `cmov`·벡터 명령으로 제거하지 않았는지 확인한다.
3. **false sharing**: worker 2~4개가 인접 카운터와 캐시 라인 크기로 패딩된 카운터를 각각 갱신한다. 시간과 worker 수별 확장성 곡선을 기본 증거로 사용하고, C 심화 트랙에서는 지원되는 coherence 이벤트를 추가로 확인한다.

working set 크기별 원소당 접근 시간 곡선은 02 본문의 관찰 예제로 둔다. stride 기반 캐시 라인 추정은 실습의 선택 확장으로 두어 필수 산출물의 범위를 늘리지 않는다.

### 리포트 요구사항과 완료 기준

- 시나리오마다 결과를 보기 전에 경쟁 가설을 둘 이상 쓰고 각 가설의 반증 조건을 명시한다.
- 한 변수만 바꾼 최소 대조군을 만들고, 시간 결과와 카운터가 각각 말하는 것과 말하지 못하는 것을 구분해 해석한다.
- 필수 Node 트랙은 한 명령으로 세 벤치마크를 재현한다. C 심화 트랙은 단일 `Makefile`과 시나리오별 `perf stat` 명령을 제공한다.
- CPU 모델, 코어·스레드 토폴로지, 캐시 크기·라인 크기, OS·커널, Node 또는 컴파일러 버전, 코어 고정 여부를 기록한다. 캐시 파라미터 확인 명령(`sysctl`/`lscpu`)을 절차에 포함한다.
- Node 시간 결과와 C 카운터 결과의 방향이 다르면 JIT, 자동 벡터화, dead-code elimination, GC, 프리페처, CPU 토폴로지 순으로 확인하고 조사 결과를 남긴다.
- 예상과 다른 결과는 실패로 처리하지 않는다. 가설을 반증한 증거와 다음 대조 실험을 제시하면 진단 목표를 달성한 것으로 본다.
- 기본 트랙 완료 기준은 세 현상의 Node 재현과 증거 해석이다. ROADMAP의 "하드웨어 카운터로 원인을 확인한다"는 최종 산출물까지 충족하려면 C+`perf` 심화 트랙을 완료해야 하며, 두 완료 수준을 실습 문서에서 명확히 구분한다.

## 6. 조사 노트 — 1차 자료 후보

집필 시 아래를 우선 확인하고, 지연 수치·캐시 파라미터처럼 하드웨어 세대에 의존하는 값은 "자릿수 감각 + 측정으로 확인" 형태로 제시한다.

- **아키텍처 일반**: Hennessy & Patterson *Computer Architecture: A Quantitative Approach*(파이프라인·캐시·일관성의 표준 서술), Patterson & Hennessy *Computer Organization and Design*(입문 층위 보조)
- **메모리 계층**: Ulrich Drepper *What Every Programmer Should Know About Memory*(캐시 구조·측정의 1차 참조 — ch-1에서 수치 인용으로 예고한 자료), Intel Optimization Reference Manual·Intel SDM(x86 캐시·프리페처의 공식 서술)
- **마이크로아키텍처 실측**: Agner Fog의 마이크로아키텍처 매뉴얼(비순차 실행·분기 예측기의 실측 기반 서술), perf 공식 문서와 Brendan Gregg *Systems Performance*(카운터 해석)
- **일관성**: Sorin, Hill, Wood *A Primer on Memory Consistency and Cache Coherence*(MESI와 일관성·순서의 구분)
- **투기 실행 경계**: Spectre 논문(Kocher et al., 2019)·Meltdown 논문 — 구조적 원인 서술의 근거로만
- **보조 검증**: Igor Ostrovsky "Gallery of Processor Cache Effects", Stroustrup 연결 리스트 실측(ch-1과 공유) — 블로그류는 자체 재현 후 보조로만 인용

### 통념 검증 목록 (본문에서 정면으로 다룰 것)

- "CPU는 코드를 한 줄씩 순서대로 실행한다" → 파이프라인·비순차·투기 실행. 순차성은 결과에 대한 약속이지 실행 방식이 아니다.
- "메모리 접근 비용은 어디든 같다" → L1과 DRAM은 자릿수가 다르다. RAM(random access)이라는 이름이 균일 비용을 보장하지 않는다.
- "명령어 수가 적은 코드가 빠르다" → IPC가 지배하는 구간에서는 명령어가 많아도 의존이 얕은 쪽이 이긴다(branchless가 이기는 이유이자, 지는 이유).
- "스레드를 늘리면 코어 수만큼 빨라진다" → 공유 쓰기의 일관성 트래픽과 false sharing이 확장성을 깎는다. 논리적 공유가 없어도 발생한다.
- "캐시는 하드웨어 문제라 코드로 어쩔 수 없다" → 배치(AoS/SoA)·순회 순서·패딩은 전부 코드 결정이다.
- "분기 예측 실패가 무섭니 분기를 모두 제거하라" → 예측이 잘 되는 분기는 거의 공짜다. 데이터 의존적·비대칭 비용 분기만 문제가 된다.
- "IPC가 낮으면 메모리 병목이다" → 낮은 IPC는 분기 실패·긴 의존 체인·프론트엔드 정체로도 발생한다. 원인별 대조 실험과 추가 카운터가 필요하다.
- "cache-misses가 많으면 false sharing이다" → 일반 캐시 미스는 여러 원인을 합친 결과다. 패딩 전후와 코어 수별 확장성, 가능하면 coherence 이벤트로 증거를 보강해야 한다.

## 7. 작성 순서와 검증 계획

1. 독자는 `00 → 01 → 02` 순서로 읽는다. 00이 문제의식과 학습 지도를 세우고, 파이프라인·투기 모델(01)이 "메모리 지연이 왜 치명적인가"(02)의 전제가 된다.
2. 모든 본문 사례와 실습은 현상 → 경쟁 가설 → 최소 대조 실험 → 증거 → 개선 → 경계 조건의 진단 루프를 따른다. IPC나 단일 이벤트만으로 원인을 확정하는 서술은 허용하지 않는다.
3. 모든 Node 예제는 Node.js 24에서 직접 실행해 관찰 결과를 확인한 뒤 수치를 기재한다. 카운터 수치는 PMU 접근이 가능한 Linux 실기 또는 VM에서 C 대조 구현으로 확인하고, 실행 환경(CPU 모델·캐시 파라미터·커널 버전)을 명시한다.
4. 분기 예제는 생성 코드를 확인해 JIT·컴파일러가 분기, `cmov`, 벡터 명령 중 무엇을 만들었는지 기록한다. false sharing은 일반 `cache-misses`만으로 입증했다고 서술하지 않는다.
5. 하드웨어 의존 수치(지연, 라인 크기)는 특정 값 단정 대신 대표 자릿수 + 독자 확인 절차(`lscpu`, `sysctl hw`)로 제시한다.
6. 확인 문제는 용어 암기보다 주어진 시간·IPC·branch-misses·확장성 결과에서 경쟁 가설을 세우고 다음 대조 실험을 선택하게 만든다.
7. 02 집필 시 ch-1 01·02의 캐시 관련 예고 문장과 ch-8의 TLB·컨텍스트 스위치 설명에 상대 링크를 걸어 연결을 완성한다. 기존 본문 수정은 링크 추가 수준으로 한정한다.
8. `docs/ch-7/`을 처음 만들 때 `docs/.vitepress/navigation.ts`의 `PHASE_LABELS`에 챕터 7 레이블을 등록하고 빌드로 nav·sidebar 노출을 확인한다(저장소 운영 지침 적용).
9. 문서 수나 완료 상태가 실제로 바뀔 때만 `PROGRESS.md`를 갱신한다. 실습 문서(`exercises/ch-7/`)는 인트로와 본문 2편 완료 후 §5 기획을 기준으로 작성한다.
