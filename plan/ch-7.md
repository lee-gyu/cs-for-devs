# 챕터 7 기획 — 컴퓨터 구조

[ROADMAP.md](../ROADMAP.md)의 챕터 7(`docs/ch-7/`, 문서 2편)을 집필하기 위한 상세 기획이다.
범위·경로가 ROADMAP과 어긋나면 ROADMAP을 우선한다.

## 1. 챕터의 관점

독자는 5년차 이상 개발자다. 어셈블리를 매일 쓰지는 않지만, "캐시 친화적으로 짜라", "분기를 줄여라" 같은 조언을 들어 봤고 ch-1에서 점근 표기가 같은 코드의 실측이 크게 갈리는 것을 관찰했다.
이 챕터는 ISA와 마이크로아키텍처를 백과사전식으로 나열하지 않고, **"내 코드의 성능을 예측하고 설명하는 하드웨어 모델"**을 세우는 데 집중한다. 더하는 것은 세 가지다.

1. **순차 실행이라는 착시 걷어내기** — 코드는 한 줄씩 실행되는 것처럼 보이지만 실제로는 파이프라인·슈퍼스칼라·비순차 실행·분기 예측 위에서 흐른다. 이 간극이 같은 명령어 수의 코드를 수 배 차이로 가른다.
2. **메모리가 병목이라는 사실을 수치로 갖기** — 캐시 계층·캐시 라인·프리페처를 근거로 데이터 배치와 접근 패턴을 판단한다. ch-1에서 현상으로 관찰한 캐시 효과의 원인을 여기서 규명한다.
3. **멀티코어의 숨은 비용** — 캐시 일관성 프로토콜이 코어 간에 만드는 트래픽과 false sharing을 이해하고, "스레드를 늘렸는데 느려지는" 사례를 진단한다.

두 문서 모두 "하드웨어 카운터로 확인 가능한 주장"을 원칙으로 삼는다. 성능 서사는 반드시 측정 절차와 함께 제시한다.

## 2. 범위 결정

### 다루는 것

- ISA와 마이크로아키텍처의 구분(계약 vs 구현), 명령어 실행의 실제 경로
- 파이프라인과 해저드(데이터·제어), 분기 예측, 슈퍼스칼라와 ILP, 비순차 실행, 투기 실행(Spectre는 경계 조건 사례로 구조만)
- IPC(instructions per cycle)를 중심으로 한 프론트엔드/백엔드 병목 판별의 기초
- 캐시 계층(L1/L2/L3)과 대표 지연 수치, 캐시 라인, 공간·시간 지역성, 연관도(associativity)와 conflict miss, 하드웨어 프리페처
- 캐시 미스 분류(compulsory·capacity·conflict·coherence)와 각 미스를 줄이는 데이터 배치(행/열 순회, AoS vs SoA)
- 멀티코어 캐시 일관성(MESI 수준의 상태 모델), false sharing, 일관성 트래픽이 확장성을 깎는 구조

### 다루지 않기로 결정한 것

- **어셈블리 작성 교육은 하지 않는다.** 컴파일러 출력(Compiler Explorer 수준)을 읽어 분기·cmov·벡터화 여부를 확인하는 정도만 요구한다.
- **SIMD·벡터화 상세는 다루지 않는다.** ILP의 연장선에서 존재와 효과만 소개하고, intrinsics·자동 벡터화 조건은 범위 밖이다.
- **GPU·가속기 아키텍처는 다루지 않는다.** 처리량 지향 설계라는 대비만 한 문단으로 언급한다.
- **NUMA는 이름과 증상 수준까지만.** 서버급 다중 소켓 튜닝은 커리큘럼 목표를 넘는다.
- **전력·클럭(터보 부스트, 스로틀링)은 측정 노이즈 요인으로만 언급한다.** 벤치마크 시 주의사항으로 흡수한다.
- **Spectre·Meltdown의 공격 기법 상세는 다루지 않는다.** 투기 실행이 마이크로아키텍처 상태에 흔적을 남긴다는 구조적 원인까지만.

## 3. 문서별 상세 기획

각 문서는 콘텐츠 집필 지침의 기본 구조(학습 목표 → 배경 → 핵심 개념 → 실무 관점 → 더 깊이 → 정리 → 확인 문제 → 참고 자료)를 따르고, 45~90분 분량으로 작성한다.

### `01-cpu-and-pipelines.md` — CPU는 코드를 순서대로 실행하지 않는다

- **핵심 질문**: 명령어 수가 거의 같은 두 코드의 실행 시간이 왜 수 배 차이 나는가?
- **핵심 개념 뼈대**
  - ISA vs 마이크로아키텍처: 같은 x86-64 코드가 세대별 CPU에서 다르게 수행되는 이유. ISA는 계약, 마이크로아키텍처는 구현
  - 파이프라인: 명령어를 단계로 나눠 겹치는 구조, 처리량과 지연의 구분, 해저드(앞 결과를 기다리는 데이터 해저드, 분기의 제어 해저드)
  - 분기 예측: 파이프라인을 채우기 위한 투기, 예측 실패 시 파이프라인 플러시 비용. "정렬된 배열의 조건 분기가 빠른" 고전 사례를 여기서 설명
  - 슈퍼스칼라와 비순차 실행: 독립 명령어의 병렬 발행, 의존 체인이 ILP를 제한하는 구조, 레지스터 리네이밍은 개념 수준
  - 투기 실행의 경계: 예측 실행이 캐시에 남기는 흔적 — Spectre가 드러낸 "성능 최적화 = 관찰 가능한 부작용"이라는 구조(공격 상세는 범위 밖)
  - 측정 어휘: cycles, instructions, IPC, branch-misses — 하드웨어 카운터가 세는 것
- **실무 연결**: 예측 가능한 분기 vs 예측 불가능한 분기의 구분(데이터 의존 분기), branchless 변환(조건을 산술·cmov로)이 이기는 조건과 지는 조건, 긴 포인터 추적 의존 체인(연결 리스트 순회)이 비순차 실행으로도 숨겨지지 않는 이유(ch-1 실측의 원인 절반)
- **관찰 예제**: 정렬 vs 무작위 배열에서 조건 합산 벤치마크로 branch-misses 차이를 확인하고, 같은 로직의 branchless 버전과 3자 비교한다.
- **경계 조건**: 마이크로벤치마크의 함정(루프가 통째로 예측·캐시에 안착해 실서비스와 다른 결과), 분기 예측기가 학습하는 패턴 길이의 한계, "분기 제거가 항상 이득"이 무너지는 지점(예측 잘 되는 분기 + 비싼 양쪽 계산)

### `02-memory-hierarchy.md` — 메모리 계층과 캐시 일관성

ch-1이 예고한 "캐시 효과의 원인"을 회수하는 문서이자 챕터 실습의 직접 기반이다.

- **핵심 질문**: 같은 O(n) 순회 두 개의 실측이 왜 10배 갈리는가? 스레드를 늘렸는데 왜 더 느려지는가?
- **핵심 개념 뼈대**
  - 왜 계층인가: 빠른 메모리는 작고 비싸다 — 지역성이라는 베팅 위에 세운 구조. 대표 지연 수치(L1/L2/L3/DRAM)를 자릿수 감각으로 제시
  - 캐시 라인: 전송 단위가 만드는 효과 — 인접 데이터의 공짜 로드, 낭비되는 로드. 행/열 순회 차이의 직접 원인
  - 매핑과 연관도: set-associative 구조, conflict miss가 특정 stride에서 튀는 이유(2의 거듭제곱 stride 함정)
  - 하드웨어 프리페처: 순차·stride 패턴 감지, 포인터 추적이 프리페처를 무력화하는 구조(ch-1 연결 리스트 실측의 나머지 원인)
  - 캐시 미스 분류: compulsory·capacity·conflict, 그리고 멀티코어의 coherence miss. TLB miss는 ch-8로 연결
  - 데이터 배치 설계: AoS vs SoA, 핫/콜드 필드 분리, 패딩과 정렬
  - 캐시 일관성: 코어별 사본을 일치시키는 MESI 상태 모델(Modified·Exclusive·Shared·Invalid 전이의 개념 수준), 쓰기가 다른 코어의 라인을 무효화하는 구조
  - false sharing: 서로 다른 변수가 같은 라인에 있을 때 생기는 무효화 핑퐁 — 논리적 공유가 없어도 물리적 공유가 생기는 함정
- **실무 연결**: "메모리를 더 쓰더라도 접근 패턴을 연속으로"가 이기는 조건, 멀티스레드 카운터·통계 수집에서 false sharing이 생기는 전형적 코드, 패딩·per-core 집계로 고치는 패턴
- **관찰 예제**: (1) working set 크기를 늘리며 원소당 접근 시간 곡선에서 캐시 계층 경계를 관찰, (2) stride를 늘리며 캐시 라인 크기 추정, (3) worker 2개가 인접 카운터 vs 패딩된 카운터를 갱신할 때의 처리량 비교
- **경계 조건**: 데이터가 L1에 다 들어가는 크기에서는 배치 최적화가 무의미한 지점, 프리페처가 오히려 대역폭을 낭비하는 패턴, 일관성 트래픽 때문에 코어 수 증가가 역효과가 되는 공유 쓰기 패턴

## 4. 문서 간 의존 관계

```text
ch-1 01 벤치마크 방법론 ──────────┐
ch-1 02 메모리 배치(현상) ────────┤
                                  ▼
01 파이프라인·분기 예측 ──▶ 02 메모리 계층·일관성 ──▶ 통합 실습(카운터 검증)
        │                          │
        └── 비순차 실행이 못 숨기는 └──▶ ch-8 TLB·컨텍스트 스위치 비용의 기반
            메모리 지연이라는 연결
```

- 01의 파이프라인·비순차 실행 모델은 02에서 "메모리 지연을 왜 다 숨길 수 없는가"(의존 체인, MLP 한계)의 전제가 된다.
- 02는 ch-1 01·02가 현상으로 남겨 둔 캐시 효과의 원인을 회수한다. 집필 시 ch-1 해당 문서에 상대 링크를 건다(ch-1은 이미 존재하므로 양방향 링크 가능).
- 02의 캐시 오염·TLB 개념은 ch-8(컨텍스트 스위치 간접 비용, 가상 메모리)이 위임받는다. 아직 없는 챕터로는 문장으로만 위임한다.

## 5. 실습 과제 기획 (`exercises/ch-7/`)

ROADMAP 산출물: "행/열 순회, 분기 예측 실패, false sharing을 재현하는 마이크로벤치마크를 작성하고 하드웨어 카운터로 원인을 확인한다."

### 환경 결정

- **벤치마크 본체**: TypeScript + Node.js 24(`TypedArray`, `worker_threads` + `SharedArrayBuffer`), `exercises/ch-7/` 아래 pnpm 워크스페이스 패키지. ch-1의 벤치마크 방법론(워밍업·반복·중앙값)을 그대로 적용한다.
- **하드웨어 카운터 검증**: `perf stat`은 Linux 전용이고 macOS Docker VM에서는 PMU 접근이 제한된다. 따라서 각 시나리오의 **최소 C 대조 구현**을 함께 두고, 카운터 검증(cache-misses, branch-misses, IPC)은 Linux 환경(실기 또는 PMU를 지원하는 VM)에서 C 구현 기준으로 수행한다. JIT·GC 노이즈 없이 카운터와 코드를 1:1로 대응시키기 위한 결정이다. macOS만 있는 독자를 위해 시간 기반 관찰(Node)만으로도 결론의 방향을 확인할 수 있게 시나리오를 설계한다.
- **미결**: Linux 검증 환경의 구체 형태(원격 머신 vs Lima/UTM VM)와 C 빌드 방식(Makefile 단일 파일 수준)은 실습 문서 작성 시점에 결정한다.

### 측정 시나리오 (각 시나리오는 "가설 → 시간 실측 → 카운터 확인" 3단 구성)

1. **행/열 순회**: 2차원 배열(1차원 TypedArray로 표현)을 행 우선·열 우선으로 합산. 가설: 같은 O(n²), 캐시 라인 활용률 차이. 카운터: cache-misses 비율
2. **working set 곡선**: 배열 크기를 L1 이하부터 DRAM까지 늘리며 원소당 접근 시간 기록. 캐시 계층 경계가 곡선의 계단으로 나타나는지 확인
3. **분기 예측**: 정렬 vs 무작위 배열의 조건 합산 + branchless 버전. 카운터: branch-misses, IPC
4. **false sharing**: worker 2~4개가 (a) 인접 카운터, (b) 캐시 라인 크기로 패딩된 카운터를 갱신. 가설: 논리적 경합이 없어도 (a)가 느리다. 카운터: C 구현에서 cache-misses(또는 지원 시 일관성 이벤트)

### 리포트와 완료 기준 초안

- 시나리오별로 예측 표를 먼저 쓰고 실측·카운터 수치로 검증 또는 반증한 기록을 남긴다.
- 시간 실측(Node)과 카운터(C + perf)가 같은 방향을 가리키는지, 다르면 왜인지(JIT·자동 벡터화 등)를 서술한다.
- 한 명령으로 Node 벤치 전체를 재현할 수 있고, C 대조 실험은 실행 절차를 문서화한다.
- 실행 환경(CPU 모델, 캐시 크기·라인 크기, OS, Node 버전)을 리포트에 명시한다. 캐시 파라미터 확인 명령(`sysctl`/`lscpu`)을 절차에 포함한다.
- 상세 완료 기준은 실습 문서 작성 시 확정한다.

## 6. 조사 노트 — 1차 자료 후보

집필 시 아래를 우선 확인하고, 지연 수치·캐시 파라미터처럼 하드웨어 세대에 의존하는 값은 "자릿수 감각 + 측정으로 확인" 형태로 제시한다.

- **아키텍처 일반**: Hennessy & Patterson *Computer Architecture: A Quantitative Approach*(파이프라인·캐시·일관성의 표준 서술), Patterson & Hennessy *Computer Organization and Design*(입문 층위 보조)
- **메모리 계층**: Ulrich Drepper *What Every Programmer Should Know About Memory*(캐시 구조·측정의 1차 참조 — ch-1에서 수치 인용으로 예고한 자료), Intel Optimization Reference Manual·Intel SDM(x86 캐시·프리페처의 공식 서술)
- **마이크로아키텍처 실측**: Agner Fog의 마이크로아키텍처 매뉴얼(비순차 실행·분기 예측기의 실측 기반 서술), perf 공식 문서와 Brendan Gregg *Systems Performance*(카운터 해석)
- **일관성**: Sorin, Hill, Wood *A Primer on Memory Consistency and Cache Coherence*(MESI와 일관성·순서의 구분)
- **투기 실행 경계**: Spectre 논문(Kocher et al., 2019)·Meltdown 논문 — 구조적 원인 서술의 근거로만
- **보조 검증**: Igor Ostrovsky "Gallery of Processor Cache Effects", Stroustrup 연결 리스트 실측(ch-1과 공유) — 블로그류는 자체 재현 후 보조로만 인용

### 통념 검증 목록 (본문에서 정면으로 다룰 것)

- "CPU는 코드를 한 줄씩 순서대로 실행한다" → 파이프라인·비순차·투기 실행. 순차성은 결과에 대한 약속이지 실행 방식이 아니다.
- "메모리 접근 비용은 어디든 같다" → L1과 DRAM은 자릿수가 다르다. RAM(random access)이라는 이름이 균일 비용을 보장하지 않는다.
- "명령어 수가 적은 코드가 빠르다" → IPC가 지배하는 구간에서는 명령어가 많아도 의존이 얕은 쪽이 이긴다(branchless가 이기는 이유이자, 지는 이유).
- "스레드를 늘리면 코어 수만큼 빨라진다" → 공유 쓰기의 일관성 트래픽과 false sharing이 확장성을 깎는다. 논리적 공유가 없어도 발생한다.
- "캐시는 하드웨어 문제라 코드로 어쩔 수 없다" → 배치(AoS/SoA)·순회 순서·패딩은 전부 코드 결정이다.
- "분기 예측 실패가 무섭니 분기를 모두 제거하라" → 예측이 잘 되는 분기는 거의 공짜다. 데이터 의존적·비대칭 비용 분기만 문제가 된다.
