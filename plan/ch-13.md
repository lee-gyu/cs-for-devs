# 챕터 13 기획 — 요구사항과 설계

[ROADMAP.md](../ROADMAP.md)의 챕터 13(`docs/ch-13/`, 인트로 1편과 본문 4편)을 집필하기 위한 상세 기획이다. 범위·경로가 ROADMAP과 어긋나면 ROADMAP을 우선한다.

## 1. 챕터의 관점

독자는 요구사항 문서, 설계 원칙, 아키텍처 다이어그램을 이미 접했지만 한 결정의 근거를 문제에서 검증까지 추적하지 못할 수 있다. 이 챕터는 방법론이나 패턴 목록이 아니라 **이해관계자의 문제를 검증 가능한 요구사항으로 바꾸고, 예상 변경과 품질 목표를 근거로 모듈·아키텍처 경계를 선택하며, 그 결정을 다시 검토할 수 있게 기록하는 모델**을 세운다.

공통 사례는 단순한 주문 시스템이 취소 정책, 결제 수단, 트래픽 급증, 감사 요구를 만나며 진화하는 과정이다. 요구사항·설계·아키텍처는 각각 독립된 개념 층위로 설명하되 모든 문서는 다음 질문을 공유한다.

1. 이 결정이 해결하려는 이해관계자의 목표와 제약은 무엇인가?
2. 성공과 실패를 어떤 관찰 가능한 반응으로 판별할 것인가?
3. 어떤 변경과 품질 속성이 현재 경계를 압박하는가?
4. 선택하지 않은 대안과 지불하는 비용은 무엇인가?
5. 어떤 증거가 생기면 결정을 다시 열어야 하는가?

## 2. 범위와 위임

### 다루는 것

- 이해관계자, 시스템 경계, 목표·제약·가정, 도출 기법과 편향
- 요구사항 갈등·우선순위, 기능 요구사항, 품질 속성 시나리오, 검증 가능성
- 요구사항 기준선, 양방향 추적성, 변경 영향 분석
- 정보 은닉, 예상 변경축, 응집도·결합도, 계약·불변식, 의존 방향
- 품질 속성 전술, 아키텍처 스타일의 조건부 비교, 위험·민감점·트레이드오프
- ADR의 맥락·대안·결과·상태·재검토 조건

### 위임하는 것

| 주제 | 위임 대상 | 챕터 13에서의 취급 |
|---|---|---|
| 테스트 수준·대역·속성 기반 테스트 | ch-14 `01-testing.md` | 요구사항의 검증 방법만 정의하고 테스트 전략은 위임 |
| 품질 지표·기술 부채·SLO·에러 버짓 | ch-14 | 품질 목표와 아키텍처 위험까지만 담당 |
| 애자일·팀 프로세스·CI/CD | ch-15 | 요구사항 변경과 ADR 상태만 다루고 운영 프로세스는 위임 |
| 분산 합의·정합성·부분 실패 | ch-10 | 아키텍처 대안의 품질 영향만 사용하고 원리는 위임 |

### 다루지 않는 것

- GoF 패턴과 아키텍처 스타일의 카탈로그식 암기
- UML·C4 표기법 자체에 대한 강좌
- 특정 프레임워크·클라우드 제품의 구현법
- DDD의 전술·전략 패턴 전체
- 결합도·복잡도 같은 정적 지표 하나로 설계 품질을 판정하는 방식

## 3. 문서별 설계

- `00-introduction.md`: 정확히 구현했지만 잘못된 주문 시스템에서 시작해 요구사항·설계·아키텍처의 책임과 결정 피드백 루프를 소개한다.
- `01-requirements-discovery-and-analysis.md`: 이해관계자·경계·목표·제약·가정을 찾고, 도출 기법의 편향과 갈등·우선순위를 분석한다. stakeholder map, context diagram, open question 목록을 산출한다.
- `02-requirements-specification-and-change.md`: 기능 요구사항과 품질 속성 시나리오를 검증 가능한 계약으로 만들고 기준선·추적성·변경 영향을 다룬다. 요구사항 명세와 traceability matrix를 산출한다.
- `03-modular-software-design.md`: 모듈을 파일 묶음이 아니라 변경 가능한 결정을 숨기는 경계로 설명한다. 주문·가격·결제의 두 분해안을 동일 변경 시나리오로 비교한다.
- `04-architecture-decisions-and-evaluation.md`: 품질 속성 전술과 모놀리스·계층형·이벤트 기반·서비스 분리의 조건을 비교하고, QAW·ATAM의 핵심을 축소한 평가로 ADR을 완성한다.

## 4. 통합 실습

[exercises/ch-13/README.md](../exercises/ch-13/README.md)의 필수 산출물은 실제 시스템의 설계 리뷰 패킷이다.

1. 실제 시스템과 중요한 변경 하나를 골라 이해관계자·경계·목표·제약·가정을 기록한다.
2. 핵심 기능 요구사항과 최소 3개의 측정 가능한 품질 속성 시나리오를 작성한다.
3. 모듈 또는 아키텍처 대안 2개 이상을 같은 시나리오로 비교한다.
4. 위험·민감점·트레이드오프와 증거가 부족한 가정을 표시한다.
5. 선택을 ADR 1~2개로 기록하고 요구사항에서 결정까지 추적 링크를 만든다.

동료 리뷰와 요구사항 변경 주입(change injection)은 선택 심화다. 실제 조직 정보와 운영 수치는 익명화한다.

## 5. 작성·검증 기준

- 요구사항 표준은 ISO/IEC/IEEE 29148:2018을 기준으로 하되 집필·개정 시 최신 발행 상태를 다시 확인한다.
- 품질 분류는 ISO/IEC 25010:2023, 품질 속성 시나리오와 평가 방법은 SEI QAW·ATAM 자료를 우선한다.
- 모듈 설계는 Parnas의 정보 은닉 원 논문을 중심에 두고 후대의 원칙은 그 모델에서 파생되는 휴리스틱으로 설명한다.
- 사례의 수치와 조건은 보편 기준이 아니라 주문 시스템의 가정임을 표시한다.
- 모든 대안은 동일한 요구사항과 품질 시나리오에서 비교하고 장점뿐 아니라 비용·위험·재검토 신호를 적는다.
- 각 확인 문제는 용어 회상이 아니라 모호한 요구사항 교정, 경계 선택, 트레이드오프 평가를 요구한다.
- `pnpm docs:build`, 내비게이션 문서 수·순서·레이블, 내부 링크, `git diff --check`를 완료 조건으로 삼는다.
