# 챕터 14 기획 — 품질과 신뢰성

[ROADMAP.md](../ROADMAP.md)의 챕터 14(`docs/ch-14/`, 인트로 1편과 본문 3편)를 집필하기 위한 상세 기획이다. 범위·경로가 ROADMAP과 어긋나면 ROADMAP을 우선한다.

## 1. 챕터의 관점

독자는 테스트를 쓰고 코드 리뷰를 하고 대시보드를 보아 왔지만, "테스트를 얼마나 써야 충분한가", "가용성 목표는 왜 99.9%이고 99.99%가 아닌가"에 근거로 답하지 못할 수 있다. 이 챕터는 기법 카탈로그가 아니라 **품질과 신뢰성을 위험에 대한 증거(evidence)와 예산(budget)으로 다루는 판단 모델**을 세운다. 완벽한 검증은 불가능하므로, 어떤 위험에 어떤 증거를 얼마의 비용으로 살지, 남는 위험을 어떤 예산으로 관리할지가 판단의 대상이다.

공통 사례는 [챕터 0의 주문 확정 API](../docs/ch-0/02-cs-in-practice.md)를 이어받는다. 중복 주문과 p99 급등 장애를 겪은 팀이 테스트 없는 서비스에 변경을 안전하게 가하고, 신뢰성 목표를 수치로 합의해 배포 판단에 연결하는 문제를 다룬다. 모든 문서는 다음 판단 루프를 공유한다.

1. 지키려는 불변식과 품질 목표를 검증 가능한 형태로 명시한다.
2. 위험을 줄일 증거 수단(테스트 수준, 리뷰, 정적 지표, SLI)의 비용과 신뢰도를 비교한다.
3. 싼 증거부터 배치하고, 각 증거가 보장하는 것과 보장하지 못하는 것의 경계를 기록한다.
4. 남은 위험을 예산(에러 버짓, 부채 상환 계획)으로 관리하고, 소진 신호를 배포 중단·상환 우선순위 같은 행동에 연결한다.

Part 4는 코드가 아니라 시스템과 프로세스가 분석 대상이므로, 패킷 캡처나 하드웨어 카운터 대신 **테스트 실행 결과, git 로그, 시뮬레이션 계산**을 관찰 수단으로 삼는다. 절대 수치보다 통제된 조건에서의 판단 근거와 트레이드오프 기록을 우선한다.

## 2. 범위와 위임

### 다루는 것

- 테스트 수준(단위·통합·E2E)별 작성·실행·유지 비용과 잡아내는 결함 종류의 트레이드오프
- 테스트 대역(스텁·목·페이크)이 늘어날수록 잃는 보장, 특성화 테스트(characterization test)와 seam
- 속성 기반 테스트(property-based testing)의 불변식 표현과 축소(shrinking)
- flaky 테스트의 구조적 원인(시간·공유 상태·비동기 순서)과 신뢰할 수 있는 테스트의 조건
- 커버리지가 보장하는 것과 보장하지 않는 것, 뮤테이션 테스트로 확인하는 검증 강도
- 코드 리뷰가 실제로 잡는 결함의 연구 증거와 한계
- 정적 지표(커버리지·순환 복잡도·중복)의 의미와 한계, 지표가 목표가 될 때의 왜곡(Goodhart의 법칙)
- 기술 부채의 식별(변경 빈도×복잡도 핫스팟)과 상환의 경제학(이자 = 변경 비용 증가)
- SLI·SLO·에러 버짓, burn rate 경보, 장애 대응과 포스트모템, 점진적 배포와 에러 버짓 정책의 연결

### 위임하는 것

| 주제 | 위임 대상 | 챕터 14에서의 취급 |
|---|---|---|
| 품질 요구사항 도출과 품질 속성 시나리오 | ch-13 `02-requirements-specification-and-change.md` (작성 중) | 정의된 목표를 검증·측정하는 부분만 담당 |
| CI/CD 파이프라인 구축, 관찰 가능성 스택(메트릭·로그·트레이스) | ch-15 | 배포 정책의 판단 기준(에러 버짓 연동)만 담당 |
| 정적 분석의 원리(데이터 흐름, 건전성·완전성) | ch-5 [`04-static-analysis.md`](../docs/ch-5/04-static-analysis.md) | 지표의 운영적 의미와 한계만 담당 |
| 부분 실패, 재시도, 멱등성의 분산 원리 | ch-10 | 신뢰성 목표의 배경으로만 언급 |

### 다루지 않는 것

- 특정 테스트 프레임워크·도구의 사용법 카탈로그와 비교는 다루지 않는다.
- 형식 검증(TLA+, 모델 체킹)은 완전한 검증이 불가능하다는 경계의 반례로만 언급하고 확장하지 않는다.
- 카오스 엔지니어링의 상세 실행과 도구는 다루지 않는다.
- 온콜 로테이션·팀 구조 같은 조직 설계는 다루지 않는다.
- 보안 테스트(침투 테스트, SAST/DAST 운용)는 독립 주제로 확장하지 않는다.

## 3. 문서별 설계

- `00-introduction.md`: 챕터 0의 주문 확정 API 장애를 재방문해 "무엇을 알았더라면 이 배포를 막을 수 있었나"를 묻는다. 100% 신뢰성이 목표가 될 수 없는 비용 곡선을 세우고, 품질을 위험에 대한 증거로, 신뢰성을 실패의 예산으로 재정의한 뒤 챕터 지도를 그린다.
- `01-testing.md`: 테스트 수준을 비용(작성·실행·유지)×신뢰도(잡는 결함의 종류) 트레이드오프로 재구성한다. 테스트 대역이 많아질수록 잃는 보장, 주문 금액·멱등성 불변식을 검증하는 속성 기반 테스트, flaky의 구조적 원인을 다룬다. 커버리지 리포트가 보장하지 않는 것을 뮤테이션 테스트로 관찰한다.
- `02-quality-and-maintainability.md`: 코드 리뷰가 실제로 잡는 결함의 연구 증거와 리뷰가 못 잡는 것을 구분한다. 정적 지표의 의미와 지표가 목표가 될 때의 왜곡을 다루고, git 로그 기반 핫스팟(변경 빈도×복잡도)으로 기술 부채를 계측해 상환을 이자(변경 비용)로 판단하는 모델을 세운다.
- `03-reliability-engineering.md`: 좋은 SLI의 조건(좋은 이벤트/전체 이벤트), 사용자 여정 기준의 SLO 선택, 에러 버짓(1−SLO)의 운영적 의미를 다룬다. 빠른·느린 burn rate 경보 설계, 장애 대응의 역할 분리와 비난 없는 포스트모템, 카나리·점진 배포가 에러 버짓을 지키는 메커니즘을 배포 정책으로 연결한다.

## 4. 통합 실습

`exercises/ch-14/`의 산출물은 **테스트 하네스 구축 기록과 SLO·에러 버짓 설계 리포트**다. 두 부로 구성한다.

- **Part A — 테스트 하네스 구축**: 저장소가 제공하는 소형 레거시 주문 확정 서비스(TypeScript, Node 24 기준, 시간·전역 상태·외부 호출이 의도적으로 얽힌 구조)에 특성화 테스트 → seam 도입 → 테스트 대역 교체 → 속성 기반 테스트 순으로 하네스를 구축한다. 각 단계에서 어떤 위험에 대한 증거가 생겼고 무엇이 여전히 검증되지 않았는지 기록한다. 테스트 러너는 의존성이 가장 적은 `node:test`를 기본으로 하되 최종 선택은 실습 작성 시 확정한다.
- **Part B — SLO·에러 버짓 시뮬레이션**: 가상 트래픽·장애 시나리오를 생성하는 스크립트로 요청 로그를 산출하고, SLI 계산 → SLO·에러 버짓 정의 → burn rate 경보 계산 → 가상 배포 시나리오에서 진행/중단 판단 기록까지 수행한다. 외부 계측 도구 없이 로컬에서 재현 가능해야 하며, 산출물은 SLO 문서와 판단 리포트다.

Part A의 레거시 코드는 모든 독자가 같은 시작점에서 재현할 수 있도록 저장소에 포함한다. 자신의 실무 코드로 대체하는 것은 심화 트랙으로 허용한다.

## 5. 작성·검증 기준

- 연구 결과와 수치 인용은 1차 자료(논문, 공식 공개 문서)로 한정하고, 통념(예: 테스트 피라미드의 비율)은 근거의 한계를 함께 적는다.
- 성능·비용 주장 대신 판단 주장(이 증거는 이 위험을 줄인다)을 쓸 때도 확인 방법(테스트 실행, 뮤테이션 결과, 시뮬레이션 수치)을 함께 제시한다.
- 관찰 수단은 테스트 실행 결과, git 로그, 시뮬레이션 계산으로 삼고 실행 환경(Node 버전)을 명시한다.
- 각 확인 문제는 용어 회상이 아니라 주어진 테스트 실패·지표·버짓 소진 상황에서 다음 판단을 고르게 한다.
- `pnpm docs:build`, 내비게이션 문서 수·순서, 내부 링크, `git diff --check`를 완료 조건으로 삼는다.

## 6. 1차 자료

- Google SRE Book — [Embracing Risk](https://sre.google/sre-book/embracing-risk/), [Service Level Objectives](https://sre.google/sre-book/service-level-objectives/), [Postmortem Culture](https://sre.google/sre-book/postmortem-culture/): 에러 버짓·SLO·포스트모템의 원 정의
- Google SRE Workbook — [Implementing SLOs](https://sre.google/workbook/implementing-slos/), [Alerting on SLOs](https://sre.google/workbook/alerting-on-slos/): SLI 선택과 burn rate 경보 설계
- Claessen & Hughes, [QuickCheck (ICFP 2000)](https://doi.org/10.1145/351240.351266): 속성 기반 테스트의 원 논문. [fast-check 공식 문서](https://fast-check.dev/)를 구현 기준으로 사용
- Luo et al., [An Empirical Analysis of Flaky Tests (FSE 2014)](https://doi.org/10.1145/2635868.2635920): flaky 원인 분류의 근거
- Bacchelli & Bird, [Expectations, Outcomes, and Challenges of Modern Code Review (ICSE 2013)](https://www.microsoft.com/en-us/research/publication/expectations-outcomes-and-challenges-of-modern-code-review/): 코드 리뷰가 실제로 잡는 것의 근거
- Cunningham, [The WyCash Portfolio Management System (OOPSLA 1992)](http://c2.com/doc/oopsla92.html): 기술 부채 메타포의 원문
- Michael Feathers, *Working Effectively with Legacy Code* (Prentice Hall, 2004): seam과 특성화 테스트
- [ISO/IEC 25010:2023](https://www.iso.org/standard/78176.html): 품질 모델의 표준 용어(개요 수준), [Stryker 공식 문서](https://stryker-mutator.io/): 뮤테이션 테스트 도구 기준
