# 챕터 3 기획 — 정보와 암호

[ROADMAP.md](../ROADMAP.md)의 챕터 3(`docs/ch-3/`, 인트로 1편과 본문 4편)을 집필하기 위한 상세 기획이다.
범위·경로가 ROADMAP과 어긋나면 ROADMAP을 우선한다.

## 1. 챕터의 관점

독자는 5년차 이상 개발자다. gzip 압축, HTTPS, 해시 함수, JWT를 매일 사용하지만, 각 도구가 무엇을 근거로 어디까지 보장하는지를 모델로 설명하지는 못할 수 있다.
이 챕터는 정보 이론과 암호학을 정의와 수식의 목록으로 가르치지 않고, 데이터의 표현·보호·신뢰를 판단하는 네 질문으로 조직한다.

1. **정보 — 데이터의 실제 크기는 얼마인가?** 불확실성(엔트로피)이 압축과 부호화의 하한을 정한다.
2. **기밀성과 무결성 — 내용을 숨기고 변조를 감지하려면 각각 무엇이 필요한가?** 대칭 암호·해시·MAC의 보장을 구분하고 조합한다.
3. **키 분배 — 만난 적 없는 상대와 어떻게 비밀을 공유하는가?** 비대칭 암호가 푸는 문제와 풀지 않는 문제를 구분한다.
4. **신뢰 — 그 공개키가 진짜라는 것을 누가 보증하는가?** 인증서·CA·TLS로 프리미티브가 신뢰 사슬로 조립된다.

각 프리미티브는 "무엇을 보장하고, 무엇을 가정하며, 어떤 오용에서 무너지는가"라는 계약으로 서술한다.
목적은 수학적 정당성 증명이 아니라 보장·가정·오용의 경계를 판단하는 능력이다. 정의는 정확히 제시하되 수식은 경계를 이해하는 데 필요한 만큼만 사용한다.

## 2. 범위 결정

### 다루는 것

- 정보량과 엔트로피, 고정·가변 길이 부호와 prefix code, 허프만 부호화, Shannon source coding 한계
- 오류 감지·정정의 원리: 해밍 거리, 패리티, 해밍 코드의 구성 원리와 실무 오류 정정 기술의 지도
- Kerckhoffs 원리, 스트림·블록 암호, AES, 운용 모드(ECB의 실패 → CBC → CTR → GCM/AEAD)와 nonce 오용
- 암호학적 해시의 세 성질, 길이 확장 공격과 HMAC, KDF(HKDF·argon2 계열)와 비밀번호 저장
- 키 분배 문제, Diffie–Hellman 키 교환, RSA의 구조, 타원곡선(ECDHE), 디지털 서명, 하이브리드 암호, forward secrecy
- X.509 인증서와 CA 체인, 서명 체인·이름 검증과 실패 모드, TLS 1.3 cipher suite의 프리미티브 분해

### 다루지 않기로 결정한 것

- **LZ 계열·zstd의 내부 구현은 다루지 않는다.** 사전 기반 압축은 엔트로피 한계의 관점에서 결과만 관찰하고, 01의 중심은 정보량 모델과 허프만 부호화에 둔다.
- **Reed-Solomon과 유한체 수학은 제외한다.** 오류 정정은 해밍 거리·패리티·해밍 코드로 원리를 세우고, CRC·checksum·ECC 메모리·Reed-Solomon은 어디에 쓰이는지 실무 지도로만 연결한다.
- **RSA 정당성의 수론 증명은 다루지 않는다.** 오일러 정리 전개보다 "공개키로 잠그고 개인키로 여는 구조가 어떤 난제에 기대는가"가 실무 판단에 더 중요하다.
- **블록체인과 양자 암호는 제외한다.** 포스트 양자 암호는 03에서 현황 한 단락으로만 다룬다.
- **OAuth·세션 관리 등 인증 프로토콜 설계는 다루지 않는다.** JWT는 03의 서명 검증 사례(HS256/RS256 혼동)로만 사용한다.
- **프로덕션 암호 코드의 직접 구현을 권장하지 않는다.** 본문·실습의 toy 구현은 계약과 실패 모드를 계측하기 위한 학습 도구이며, 실무에서는 검증된 라이브러리와 프로토콜을 사용해야 함을 각 문서에 명시한다.

## 3. 문서별 상세 기획

본문 4편은 콘텐츠 집필 지침의 기본 구조(학습 목표 → 배경 → 핵심 개념 → 실무 관점 → 더 깊이 → 정리 → 확인 문제 → 참고 자료)를 따르고, 45~90분 분량으로 작성한다. 각 본문에는 10~20분 안에 실행·관찰할 수 있는 TypeScript(Node.js 24) 미니 실험을 하나씩 둔다. 00 인트로는 10~15분 분량의 오리엔테이션으로 구성한다.

### `00-introduction.md` — 비트에는 양·변조·도청·신원의 문제가 따라온다

- **핵심 질문**: 데이터는 그냥 저장하고 전송하면 될 것 같은데, 왜 비트의 양, 변조, 도청, 상대의 신원이 각각 별개의 문제가 되는가?
- 같은 HTTPS 요청 하나에 압축(양), 무결성 검사(변조), 암호화(도청), 인증서 검증(신원)이 모두 개입함을 보여 문제를 분리한다.
- 압축은 중복을 제거하는 방향, 오류 정정은 중복을 의도적으로 넣는 방향, 암호는 구조를 의도된 무질서로 바꾸는 방향이라는 하나의 축을 세운다.
- 신뢰는 수학만으로 완결되지 않고 CA라는 제도와 결합된다는 관점을 예고하며, 네 질문을 01~04의 학습 지도로 제시한다.

### `01-information-theory.md` — 데이터의 실제 크기와 압축의 한계

허프만 실습의 이론 기반이며, 챕터 전체가 사용할 "불확실성과 중복"의 어휘를 세운다.

- **핵심 질문**: 이 데이터는 얼마나 더 줄일 수 있고, 그 한계는 누가 정하는가?
- **핵심 개념 뼈대**
  - 정보량: 드문 사건일수록 많은 정보를 담는다는 직관을 `-log₂p`로 정식화한다
  - 엔트로피: 심볼 분포의 평균 정보량, 균등 분포에서 최대가 되는 이유, 단위로서의 비트
  - 부호화: 고정 길이 부호의 낭비, 가변 길이 부호와 복호 가능성, prefix code와 부호 트리의 대응
  - Kraft 부등식: prefix code가 쓸 수 있는 부호 길이의 예산이라는 직관 수준까지만
  - 허프만 부호화: 빈도 기반 트리 구성, 심볼 단위 최적성의 의미와 한계(분수 비트, 심볼 간 상관관계). 그리디 정당성은 ch-1로 위임
  - Shannon source coding 한계: 평균 부호 길이가 엔트로피 아래로 내려갈 수 없다는 의미, 이미 압축된 데이터가 더 줄어들지 않는 이유
  - 오류 감지·정정: 중복을 빼는 압축과 중복을 설계해서 넣는 부호화의 대칭, 해밍 거리와 감지·정정 능력의 관계, 패리티 비트, 해밍 코드의 구성 원리
- **실무 연결**
  - gzip·zstd·brotli는 사전 기반 변환과 엔트로피 부호화의 조합이며, 압축률의 차이는 모델링의 차이임을 지도 수준으로 정리한다
  - 압축 포맷(JPEG·PNG·MP3)의 손실·무손실 구분을 엔트로피 관점으로 해석한다
  - CRC(전송 오류 감지), checksum, ECC 메모리, RAID·QR 코드의 Reed-Solomon을 "어떤 오류 모델에 어떤 중복 설계인가"의 지도로 연결한다
  - 오류 감지용 CRC·checksum과 변조 감지용 MAC의 차이를 예고해 02로 연결한다
- **미니 실험**: 텍스트 코퍼스의 문자 빈도로 엔트로피를 계산하고, `node:zlib` gzip 압축률과 대조한다. 자연어·균등 랜덤 바이트·이미 gzip된 데이터 세 입력에서 엔트로피가 압축률을 예측하는지 관찰한다.
- **경계 조건**: 엔트로피는 심볼 모델에 대한 값이며 모델이 달라지면(문자 단위 vs 단어 단위) 값도 달라진다. "압축을 반복하면 계속 줄어든다"는 통념을 랜덤 데이터와 재압축 실험으로 반박한다.

### `02-symmetric-crypto-and-integrity.md` — 기밀성과 무결성은 다른 계약이다

같은 키를 공유한 두 당사자 사이의 보장을 다루고, 03의 "그 키를 어떻게 나누는가"로 이어지는 출발점이다.

- **핵심 질문**: 같은 키를 가진 둘 사이에서 기밀성과 무결성은 각각 무엇으로 보장되며, 암호화만 하면 왜 부족한가?
- **핵심 개념 뼈대**
  - Kerckhoffs 원리: 알고리즘이 아니라 키만 비밀이어야 하는 이유, 보안을 비밀 알고리즘에 기대는 설계의 실패
  - one-time pad: XOR과 완전 비밀성, 키 길이·재사용 불가라는 비현실성이 실용 암호의 문제 설정을 만드는 과정
  - 스트림 암호와 블록 암호: 짧은 키에서 긴 키스트림을 펼치는 구조와 고정 크기 블록 치환 구조
  - AES: 라운드 반복 구조의 개요까지만. 내부 수학보다 "표준화·검증된 블록 치환"이라는 계약에 집중
  - 운용 모드: 같은 평문 블록이 같은 암호문이 되는 ECB의 구조 노출, CBC의 연쇄와 IV, CTR의 카운터 기반 병렬성, nonce 재사용이 키스트림 재사용이 되는 재앙
  - 암호학적 해시: 역상·제2역상·충돌 저항의 구분, 생일 역설과 충돌 비용
  - 길이 확장 공격과 HMAC: `hash(key ∥ message)`가 무너지는 구조와 HMAC의 이중 해시 설계
  - 암호화 ≠ 무결성: CTR 암호문의 비트 반전이 평문을 예측 가능하게 바꾸는 사례, Encrypt-then-MAC과 AEAD(GCM)가 두 계약을 하나로 묶는 방식
  - 키 유도: 패스워드는 균등 랜덤 키가 아니다 — 빠른 해시와 느린 KDF(argon2·bcrypt·scrypt)의 목적 차이, 키 계층을 만드는 HKDF
- **실무 연결**
  - "비밀번호는 SHA-256으로 해시하면 안전하다"는 통념을 GPU 병렬 탐색 비용으로 반박하고 salt·work factor의 역할을 정리한다
  - 라이브러리 API에서 mode·IV·nonce·tag 파라미터를 계약 위반 없이 쓰는 기준을 세운다 (nonce 생성 전략, tag 검증 생략의 위험)
  - 01의 압축과 연결: 암호문이 압축되지 않는 이유(무질서), 압축을 암호화 앞에 두면 생기는 길이 부채널은 ch-9 맥락으로 위임
- **미니 실험**: `node:crypto`로 단순한 비트맵 이미지를 AES-ECB와 AES-GCM으로 각각 암호화하고 픽셀로 다시 렌더링해, ECB에서 원본 구조가 눈에 보이는 현상(ECB 펭귄)을 재현한다.
- **경계 조건**: AEAD도 nonce 재사용·tag 미검증·키 재사용 범위 초과에서는 무너진다. "표준 알고리즘을 썼다"와 "계약을 지켰다"를 구분한다.

### `03-asymmetric-crypto-and-key-exchange.md` — 공유한 비밀 없이 비밀을 만드는 법

02가 전제한 "공유된 대칭 키"가 실제로 어떻게 만들어지는지를 다루고, 04의 신원 문제를 남긴다.

- **핵심 질문**: 사전에 공유한 비밀이 전혀 없는 두 당사자가, 모든 통신을 지켜보는 도청자 앞에서 어떻게 비밀 키를 합의하는가?
- **핵심 개념 뼈대**
  - 키 분배 문제: n명 전원 통신에 n²개의 사전 공유 키가 필요한 구조, 키 배송이라는 순환 문제
  - Diffie–Hellman: 공개 파라미터와 비밀 지수, 한 방향은 쉽고 역방향(이산로그)은 어려운 함수에 기대는 구조. 난제의 "어려움"은 ch-2의 계산 비용 개념으로 문장 위임
  - RSA: 소인수분해 기반의 공개키·개인키 구조 개요, 암호화와 서명에 같은 수학이 다른 방향으로 쓰이는 방식. 수론 증명은 생략을 명시
  - "RSA로 데이터를 암호화하지 않는" 이유: 비용, 메시지 크기 제한, 패딩의 위험 — 비대칭은 키 교환·서명에만 쓰고 데이터는 대칭이 맡는 하이브리드 구조
  - 타원곡선: 같은 보안 수준을 훨씬 짧은 키로 얻는 비용 구조, X25519·ECDHE가 기본값이 된 배경
  - 디지털 서명: 해시-후-서명 구조, MAC과의 차이(검증자와 서명자의 키 분리, 부인 방지), 서명이 "내용 승인"이 아니라 "키 보유 증명"이라는 계약
  - ephemeral 키와 forward secrecy: 장기 키 유출이 과거 트래픽을 소급 복호화하지 못하게 하는 구조, TLS 1.3이 static RSA 키 교환을 제거한 이유
  - 포스트 양자 현황: Shor 알고리즘이 위협하는 대상과 NIST FIPS 203–205 표준화 현황을 한 단락으로
- **실무 연결**
  - JWT의 HS256(HMAC)과 RS256(서명)을 혼동해 검증 키를 잘못 다루는 사례로 MAC과 서명의 계약 차이를 확인한다
  - SSH 키 페어, 코드 서명, 패키지 서명을 "공개키를 미리 신뢰할 수 있는 채널"의 유무로 분류하고 04의 인증서 문제를 예고한다
- **미니 실험**: 작은 소수 파라미터로 toy DH를 구현하고 brute force로 이산로그를 깨서 파라미터 크기가 곧 보안임을 계측한다. `node:crypto`의 X25519 키 교환과 실행 시간·키 크기를 대조한다.
- **경계 조건**: DH는 도청자(passive attacker)만 막는다. 중간에서 양쪽과 각각 키를 합의하는 MITM은 키 교환만으로 막을 수 없으며, 이 공백이 04의 존재 이유다.

### `04-certificates-and-tls-trust-chain.md` — 공개키를 신뢰 사슬로 묶는 법

01~03의 프리미티브가 TLS라는 하나의 프로토콜로 조립되는 지점이며, 챕터의 종착점이다.

- **핵심 질문**: 처음 접속한 서버가 보낸 공개키가 정말 그 서버의 것인지 누가 어떻게 보증하는가?
- **핵심 개념 뼈대**
  - MITM 재방문: 03의 키 교환 공백을 명시적으로 이어받아, 공개키와 신원의 바인딩이 빠져 있음을 문제로 세운다
  - X.509 인증서: 신원(도메인)과 공개키를 CA의 서명으로 묶은 문서라는 본질, 주요 필드(subject·SAN·유효기간·키 용도)
  - 신뢰 사슬: 리프 → 중간 CA → 루트, 루트 저장소가 OS·브라우저에 내장되는 구조, 신뢰의 뿌리가 수학이 아니라 제도·감사라는 사실
  - 체인 검증: 서명 체인 검증, 유효기간, 이름 검증(SAN 대조), 폐기(CRL·OCSP)의 현실적 한계
  - 실패 모드: 만료, 자체 서명, 중간 CA 누락, 호스트명 불일치를 브라우저·curl·Node의 실제 에러 메시지와 연결한다
  - TLS 1.3 조립: cipher suite와 negotiated group·signature scheme을 01~03의 프리미티브로 분해한다 — ECDHE(03 키 교환) + 인증서 서명(03 서명 + 04 신뢰 사슬) + AEAD(02) + HKDF(02). 핸드셰이크의 전송·성능 관점(RTT·재개·0-RTT·QUIC)은 ch-9 [04-tls-quic-and-http3.md](../docs/ch-9/04-tls-quic-and-http3.md)로 위임
  - HTTPS가 보장하는 것과 않는 것: 기밀성·무결성·서버 신원 인증까지가 계약이고, 서버의 선의·콘텐츠의 안전·메타데이터(SNI·트래픽 패턴) 은닉은 계약 밖임을 정리
- **실무 연결**
  - 인증서 에러를 "무시하고 진행"하는 옵션(`-k`, `rejectUnauthorized: false`)이 정확히 어떤 계약을 포기하는지 명시한다
  - 사내 CA·mTLS·certificate pinning을 신뢰 사슬의 변형으로 분류한다
  - Let's Encrypt·ACME 자동화가 바꾼 유효기간·발급 관행을 지도 수준으로 정리한다
- **미니 실험**: `openssl s_client`로 실제 사이트의 인증서 체인을 덤프해 subject·issuer가 사슬로 이어지는 구조를 추적하고, `node:tls`로 자체 서명 인증서 서버에 접속해 검증 실패 에러를 의도적으로 재현한다.
- **경계 조건**: 검증을 통과한 인증서는 "그 도메인의 키"임을 보증할 뿐이다. CA가 오발급하면 사슬 전체가 오염되며(실제 CA 퇴출 사례 언급), 신뢰 사슬은 수학적 증명이 아니라 위험을 제도적으로 관리하는 구조임을 확인한다.

## 4. 문서 간 의존 관계

```text
01 정보량·엔트로피·중복 ──▶ 02 무질서로서의 암호문, 압축 부채널 관찰
            │                        │
            │                        ├──▶ 03 "그 대칭 키는 어떻게 나누는가"
            │                        │            │
            └── 허프만 ──▶ 실습 A    │   (MITM 공백)▼
                                     └──────▶ 04 인증서·TLS 조립 ──▶ 실습 B
```

- 01의 "중복과 구조" 어휘는 02의 ECB 구조 노출과 압축 부채널 관찰에 재사용한다.
- 02는 대칭 키 공유를 전제하고, 03이 그 전제를 키 분배 문제로 되돌려 해결한다.
- 03의 키 교환은 도청자만 막고 MITM 공백을 남기며, 04가 인증서로 그 공백을 메운다.
- 04는 02(AEAD·HKDF)와 03(ECDHE·서명)을 TLS 1.3 조립으로 종합하고, 전송·성능 관점은 기존 ch-9 문서에 상대 링크로 위임한다.
- 03의 계산 난제는 미집필된 ch-2에 "챕터 2에서 다룬다"는 문장으로 위임하고 TODO 링크를 남기지 않는다. 01의 허프만 정당성은 기존 ch-1 문서로 링크한다.

## 5. 실습 과제 기획 (`exercises/ch-3/`)

ROADMAP 산출물: "허프만 부호화 압축기를 구현해 엔트로피 한계와 비교하고, 실제 TLS 핸드셰이크를 캡처해 암호 프리미티브의 조합을 분석한다."

### 환경

- **구현**: TypeScript + Node.js 24, `node:test`, 별도 런타임 의존성 없음. 실습 코드는 `exercises/ch-3/` 아래에 둔다 (ch-4 실습의 package.json 구성 선례를 따른다).
- **TLS 분석 도구**: `openssl s_client`, `curl -v`, tshark(패킷 캡처, `SSLKEYLOGFILE` 복호화). 모든 절차는 macOS에서 검증 가능하게 구성한다 (ch-8·9의 Linux 의존 TODO와 달리 이 챕터는 macOS로 완결한다).

### Part A — 허프만 압축기

- **구현 범위**: 바이트 단위 빈도 집계 → 허프만 트리 구성 → 부호표 생성 → 비트 스트림 인코딩/디코딩 왕복. 트리(또는 부호표) 직렬화를 포함해 단일 파일 압축·해제가 완결되게 한다.
- **계측 인터페이스**: 입력의 바이트 단위 엔트로피(이론 하한), 허프만 평균 부호 길이, 실제 압축률(헤더 오버헤드 포함/제외), `node:zlib` gzip 대조값을 공통 리포트로 출력한다.
- **검증 시나리오**
  - **왕복 무손실**: 임의 바이트 열을 포함한 다양한 입력에서 인코딩→디코딩이 원본과 일치하는지 단위 테스트한다. 빈 입력, 단일 심볼 입력, 모든 심볼이 균등한 입력을 경계 사례로 포함한다.
  - **엔트로피 예측**: 자연어 텍스트 / 균등 랜덤 바이트 / 이미 gzip된 데이터 세 입력에서 엔트로피가 압축률을 예측하는지 확인한다.
  - **한계 확인**: 평균 부호 길이가 엔트로피 하한과 엔트로피+1 사이에 들어오는지 확인하고, gzip이 허프만 단독보다 잘 줄이는 입력에서 그 차이를 심볼 간 상관관계(모델링의 차이)로 설명한다.
- **완료 기준**: 왕복 무손실 테스트 전체 통과, 한 명령으로 세 시나리오의 계측 리포트 재현, 평균 부호 길이와 엔트로피 한계의 관계를 수치로 확인한 분석.

### Part B — TLS 핸드셰이크 분석

- **대상**: 실제 공개 사이트 2곳 이상(서로 다른 CA·키 교환 그룹이 관찰되도록 선정)과 자체 서명 인증서를 쓰는 로컬 Node.js TLS 서버.
- **수행 절차**: `SSLKEYLOGFILE`을 설정한 `curl`로 요청을 보내며 tshark로 캡처하고 핸드셰이크를 복호화한다. `openssl s_client`로 같은 대상의 인증서 체인을 덤프한다.
- **식별 항목**: TLS 버전, cipher suite와 그 분해(키 교환 그룹 / 인증서 서명 알고리즘 / AEAD / 해시), 인증서 체인 구조(리프→중간→루트), 세션 재개 여부.
- **실패 재현**: 자체 서명 서버에 대해 검증 실패 에러를 관찰하고, 검증을 끈 접속과의 차이를 캡처로 대조한다.
- **완료 기준**: 관찰된 각 필드를 02~04의 프리미티브 계약(어떤 보장을 누가 제공하는가)과 연결한 분석 리포트. 핸드셰이크의 RTT·재개·0-RTT 분석은 ch-9 실습 범위로 위임하고 반복하지 않는다.

## 6. 조사 노트 — 1차 자료 후보

집필 시 아래 원문과 표준을 우선 확인하고, 도구 출력·기본 cipher suite·루트 정책처럼 구현과 버전에 의존하는 내용은 인용 시점에 다시 검증한다.

- **정보 이론**: Shannon "A Mathematical Theory of Communication"(1948), Huffman "A Method for the Construction of Minimum-Redundancy Codes"(1952), Hamming "Error Detecting and Error Correcting Codes"(1950). RFC 1952(gzip 포맷)는 실험 대조용으로 확인한다.
- **대칭 암호와 무결성**: FIPS 197(AES), NIST SP 800-38 시리즈(운용 모드, 특히 38A와 38D의 GCM), FIPS 180-4·202(SHA-2·SHA-3), RFC 2104(HMAC), RFC 5869(HKDF), RFC 9106(Argon2), NIST SP 800-63B(비밀번호 저장 지침).
- **비대칭 암호**: Diffie–Hellman "New Directions in Cryptography"(1976), Rivest–Shamir–Adleman(1978), RFC 7748(X25519), RFC 8017(PKCS#1), RFC 8032(EdDSA), NIST FIPS 203–205(포스트 양자 표준).
- **인증서와 TLS**: RFC 8446(TLS 1.3), RFC 5280(X.509·체인 검증), RFC 6125·9525(호스트명 검증), CA/Browser Forum Baseline Requirements, 주요 루트 프로그램(Mozilla·Apple·Chrome) 정책 문서. Node.js `node:crypto`·`node:tls`와 OpenSSL의 공식 문서로 도구 동작을 검증한다.

### 통념 검증 목록 (본문에서 정면으로 다룰 것)

- "압축을 반복하면 계속 줄어든다" → 엔트로피 한계와 재압축 실험으로 반박한다. (01)
- "암호화하면 변조도 막힌다" → CTR 비트 반전 사례로 기밀성과 무결성을 분리하고 AEAD의 존재 이유를 세운다. (02)
- "해시는 암호화의 일종이다" → 키의 유무와 비가역성으로 계약을 구분한다. (02)
- "비밀번호는 SHA-256이면 충분하다" → 빠른 해시와 느린 KDF의 목적 차이, salt·work factor로 반박한다. (02)
- "RSA로 본문 데이터를 암호화한다" → 하이브리드 구조와 TLS 1.3의 static RSA 제거로 반박한다. (03)
- "인증서가 암호화를 제공한다" → 인증서는 신원-공개키 바인딩이고 세션 키는 ECDHE가 만든다는 분해로 바로잡는다. (04)
- "HTTPS면 안전한 사이트다" → HTTPS의 계약(기밀성·무결성·서버 신원)과 계약 밖(서버의 선의, 메타데이터)을 구분한다. (04)

## 7. 작성 순서와 검증 계획

1. 독자는 `00 → 01 → 02 → 03 → 04` 순서로 읽는다. 00이 네 질문과 학습 지도를 세우고, 정보의 어휘에서 대칭 암호·키 교환·신뢰 사슬로 확장한다.
2. 집필은 01·02를 먼저 진행하고, 허프만 starter와 엔트로피 계측 모델을 검증해 본문 설명과 실습 A의 모델이 어긋나지 않게 한 뒤 03·04로 넘어간다.
3. 각 문서의 미니 실험은 Node.js 24에서 직접 실행하고, 출력·수치·실행 환경(macOS, Node·OpenSSL 버전)을 결과 해석에 필요한 수준으로 명시한다.
4. toy 구현(DH, ECB 관찰)에는 학습 계측 목적임과 프로덕션 사용 금지를 명시하고, 실무 권장은 검증된 라이브러리·프로토콜로 연결한다.
5. 각 문서의 확인 문제는 정의 회상이 아니라 압축률 이상 진단, 모드·nonce 선택, MAC과 서명 선택, 인증서 에러 원인 분석처럼 원인과 트레이드오프를 설명하게 한다.
6. `docs/ch-3/`을 만들 때 `docs/.vitepress/navigation.ts`의 `PHASE_LABELS`에 챕터 3 레이블을 등록하고, VitePress 빌드로 nav·sidebar와 내부 링크를 확인한다.
7. 문서 수나 완료 상태가 실제로 바뀔 때만 `PROGRESS.md`를 갱신한다. 실습 문서는 인트로와 본문 4편의 모델과 예제가 안정된 뒤 이 기획의 §5를 기준으로 작성한다.
