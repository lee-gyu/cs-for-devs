# 챕터 8 기획 — 운영체제

[ROADMAP.md](../ROADMAP.md)의 챕터 8(`docs/ch-8/`, 인트로 1편과 본문 3편)을 집필하기 위한 상세 기획이다.
범위·경로가 ROADMAP과 어긋나면 ROADMAP을 우선한다.

## 1. 챕터의 관점

독자는 5년차 이상 개발자다. 서버를 운영하며 CPU 사용률, load average, RSS, iowait 같은 지표를 봐 왔고, 스레드 풀 크기나 이벤트 루프 기반 프레임워크를 선택해 본 경험이 있다. 그러나 그 지표와 선택지를 커널의 동작 모델로 설명하지 못하면, 지표가 이상할 때 추측으로 대응하게 된다.
이 챕터는 커널 기능의 카탈로그가 아니라 **"시스템 지표와 아키텍처 선택을 커널 모델로 해석하는 능력"**을 세운다. 더하는 것은 세 가지다.

1. **실행의 단위와 비용** — 프로세스·스레드·컨텍스트 스위치·스케줄러 모델로 "코어보다 많은 스레드가 도는 방식"과 그 비용을 이해하고, CPU 지표와 지연 문제를 진단한다.
2. **메모리의 착시 걷어내기** — 가상 메모리 위에서 RSS·virtual·swap·페이지 캐시가 각각 무엇을 세는지 구분하고, 페이지 폴트·OOM·thrashing을 동작 모델로 해석한다.
3. **I/O 경로와 서버 아키텍처** — `write()`가 리턴한 데이터가 실제로 어디에 있는지(페이지 캐시, fsync), 그리고 블로킹·다중화(epoll)·완료 기반(io_uring) 모델의 구조 차이가 thread-per-connection vs 이벤트 루프라는 아키텍처 선택으로 이어지는 과정을 세운다.

기준 OS는 Linux다. POSIX가 보장하는 것과 Linux 구현 세부를 구분해 서술한다.

## 2. 범위 결정

### 다루는 것

- 프로세스와 스레드의 커널 표현, `fork`/`exec` 모델, 시스템 콜과 모드 전환 vs 컨텍스트 스위치의 구분
- 컨텍스트 스위치의 직접 비용(레지스터·커널 경로)과 간접 비용(캐시·TLB 오염), 스케줄링(타임슬라이스, 우선순위·nice, CFS→EEVDF의 설계 방향), run queue와 load average의 정확한 의미
- 페이지 테이블과 주소 변환, TLB, demand paging, minor/major 페이지 폴트, `mmap`(파일·익명), copy-on-write, overcommit과 OOM killer, 스왑과 thrashing
- VFS와 파일 디스크립터, 페이지 캐시(read-ahead, write-back, dirty page), `fsync`와 내구성 계약, 저널링의 목적
- I/O 모델의 계보: 블로킹 + thread-per-connection → 논블로킹 + readiness 다중화(select/poll/epoll, level/edge trigger) → completion 기반(io_uring의 SQ/CQ 구조), C10K 문제
- 정규 파일에 readiness 모델이 통하지 않는 이유와 런타임(libuv)의 스레드 풀 우회

### 다루지 않기로 결정한 것

- **커널 소스 수준의 자료구조 상세는 다루지 않는다.** task_struct 필드 나열이 아니라 "커널이 무엇을 저장해야 스위치가 가능한가"의 수준까지만.
- **실시간 스케줄링(SCHED_FIFO/RR/DEADLINE)은 이름과 용도만.** 일반 서버 워크로드 진단이 목표다.
- **동기화 프리미티브의 구현(futex, 락 알고리즘)은 전용 절을 두지 않는다.** 스레드 문서에서 "경합이 스위치를 유발한다"는 비용 관점으로만 다룬다.
- **특정 파일시스템 비교(ext4 vs btrfs vs xfs)는 하지 않는다.** 저널링은 목적과 보장 수준(메타데이터/데이터)까지만.
- **시그널, 디바이스 드라이버, 윈도우 IOCP는 다루지 않는다.** IOCP는 completion 모델의 선례로 이름만 언급한다.

## 3. 문서별 상세 기획

본문 3편은 콘텐츠 집필 지침의 기본 구조(학습 목표 → 배경 → 핵심 개념 → 실무 관점 → 더 깊이 → 정리 → 확인 문제 → 참고 자료)를 따르고, 45~90분 분량으로 작성한다. 00 인트로는 같은 학습 흐름을 유지하되 10~15분 분량의 오리엔테이션으로 구성한다.

### `00-introduction.md` — 애플리케이션 아래의 계약

- **핵심 질문**: 애플리케이션 개발자에게 운영체제의 동작 모델이 왜 필요한가?
- **도입 사례**: p99 지연이 급증한 서버에서 CPU 사용률, load average, RSS, iowait가 엇갈리는 신호를 보이는 하나의 장애 상황으로 시작한다. 지표별 처방을 바로 내리지 않고 "지표는 원인이 아니라 커널 동작의 결과"라는 문제의식을 세운다.
- **핵심 관점**: 운영체제를 CPU·메모리·장치를 배분하는 자원 관리자이자 프로세스·가상 메모리·파일이라는 추상화를 제공하는 계층으로 소개한다. 스레드 실행, 메모리 할당, 파일 쓰기, 비동기 I/O에서 애플리케이션의 표현과 커널이 실제로 제공하는 계약의 간극을 보여 준다.
- **실무 연결**: 성능·장애 진단, 동시성·용량 설계, 데이터 완료·내구성 판단이라는 세 순간에 OS 모델이 필요한 이유를 설명한다.
- **학습 지도**: 01은 "누가 언제 CPU를 쓰는가", 02는 "주소와 실제 메모리가 언제 연결되는가", 03은 "데이터와 I/O 요청이 어디를 거쳐 완료되는가"라는 질문으로 연결한다.
- **범위 경계**: 용어와 내부 구조를 미리 가르치거나 별도 실습을 추가하지 않는다. 프로세스 정의, 페이지 테이블, epoll 등의 설명은 본문에 위임한다.

### `01-processes-and-scheduling.md` — 실행의 단위와 스케줄러

- **핵심 질문**: 코어 8개에서 스레드 800개가 어떻게 도는가? 그 비용은 어디서 지불되는가?
- **핵심 개념 뼈대**
  - 프로세스 vs 스레드: 주소 공간·파일 디스크립터의 공유 여부가 만드는 차이, 커널 관점에서는 둘 다 스케줄링 단위(태스크)라는 통일된 모델
  - `fork`/`exec`: 복제 후 교체라는 UNIX 모델, CoW 덕분에 fork가 싼 이유(상세는 02)
  - 시스템 콜과 모드 전환: 유저→커널 경계 넘기의 비용, 컨텍스트 스위치와 구분되는 별개 개념이라는 점
  - 컨텍스트 스위치: 무엇을 저장·복원하는가(직접 비용), 캐시·TLB 워밍업 상실(간접 비용 — ch-7 연결), 자발적(블로킹) vs 비자발적(타임슬라이스 소진) 스위치
  - 스케줄링: 타임 셰어링의 목표(공정성 vs 지연), 우선순위와 nice, CFS의 가상 런타임 개념과 EEVDF로의 전환 방향, run queue
  - load average의 정확한 의미: Linux는 uninterruptible 태스크를 포함한다 — CPU 포화와 I/O 대기를 구분해 읽는 법
- **실무 연결**: 스레드 풀 크기 판단(CPU 바운드 vs I/O 바운드), oversubscription이 지연을 키우는 구조, `/proc/[pid]/status`의 voluntary/nonvoluntary_ctxt_switches로 워크로드 성격 읽기, pidstat·vmstat의 cs 지표
- **관찰 예제**: CPU 바운드 작업을 코어 수 이하/이상의 스레드로 돌려 처리량과 비자발적 스위치 수의 관계를 관찰한다(정밀 측정은 실습으로 위임).
- **경계 조건**: 스레드를 늘려도 처리량이 늘지 않고 지연만 커지는 포화 지점, 컨테이너 CPU 제한(quota) 아래에서 스레드 수 판단이 달라지는 사례(상세는 ch-12), load average가 높은데 CPU는 놀고 있는 경우(I/O 대기)

### `02-virtual-memory.md` — 가상 메모리라는 착시

- **핵심 질문**: 프로세스가 보는 주소와 물리 메모리는 어떻게 연결되며, "메모리 사용량"이라는 숫자들은 각각 무엇을 세는가?
- **핵심 개념 뼈대**
  - 가상 주소 공간: 프로세스별 독립 주소 공간이라는 추상화가 사는 문제(격리, 연속성 착시, 공유)
  - 주소 변환: 다단계 페이지 테이블, TLB가 변환을 캐시하는 구조(ch-7의 캐시 모델 재사용), TLB miss·flush 비용
  - demand paging: 할당은 약속이고 페이지 폴트가 실제 지급이라는 모델, minor fault(매핑만) vs major fault(디스크 I/O)
  - `mmap`: 파일 매핑과 익명 매핑, 페이지 캐시와의 연결(03으로 이어짐), ch-6 할당자가 mmap을 쓰는 지점 회수
  - copy-on-write: fork가 싼 이유, 쓰기 순간의 폴트와 복사
  - overcommit과 OOM killer: 약속(virtual)이 지급 능력(물리+스왑)을 초과할 수 있는 정책, OOM 판정과 점수
  - 스왑과 thrashing: working set이 물리 메모리를 넘을 때의 급락, "스왑 사용 = 장애 전조"가 아닌 경우
  - 지표 해석: VSZ/RSS/PSS/shared의 구분, 페이지 캐시가 "사용 중 메모리"로 보이는 문제(free 명령의 available)
- **실무 연결**: 컨테이너 메모리 제한과 OOM(cgroup 상세는 ch-12로), leak처럼 보이지만 아닌 것(할당자 보유분 — ch-6, 페이지 캐시), major fault가 만드는 지연 스파이크 진단
- **관찰 예제**: (1) 큰 할당 직후와 실제 쓰기 후의 RSS 차이로 demand paging 관찰, (2) fork 후 부모·자식의 RSS·shared 변화로 CoW 관찰, (3) `/usr/bin/time -v`(또는 getrusage)로 mmap 파일 읽기의 major/minor fault 수 확인
- **경계 조건**: overcommit이 성립시키는 편리함과 OOM kill이라는 청구서, huge page가 필요해지는 TLB 한계(이름 수준), 페이지 캐시를 포함해 "메모리가 부족하다"고 오판하는 사례

### `03-file-systems-and-io.md` — 파일 I/O 경로와 I/O 모델

- **핵심 질문**: `write()`가 리턴했을 때 데이터는 어디에 있는가? 하나의 스레드로 만 개의 연결을 어떻게 처리하는가?
- **핵심 개념 뼈대**
  - VFS: 파일시스템 위의 공통 인터페이스, 파일 디스크립터가 가리키는 것(열린 파일 테이블)
  - 페이지 캐시: read의 캐시 적중과 read-ahead, write의 write-back(리턴 ≠ 디스크 도달), dirty page와 flush 시점
  - `fsync`와 내구성 계약: 어디까지 내려가야 "저장됐다"인가, 저널링이 지키는 것(메타데이터 정합성)과 지키지 않는 것(기본 모드에서의 데이터), DB의 WAL이 이 계약 위에 선다는 예고(ch-11)
  - 블로킹 I/O와 thread-per-connection: 모델의 단순함과 연결당 스레드 비용(스택 메모리, 스위치 — 01 회수)
  - readiness 모델: 논블로킹 fd + 다중화, select/poll의 O(n) 한계와 epoll의 구조(관심 집합 등록, 준비된 것만 반환), level vs edge trigger의 계약 차이
  - completion 모델: io_uring의 제출·완료 큐(SQ/CQ) 구조, readiness와의 근본 차이(준비 알림 vs 완료 알림), 시스템 콜 횟수 절감
  - 정규 파일의 함정: 디스크 파일은 "항상 ready"라 epoll이 무의미 — libuv가 파일 I/O만 스레드 풀로 우회하는 이유, C10K 문제의 역사적 맥락
- **실무 연결**: 이벤트 루프 서버에서 CPU 바운드·동기 파일 작업이 전체를 멈추는 구조적 이유, "비동기 = 빠르다"가 아니라 "적은 스레드로 많은 동시성"이라는 정확한 이득, 페이지 캐시 때문에 실제보다 빨라 보이는 I/O 벤치마크
- **관찰 예제**: (1) strace로 Node.js 서버의 epoll_ctl/epoll_wait 호출 관찰, (2) 쓰기 후 `/proc/meminfo`의 Dirty 증가와 sync 후 감소 관찰, (3) fsync 유무에 따른 쓰기 지연 차이 실측
- **경계 조건**: fsync가 리턴해도 배신당하는 사례(디스크 캐시, 과거 fsync 오류 처리 문제 — "fsyncgate"는 사례 이름으로만), edge trigger에서 버퍼를 끝까지 읽지 않으면 이벤트가 다시 오지 않는 함정, epoll이 이기지 않는 저동시성 구간

## 4. 문서 간 의존 관계

```text
00 문제의식·학습 지도 ──▶ 01 태스크·스위치 비용 ──────▶ 03 thread-per-conn vs 이벤트 루프의 비용 논거
                              │
                              └──▶ 02 주소 공간(스위치가 바꾸는 것) ──▶ 03 페이지 캐시(mmap·write-back의 기반)
                                          │
                                          └──▶ ch-6 할당자 경계 회수, ch-7 TLB 회수
```

- 00은 세 본문을 읽어야 할 실무적 이유와 핵심 질문만 세우고, 동작 원리 설명은 01~03에 위임한다.
- 01의 컨텍스트 스위치 비용은 03에서 thread-per-connection 모델의 확장 한계를 정량적으로 논증하는 근거다.
- 02의 페이지 캐시·mmap은 03의 파일 I/O 경로 서술의 전제다. 02를 읽지 않으면 write-back을 이해할 수 없으므로 순서를 유지한다.
- 02는 ch-6 02(할당자와 커널 경계)와 ch-7 02(TLB·캐시)의 예고를 회수한다. 집필 시점에 해당 문서가 존재하면 상대 링크, 없으면 문장 위임으로 처리한다.
- 03의 fsync·내구성은 ch-11, epoll은 ch-9의 소켓 프로그래밍으로 이어진다. 아직 없는 챕터로는 문장으로만 위임한다.

## 5. 실습 과제 기획 (`exercises/ch-8/`)

ROADMAP 산출물: "컨텍스트 스위치 비용과 I/O 모델(블로킹, epoll 계열)별 처리량을 직접 측정하고 시스템 지표로 차이의 원인을 설명한다."

### 환경 결정

- **Linux 환경이 필수다**(epoll, `/proc`, taskset). 개발 머신이 macOS이므로 실습은 Linux 컨테이너 또는 VM에서 실행하는 것을 기준으로 작성한다. 시스템 콜·`/proc` 관찰은 컨테이너로 충분하다(ch-7의 PMU 카운터와 달리 특권 하드웨어 접근이 없음). 측정 노이즈를 줄이는 조건(코어 고정, 다른 부하 격리)을 절차에 명시한다.
- **언어**: 측정 대상 프로그램은 **C**로 작성한다. 컨텍스트 스위치·I/O 모델 비교는 런타임(GC·JIT·libuv)이 끼면 원인 분리가 불가능하기 때문이다. 비교군으로 Node.js 서버를 추가해 "libuv = epoll 사용자"임을 strace로 확인한다. 부하 생성 클라이언트는 측정 재현성을 위해 자체 작성한다(동시 연결 수·요청 크기 제어).
- **미결**: Linux 환경의 구체 형태(Docker vs Lima VM)와 C 빌드 구성은 실습 문서 작성 시점에 결정한다. ch-7 실습의 Linux 환경 결정과 일치시킨다.

### Part A — 컨텍스트 스위치와 모드 전환 비용 분리

1. **모드 전환 단독**: 가장 싼 시스템 콜(예: 무의미한 `read`/`getpid` 계열) 반복으로 유저↔커널 왕복 비용 측정. vDSO로 시스템 콜이 아닌 것(gettimeofday)과 대조
2. **컨텍스트 스위치**: 두 프로세스(또는 스레드)를 같은 코어에 고정(taskset)하고 pipe ping-pong으로 왕복당 비용 측정 — 모드 전환 비용을 빼서 스위치 비용을 분리
3. **간접 비용 관찰**: ping-pong 사이에 working set 크기가 다른 메모리 접근을 끼워 스위치 후 캐시 재워밍업 비용이 커지는 것을 관찰(ch-7 연결)
4. `/proc/[pid]/status`의 ctxt_switches 카운터로 측정 횟수와 스위치 횟수가 일치하는지 교차 검증

### Part B — I/O 모델별 echo 서버 처리량

- **서버 3종**: (1) 블로킹 + thread-per-connection(C, pthread), (2) 논블로킹 + epoll 단일 스레드(C), (3) Node.js `net` 서버(libuv/epoll 대조군)
- **측정 시나리오**: 동시 연결 수를 10 → 100 → 1k → 10k(가능 범위)로 올리며 각 지점에서 처리량, 지연(p50/p99), 프로세스 메모리(RSS), 컨텍스트 스위치 수(pidstat), CPU 사용률(user/sys 분리)을 기록한다.
- **분석 포인트**: thread-per-connection의 스레드 수 증가가 메모리·스위치·sys CPU에 나타나는 방식, epoll 서버가 저동시성에서 이점이 없는(또는 지는) 구간, Node 서버의 strace에서 epoll_wait 확인
- **선택 확장**: io_uring 버전 echo 서버를 추가해 시스템 콜 횟수(strace -c) 차이를 관찰

### 리포트와 완료 기준 초안

- 모드 전환 vs 컨텍스트 스위치 비용을 분리한 수치와 측정 방법의 한계를 서술한다.
- 동시 연결 수에 따른 모델별 처리량·지연·메모리·스위치 곡선을 제시하고, 교차 지점(어느 규모부터 이벤트 루프가 이기는가)을 시스템 지표로 설명한다.
- 한 명령(또는 스크립트 하나)으로 각 측정을 재현할 수 있고, 실행 환경(커널 버전, CPU, 컨테이너/VM 여부, 코어 고정 설정)을 명시한다.
- 상세 완료 기준은 실습 문서 작성 시 확정한다.

## 6. 조사 노트 — 1차 자료 후보

집필 시 아래를 우선 확인하고, 커널 버전에 의존하는 동작(스케줄러, overcommit 기본값)은 인용 시점의 커널 문서로 재검증한다.

- **OS 일반**: Arpaci-Dusseau *Operating Systems: Three Easy Pieces*(무료 공개, 가상 메모리·스케줄링·I/O의 표준 교재 — 챕터 전체의 뼈대 참조), Love *Linux Kernel Development*(Linux 구현 관점 보조)
- **Linux 공식**: man pages(epoll(7), select(2), mmap(2), fork(2), fsync(2), proc(5)), kernel.org 문서(CFS/EEVDF 스케줄러 문서, overcommit-accounting, cgroup은 ch-12와 공유)
- **스케줄러**: CFS 설계 문서와 EEVDF 전환 관련 커널 문서·LWN 기사(LWN은 보조 검증용), Brendan Gregg "Linux Load Averages: Solving the Mystery"(uninterruptible 포함의 역사적 경위 — 자체 확인 후 인용)
- **I/O 모델**: Dan Kegel "The C10K problem"(역사적 맥락), Jens Axboe "Efficient IO with io_uring"(io_uring 설계 문서), libuv 공식 설계 문서(스레드 풀과 이벤트 루프의 역할 분담)
- **내구성**: fsync 관련 POSIX 규정과 Linux man page의 주의사항, PostgreSQL fsync 오류 처리 논의(fsyncgate — 내구성 계약의 경계 사례로, 상세는 ch-11과 분담)
- **측정 도구**: Brendan Gregg *Systems Performance* 2판(지표 해석·측정 방법론), pidstat·vmstat·strace 공식 문서

### 통념 검증 목록 (본문에서 정면으로 다룰 것)

- "load average는 CPU 사용률이다" → Linux에서는 실행 대기 + uninterruptible(주로 I/O 대기) 태스크 수다. 1.0의 의미는 코어 수에 상대적이다.
- "write()가 리턴하면 디스크에 저장된 것이다" → 페이지 캐시까지만 보장된다. 내구성은 fsync 계약이고, 그마저 경계 사례가 있다.
- "비동기 I/O는 블로킹 I/O보다 빠르다" → 단일 요청 지연은 오히려 비슷하거나 나쁠 수 있다. 이득은 속도가 아니라 적은 스레드로 감당하는 동시성이다.
- "스레드는 가벼우니 많이 만들어도 된다" → 스택 메모리와 스위치 비용은 연결 수에 비례해 쌓이고, 포화 후에는 지연만 커진다.
- "컨텍스트 스위치 비용은 마이크로초 수준이라 무시해도 된다" → 직접 비용 뒤에 캐시·TLB 재워밍업이라는 더 큰 간접 비용이 있다(ch-7 근거).
- "epoll은 모든 I/O를 비동기로 만든다" → 정규 파일에는 readiness 모델이 무의미하다. libuv가 파일 I/O를 스레드 풀로 우회하는 이유다.
- "스왑이 잡혀 있으면 장애 신호다" → 비활성 페이지의 스왑은 정상일 수 있다. 문제는 스왑 사용량이 아니라 스왑 in/out의 지속적 발생(thrashing)이다.

## 7. 작성 순서와 검증 계획

1. 독자는 `00 → 01 → 02 → 03` 순서로 읽는다. 00이 문제의식과 학습 지도를 세우고, 태스크·스위치 모델(01)과 주소 공간·페이지 캐시(02)가 I/O 모델 논의(03)의 전제가 된다.
2. 모든 관찰 예제는 Linux 환경(컨테이너 또는 VM, 커널 버전 명시)에서 직접 실행해 확인한 뒤 수치를 기재한다. macOS와 동작이 다른 지점(예: `/proc` 부재)은 본문에 명시한다.
3. POSIX가 보장하는 것과 Linux 구현 세부(overcommit 정책, 스케줄러)를 구분해 서술하고, 커널 버전 의존 동작은 기준 버전을 명시한다.
4. 확인 문제는 지표 해석(주어진 vmstat/loadavg/RSS 스냅샷에서 원인 추론)과 아키텍처 선택(워크로드 조건에서 I/O 모델 판단)처럼 진단·판단형으로 만든다.
5. `docs/ch-8/`을 처음 만들 때 `docs/.vitepress/navigation.ts`의 `PHASE_LABELS`에 챕터 8 레이블을 등록하고 빌드로 nav·sidebar 노출을 확인한다(저장소 운영 지침 적용).
