# 챕터 2 기획 — 계산 이론

[ROADMAP.md](../ROADMAP.md)의 챕터 2(`docs/ch-2/`, 인트로 1편과 본문 4편)을 집필하기 위한 상세 기획이다.
범위·경로가 ROADMAP과 어긋나면 ROADMAP을 우선한다.

## 1. 챕터의 관점

독자는 5년차 이상 개발자다. 정규식·파서·타입 검사기·정적 분석기·최적화 도구를 사용해 본 경험은 있지만, 각 도구가 다룰 수 있는 문제의 경계를 계산 모델로 설명하지는 못할 수 있다.
이 챕터는 계산 이론을 정의와 증명의 목록으로 가르치지 않고, 실무 도구의 표현력·정확성·성능 경계를 판단하는 네 질문으로 조직한다.

1. **논리 — 무엇을 정확히 표현하고 추론할 수 있는가?** 명세를 논리식으로 옮기고, 만족 가능성·타당성·함의를 구분한다.
2. **형식 언어 — 어떤 입력 구조를 어떤 기계가 인식할 수 있는가?** 언어의 구조와 인식기의 계산 능력을 함께 판단한다.
3. **계산 가능성 — 그 판정을 수행하는 알고리즘이 항상 존재하는가?** 완벽한 분석기가 원리적으로 불가능한 지점을 식별한다.
4. **복잡도 — 알고리즘이 존재해도 현실적인 자원으로 실행 가능한가?** exact·근사·휴리스틱·솔버 사이에서 보장과 비용을 선택한다.

각 문서는 실무 실패 사례에서 출발해 동작 모델을 세우고, 핵심 논증으로 경계를 확인한 뒤, 실무 대응과 실행 가능한 미니 실험으로 마무리한다. 정의는 정확히 제시하되 증명은 경계를 이해하는 데 필요한 핵심 단계까지만 다룬다.

## 2. 범위 결정

### 다루는 것

- 명제·술어 논리의 구문과 의미, 만족 가능성·타당성·함의, 증명 체계의 건전성·완전성, CNF와 SAT
- 촘스키 계층, 정규 언어와 유한 오토마타, DFA·NFA 동등성, Thompson 구성, CFG·PDA가 필요한 경계
- 결정 가능성과 인식 가능성, 튜링 기계, 처치-튜링 논제, 정지 문제, Rice 정리와 정적 분석의 한계
- P·NP·NP-완전·NP-난해, 다항 시간 환원, exact·근사·휴리스틱·SAT/ILP의 선택 기준
- 같은 정규식 언어를 처리하는 lockstep NFA와 naïve backtracking 실행 전략의 비용 차이, ReDoS의 구조적 원인

### 다루지 않기로 결정한 것

- **깊은 모델 이론과 증명론은 다루지 않는다.** 01은 개발자가 명세·제약·도구의 결과를 해석할 수 있는 논리 어휘를 세우는 데 집중한다.
- **오토마타 최소화 알고리즘과 문맥 의존 문법의 세부 구현은 제외한다.** 촘스키 계층에서 위치와 표현력은 설명하지만 실습은 정규 언어에 집중한다.
- **튜링 기계 구현 자체를 과제로 두지 않는다.** 전이표 작성보다 보편 계산과 결정 불가능성의 의미가 실무 판단에 더 중요하다.
- **복잡도 동물원 전체를 나열하지 않는다.** co-NP·PSPACE·EXPTIME은 P·NP의 경계를 오해하지 않게 하는 보조 설명 이상으로 확장하지 않는다.
- **PCP 정리, 회로 복잡도, 양자 계산은 제외한다.** 네 문서의 학습 목표와 최종 정규식 엔진 실습에 직접 기여하지 않는다.

## 3. 문서별 상세 기획

본문 4편은 콘텐츠 집필 지침의 기본 구조(학습 목표 → 배경 → 핵심 개념 → 실무 관점 → 더 깊이 → 정리 → 확인 문제 → 참고 자료)를 따르고, 45~90분 분량으로 작성한다. 각 본문에는 10~20분 안에 실행·관찰할 수 있는 TypeScript 미니 실험을 하나씩 둔다. 00 인트로는 10~15분 분량의 오리엔테이션으로 구성한다.

### `00-introduction.md` — 도구가 풀 수 있는 문제에는 경계가 있다

- **핵심 질문**: 정규식·파서·분석기·솔버가 어떤 문제는 정확히 풀고 어떤 문제는 원리적으로 풀지 못하는 이유는 무엇인가?
- 실무 요구가 표현 가능성, 계산 가능성, 현실적인 계산 비용이라는 서로 다른 경계에서 실패할 수 있음을 소개한다.
- 논리 → 형식 언어 → 계산 가능성 → 복잡도로 이어지는 네 질문을 학습 지도로 제시하고 정의·증명은 01~04에 위임한다.

### `01-mathematical-logic.md` — 프로그램을 명세하고 추론하는 언어

챕터 전체가 사용할 구문·의미·판정·증명 어휘를 세우고, 04의 SAT 문제로 이어지는 출발점이다.

- **핵심 질문**: 자연어 요구사항을 기계가 검사할 수 있는 명세로 바꾸면 무엇을 얻고 무엇을 잃는가?
- **핵심 개념 뼈대**
  - 구문(syntax)과 의미(semantics)의 분리: 논리식 자체와 해석·모델을 구분한다
  - 명제 논리: 진리 배정, 만족 가능성(satisfiability), 타당성(validity), 논리적 함의(entailment)의 차이
  - 증명 체계: 추론 규칙, 건전성(soundness)과 완전성(completeness)이 도구의 신뢰성과 탐지 범위를 뜻하는 방식
  - 정규형: CNF·DNF의 목적, CNF 변환과 식 크기 문제, Tseitin 변환은 의미를 보존하는 새 변수 도입 관점으로 소개
  - 술어 논리: 술어·관계·전칭 및 존재 양화사, 양화사 순서가 명세의 의미를 바꾸는 사례
  - SAT 연결: 논리식을 만족하는 모델 찾기와 제약 조건 해결의 관계. DPLL/CDCL은 04의 solver 전략으로 연결하고 여기서는 입출력 모델까지만
- **실무 연결**
  - 타입을 명제로, 프로그램을 증명으로 보는 관점은 ch-4로 연결한다
  - SQL `NULL`의 3값 논리가 고전 명제 논리의 진리표와 달라지는 지점을 조건식 사례로 확인한다
  - feature flag·배포 대상·의존 버전의 제약을 논리식으로 모델링하고 모순 원인을 설명한다
- **미니 실험**: 소수의 배포 제약을 명제식으로 작성하고 모든 진리 배정을 열거해 만족 모델과 모순되는 배정을 관찰한다. brute force가 빠르게 커지는 현상은 04로 연결한다.
- **경계 조건**: 형식화되지 않은 전제는 검증할 수 없고, 논리식이 만족 가능하다는 사실이 현실 요구사항을 올바르게 모델링했다는 뜻은 아니다. SQL의 3값 논리를 2값 논리처럼 다루는 오류도 함께 분석한다.

### `02-formal-languages-and-automata.md` — 언어의 구조와 인식기의 경계

최종 실습의 직접적인 이론 기반이다. “정규식”이라는 표기, 정규 언어라는 수학적 대상, 실제 정규식 엔진 구현을 명확히 분리한다.

- **핵심 질문**: 입력의 어떤 구조는 유한한 상태만으로 인식할 수 있고, 어떤 구조는 별도의 스택이나 더 강한 계산 모델을 요구하는가?
- **핵심 개념 뼈대**
  - 알파벳·문자열·언어, 생성기(grammar)와 인식기(automaton)의 대응
  - 촘스키 계층: Type 3부터 Type 0까지의 포함 관계를 지도처럼 제시하되 정규 언어와 문맥 자유 언어에 집중
  - DFA: 상태·전이·시작 상태·수용 상태, 입력당 하나의 전이라는 결정성의 의미
  - NFA와 epsilon transition: 여러 경로를 “동시에” 유지하는 상태 집합 실행, subset construction을 통한 DFA와의 표현력 동등성
  - 정규 표현식에서 NFA로: 연결·선택·반복을 조합하는 Thompson construction
  - 정규 언어의 경계: pumping lemma의 논증 구조를 따라가며 임의 깊이 중첩과 동일 개수 대응이 유한 상태로 어려운 이유를 설명
  - CFG와 PDA: 재귀 규칙과 스택이 중첩 구조를 인식하게 하는 방식. 실제 파싱 알고리즘은 ch-5로 위임
- **실무 연결**
  - 정규 언어, POSIX·ECMAScript 등의 정규식 문법, 특정 엔진의 실행 전략을 같은 것으로 취급하지 않는다
  - backreference처럼 정규 언어의 범위를 넘어서는 확장과 lookaround 같은 기능은 스펙·엔진별 기능으로 구분한다
  - “정규식으로 HTML을 파싱할 수 없다”는 구호 대신 입력 문법, 허용할 깊이, 엔진 확장, 신뢰 경계를 기준으로 판단한다
- **미니 실험**: `(a|aa)+b`에 끝의 `b`가 없는 입력을 넣고, lockstep 상태 집합과 깊이 우선 탐색의 방문 횟수를 한 단계씩 출력한다.
- **경계 조건**: 정규식 문법이 정규 언어를 표현하더라도 실행 알고리즘은 지수 시간이 될 수 있다. 반대로 실제 엔진의 최적화 결과가 이론적 최악 사례를 항상 재현한다고 가정하지 않는다.

### `03-computability.md` — 계산할 수 없는 문제와 분석 도구의 한계

“아직 좋은 알고리즘을 못 찾은 문제”와 “어떤 알고리즘도 존재할 수 없는 문제”를 구분하는 문서다.

- **핵심 질문**: 모든 프로그램에 대해 항상 끝나면서 정답을 주는 완벽한 분석기를 만들 수 없는 이유는 무엇인가?
- **핵심 개념 뼈대**
  - 결정 문제, 결정 가능(decidable), 인식 가능(recognizable), 종료하지 않을 수 있는 인식기와 항상 종료하는 판정기의 차이
  - 튜링 기계: 테이프·헤드·상태 전이를 최소 모델로 소개하고, 세부 구현보다 프로그램과 입력을 데이터로 다루는 보편 계산에 집중
  - 처치-튜링 논제: 효과적으로 계산 가능한 것에 관한 논제로서 수학적으로 증명된 정리와 구분
  - 정지 문제: 완벽한 `halts(program, input)`을 가정한 뒤 자기 참조 프로그램으로 모순을 만드는 대각선 논증
  - 환원에 의한 결정 불가능성: 이미 불가능한 문제를 새 문제의 판정기로 풀 수 있음을 보여 불가능성을 전달하는 방법
  - Rice 정리: 프로그램이 계산하는 함수의 비자명한 의미적 속성을 일반적으로 완벽히 판정할 수 없다는 의미
- **실무 연결**
  - 정적 분석기가 종료·건전성·완전성을 모두 만족하지 못하고 false positive, false negative, `unknown` 중 하나를 감수하는 이유
  - 프로그램 동등성, 모든 입력에서의 예외 부재, 악성 행위 여부를 완벽하게 판정한다는 요구사항을 검토한다
  - 시간 제한, 추상화, 탐색 깊이 제한, 사용자 annotation 같은 현실적 타협이 어떤 보장을 약화하는지 비교한다
- **미니 실험**: 제한된 명령어를 실행하는 작은 인터프리터에 step budget을 두고 `halted`, `unknown`을 반환한다. budget 초과가 비종료의 증명이 아님을 서로 다른 프로그램으로 확인한다.
- **경계 조건**: 특정 언어 조각, 유한 상태, 제한된 입력 크기에서는 판정 가능할 수 있다. 정지 문제를 근거로 모든 정적 분석이 무의미하다고 결론 내리는 오류를 반례와 함께 다룬다.

### `04-complexity-classes.md` — 풀 수 있지만 감당하기 어려운 문제

결정 불가능성과 계산 비용의 폭발을 구분하고, 어려운 문제를 만났을 때의 실무 선택 기준을 세운다.

- **핵심 질문**: 문제를 정확히 푸는 비용이 감당할 수 없을 때, 어떤 근거로 exact·근사·휴리스틱·솔버 중 하나를 선택하는가?
- **핵심 개념 뼈대**
  - 문제와 인스턴스, 입력 크기, 결정 문제와 최적화 문제, worst case를 기준으로 클래스를 정의하는 이유
  - P: 결정론적 다항 시간에 풀 수 있는 결정 문제
  - NP: 주어진 인증서(certificate)를 다항 시간에 검증할 수 있는 결정 문제. “비결정론적 다항 시간”과 “다항 시간에 풀 수 없음”이라는 흔한 오해를 바로잡는다
  - NP-완전과 NP-난해: NP에 속하는지와 모든 NP 문제만큼 어려운지를 분리한다
  - 다항 시간 환원: `A ≤p B`의 방향과, B의 알고리즘으로 A를 푼다는 의미를 작은 논리식 변환 사례로 연습한다
  - SAT에서 출발하는 난이도 전달: Cook-Levin 정리의 의미를 소개하고 전체 증명은 생략한다. 스케줄링·집합 덮개·TSP로 문제 구조를 연결한다
- **실무 전략**
  - 작은 입력·엄격한 최적성 요구: 완전 탐색, dynamic programming, branch-and-bound 같은 exact 접근
  - 최악 성능 보장이 필요한 대규모 입력: approximation ratio가 있는 근사 알고리즘
  - 평균적인 해 품질과 빠른 응답이 중요한 경우: greedy, local search, metaheuristic
  - 복잡한 제약과 반복되는 모델 변경: SAT·SMT·ILP solver에 모델링하고 timeout·최적성 상태·지원되는 경우의 unsat core를 결과 계약에 포함
- **미니 실험**: 작은 집합 덮개 문제에서 완전 탐색과 greedy를 비교한다. 입력 크기에 따른 탐색 상태 수와 실행 시간, 선택된 집합 수를 함께 기록해 속도와 해 품질을 동시에 본다.
- **경계 조건**: 지수 시간으로 동작하는 구현이 곧 NP-난해성을 뜻하지 않는다. ReDoS는 정규식 매칭 자체의 난이도가 아니라 다항 시간 알고리즘이 존재하는 문제에 중복 탐색을 허용한 실행 전략의 실패 사례로 정리한다.

## 4. 문서 간 의존 관계

```text
01 논리·SAT ───────────────────────────────▶ 04 NP·환원·솔버
     │
     └── 구문과 의미의 구분
                    ▼
02 언어·오토마타 ──▶ 통합 NFA 실습 ──▶ 실행 전략의 비용 비교
          │                                    │
          └── 인식 가능성 ──▶ 03 결정 가능성 ──┴──▶ 04 계산 가능하지만 비싼 문제
```

- 01의 구문·의미 구분은 02의 문법·언어·인식기 구분에 재사용하고, 만족 가능성과 CNF는 04의 SAT·환원·solver 논의로 이어진다.
- 02는 최종 정규식 엔진 실습의 상태 모델과 Thompson construction을 제공한다.
- 03은 “알고리즘의 존재 여부”를, 04는 “존재하는 알고리즘의 자원 비용”을 다룬다. 두 경계를 섞지 않도록 같은 정적 분석·정규식 사례를 서로 다른 관점에서 다시 판별한다.
- 04는 ch-1에서 예고한 NP-난해 문제를 이어받는다. 실제 렉싱·파싱 구현과 정적 분석 알고리즘은 ch-5로 넘긴다.
- 아직 집필되지 않은 후속 챕터는 링크 대신 “챕터 N에서 다룬다”는 문장으로 위임하고 TODO 링크를 남기지 않는다.

## 5. 실습 과제 기획 (`exercises/ch-2/`)

ROADMAP 산출물: “NFA 기반 정규식 엔진을 구현하고 백트래킹 엔진과 성능을 비교해 ReDoS가 발생하는 구조적 이유를 분석한다.”

### 환경과 문법 범위

- **환경**: TypeScript + Node.js 24, `node:test`, 별도 런타임 의존성 없음. 실습 코드는 `exercises/ch-2/` 아래 pnpm 워크스페이스 패키지로 둔다.
- **매칭 의미**: 전체 문자열 일치(full match)만 지원한다. 부분 검색, 캡처 결과, 우선순위·탐욕성의 외부 관찰 차이는 제외한다.
- **지원 문법**: ASCII literal, 연결, 선택(`|`), 그룹(`(...)`), `*`, `+`, `?`.
- **제외 문법**: `.`, 문자 클래스, anchor, escape, counted repetition, lookaround, backreference, Unicode mode. 핵심 상태 전이와 실행 전략 비교를 흐리지 않기 위해 제외한다.
- **파싱**: infix 정규식을 postfix 또는 AST로 변환하는 코드는 starter로 제공한다. 학습자는 파서가 아니라 NFA 구성과 실행기를 구현한다.

### 공통 NFA와 두 실행기

- NFA 상태는 문자 하나를 소비하는 `Char`, epsilon 분기를 만드는 `Split`, 성공을 나타내는 `Match`의 세 종류로 제한한다.
- Thompson construction으로 지원 문법의 각 연산을 fragment로 만들고 patch list로 연결한다.
- **lockstep NFA 실행기**: 현재 입력 위치의 epsilon closure를 구하고 상태 ID로 중복을 제거한 뒤, 문자 하나를 소비할 때마다 다음 상태 집합을 만든다.
- **naïve backtracking 실행기**: 동일한 NFA 그래프와 입력을 사용하되 `Split`의 각 분기를 깊이 우선으로 탐색하며 `(stateId, inputIndex)`를 memoize하지 않는다.
- 같은 NFA를 공유해 컴파일 결과나 문법 차이가 아니라 실행 순서와 중복 제거 전략만 비교하게 한다.

### 계측 인터페이스

두 실행기는 공통 결과에 다음 값을 기록한다.

- `matched`: 전체 문자열 일치 여부
- `steps`: 상태를 방문한 총횟수
- `branches`: `Split`에서 탐색한 분기 수
- `maxActiveStates`: lockstep 실행기의 최대 활성 상태 수. backtracking 결과에서는 기록하지 않음
- `maxStackDepth`: backtracking 실행기의 최대 탐색 깊이. lockstep 결과에서는 기록하지 않음
- `elapsedNs`: `process.hrtime.bigint()` 기준 경과 시간
- `aborted`: `maxSteps`를 초과해 중단했는지 여부

`maxSteps`는 두 실행기 모두 필수 입력으로 받아 실험이 항상 종료되게 한다. 성능 결론은 노이즈가 있는 시간값 하나가 아니라 결정적인 `steps` 증가와 반복 측정한 시간값을 함께 근거로 삼는다.

### 검증과 측정 시나리오

- **의미적 동등성**: `{a,b}`에서 길이 0~6의 모든 문자열을 생성하고, 정상 패턴 corpus에 대해 두 실행기의 `matched`가 같은지 확인한다.
- **기본 기능**: 빈 문자열, literal, 연결, 선택, `*`·`+`·`?`, 중첩 그룹, 실패 경로, epsilon cycle을 단위 테스트한다.
- **정상 입력**: 성공이 빠르게 결정되는 패턴과 입력에서 두 실행기의 기본 비용을 비교한다.
- **공격 입력**: `(a|aa)+b`에 `a^n`을 넣어 late failure와 모호한 반복이 결합될 때의 증가를 관찰한다.
- **구조 분리 실험**: 같은 패턴에 마지막 `b`가 있는 입력, 모호한 선택이 없는 `a+b`, 반복이 없는 `(a|aa)b`를 대조군으로 두어 세 조건 중 무엇이 폭발을 만드는지 분리한다.
- **입력 크기**: 작은 n부터 `maxSteps` 도달 전까지 늘리고, 각 지점에서 최소 5회 측정한 중앙값과 결정적 step count를 기록한다.

### 리포트와 완료 기준

- 지원 문법의 모든 단위 테스트와 두 실행기의 동등성 검사가 통과한다.
- 한 명령으로 correctness test와 benchmark를 각각 재현할 수 있다.
- 입력 길이에 따른 `steps`와 경과 시간을 표 또는 로그 스케일 그래프로 제시한다.
- 모호한 선택, 반복, late failure가 결합해 동일한 `(state, input position)`을 재탐색하는 과정을 NFA 그래프 또는 탐색 트리로 설명한다.
- lockstep 실행기가 상태 중복 제거로 반복 탐색을 접는 이유와 시간·공간 비용을 함께 설명한다.
- 완화책을 시간 제한, 입력 길이 제한, 패턴 제한·검사, 신뢰할 수 없는 패턴 금지, 선형 시간 엔진 선택으로 나누고 각각의 보장과 우회 가능성을 비교한다.
- 실제 Node.js `RegExp` 비교는 선택 과제로 둔다. 엔진 최적화로 자체 backtracking 실행기와 다른 결과가 나올 수 있으며, 이를 이론의 반례로 해석하지 않는다.

## 6. 조사 노트 — 1차 자료 후보

집필 시 아래 원문과 표준을 우선 확인하고, 정규식 문법·엔진 동작·도구 기능처럼 구현과 버전에 의존하는 내용은 인용 시점에 다시 검증한다.

- **논리와 SAT**: Davis–Putnam(1960), Davis–Logemann–Loveland(1962) 원 논문, SAT/SMT solver 공식 문서. SQL의 3값 논리는 ISO/IEC 9075와 사용 DB의 공식 문서를 대조한다.
- **형식 언어**: Chomsky의 형식 문법 계층 원 논문, Kleene의 regular events 논문, Rabin–Scott의 finite automata 논문, Thompson의 1968년 정규식 검색 알고리즘 논문.
- **정규식 표준과 구현**: ECMA-262 `RegExp` 명세, POSIX regular expression 정의, V8·RE2 등 해당 엔진의 공식 설계 문서와 소스. 정규식 문법과 실행 알고리즘을 별도 출처로 검증한다.
- **계산 가능성**: Turing(1936), Church(1936), Rice(1953) 원 논문. 용어와 현대적 정식화는 표준 계산 이론 교재로 교차 검증한다.
- **계산 복잡도**: Cook(1971), Karp(1972) 원 논문, NP-완전성과 근사 알고리즘의 표준 교재. solver 선택 기준은 각 SAT·SMT·ILP 도구의 공식 문서를 확인한다.
- **ReDoS**: CWE-1333, OWASP ReDoS 자료, 실제 엔진의 보안 문서와 알려진 취약 패턴을 보조 자료로 사용한다. 특정 패턴의 시간 복잡도 주장은 자체 계측과 엔진 구현 근거를 함께 제시한다.

### 통념 검증 목록 (본문에서 정면으로 다룰 것)

- “논리식이 만족 가능하면 요구사항이 올바르다” → solver는 모델링된 제약의 일관성만 판정하며 누락된 현실 전제까지 검증하지 않는다.
- “정규식은 모두 정규 언어이며 모든 엔진은 같은 방식으로 실행한다” → 수학적 언어, 표면 문법, 확장 기능, 실행 전략을 구분한다.
- “NFA는 가능한 경로를 스레드처럼 무한히 만든다” → lockstep simulator는 입력 위치별 상태 집합과 중복 제거로 실행한다.
- “정지 문제 때문에 정적 분석은 정확할 수 없다” → 제한된 문제에서는 정확한 판정이 가능하며, 일반 문제에서 보장 세 가지를 동시에 얻을 수 없다는 뜻이다.
- “NP는 non-polynomial의 약자다” → nondeterministic polynomial time이며, P와 NP가 다른지는 아직 증명되지 않았다.
- “지수 시간으로 실행되면 그 문제는 NP-난해하다” → 알고리즘의 비효율과 문제 자체의 하한을 구분한다. naïve 정규식 backtracking을 대표 반례로 사용한다.
- “solver를 쓰면 NP-난해성이 사라진다” → solver는 구조·휴리스틱·학습을 활용해 실용 사례를 잘 풀 수 있지만 최악 경우 보장은 바뀌지 않는다.

## 7. 작성 순서와 검증 계획
