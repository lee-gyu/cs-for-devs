# 챕터 11 기획 — 데이터베이스 시스템 내부와 실행 진단

이 문서는 [ROADMAP.md](../ROADMAP.md)의 챕터 11을 **인트로 1편과 본문 9편**으로 재구성하고, 이후 `docs/ch-11/`과 `exercises/ch-11/`을 집필하기 위한 상세 기획이다. 현재 ROADMAP에는 기존 3편 구성이 남아 있으므로, 본문 집필 전에 이 기획의 §3과 §7을 ROADMAP 및 [PROGRESS.md](../PROGRESS.md)에 먼저 반영한다. 반영 이후 범위·경로가 서로 어긋나면 ROADMAP을 우선한다.

## 1. 챕터의 관점

독자는 SQL을 작성하고 인덱스를 추가하며 `EXPLAIN`을 본 경험이 있지만, 계획의 한 줄을 페이지 접근·버퍼·통계·동시성 비용까지 연결하지 못할 수 있다. 이 챕터는 기능과 명령의 카탈로그가 아니라 **하나의 SQL 요청이 결과와 커밋에 도달하는 경로를 내부 구성 요소로 추적하고, 각 단계의 비용과 보장을 증거로 진단하는 판단 모델**을 세운다.

공통 사례는 [챕터 0의 주문 확정 API](../docs/ch-0/02-cs-in-practice.md)를 이어받는다. 주문·고객·상품·재고 데이터에 다음 조건을 순차적으로 추가한다.

- 데이터가 메모리와 버퍼 풀을 넘어 페이지 I/O가 성능을 지배한다.
- 주문 상태와 고객군의 분포가 치우쳐 옵티마이저의 독립성·균등 분포 가정이 깨진다.
- 조회를 빠르게 하려고 추가한 인덱스가 쓰기 증폭과 페이지 분할을 늘린다.
- 주문 확정과 재고 차감이 동시에 실행되어 대기, 교착, 격리 이상이 드러난다.
- 커밋 과정의 서로 다른 시점에 장애가 발생해 성공 응답과 내구성의 경계를 묻게 된다.
- 비동기 replica에서 직전 주문을 읽지 못하고, 동기 복제에서는 commit latency가 증가한다.
- 특정 고객군과 판매 이벤트가 한 shard에 몰려 저장 공간과 QPS가 함께 편향된다.
- 주문과 재고가 서로 다른 shard에 놓여 한 요청의 원자적 commit이 여러 노드의 상태와 장애에 의존한다.

모든 문서는 다음 판단 루프를 공유한다.

1. 워크로드와 지켜야 할 정확성·격리·내구성 계약을 명시한다.
2. DBMS가 선택할 접근 경로, 물리 연산자와 자원 비용을 먼저 예측한다.
3. 페이지·버퍼 카운터, 실행 계획, 행 수 추정, 대기·로그·복제·shard 상태를 관찰한다.
4. 예측과 관찰이 다른 지점을 저장·실행·동시성·복구·분산 구성 요소에서 찾는다.
5. 인덱스·쿼리·트랜잭션·배치 설계를 바꾸고 같은 증거로 효과와 새 비용을 검증한다.

분석 경로는 다음과 같다.

```text
SQL text
  → parse · bind · rewrite
  → logical plan
  → cost-based optimization
  → physical plan · operators
  → access path
  → buffer manager
  → pages · files

transaction manager · concurrency control · recovery manager는
이 전체 경로의 가시성, 대기, 원자성, 내구성 계약을 통제한다.

local recovery
  → replicated log · replica apply
  → shard routing · data placement
  → distributed concurrency control · atomic commit

workload · data relationship · query pattern · consistency requirement
  → relational / key-value / wide-column / document / graph / search
```

### 제품 중립 원칙

챕터는 특정 DBMS의 동작을 보편 원리처럼 서술하지 않는다. 각 주제는 **공통 계약 → 가능한 구현 전략 → 제품별 사례 → 관찰 가능한 증거** 순서로 설명한다.

- PostgreSQL, InnoDB, SQLite와 LSM 계열은 같은 용어를 쓰더라도 내부 구조와 보장이 다를 수 있다.
- 실행 계획은 제품별 노드 이름을 암기하지 않고 `접근 경로 / 조인 순서·방식 / 추정·실제 행 / 페이지·버퍼 / spill / wait`의 공통 관찰 축으로 정규화한다.
- 엔진이 제공하지 않는 지표는 추정하거나 다른 지표와 동일시하지 않고 `N/A`와 관찰 한계를 명시한다.
- 제품별 설정값과 튜닝 레시피는 원리를 관찰하기 위한 예제로만 사용한다.

### 경계에서만 다루는 것

- 관계형 모델·정규화는 행 폭, 키, 중복, 조인 수가 페이지 밀도와 접근 경로를 바꾸는 지점까지만 다룬다.
- row/column layout, bitmap index, Bloom filter, vectorized execution은 기본 설계의 대안과 경계 조건을 설명하는 비교 사례로 제한한다.
- N+1은 실행 계획 하나의 문제가 아니라 애플리케이션과 DB 경계에서 요청 수가 증폭되는 사례로만 다룬다.
- vacuum, clustered index, index-only scan처럼 제품별 의미가 갈리는 용어는 공통 계약과 차이를 밝힌 뒤 사례로 사용한다.
- DBMS 패러다임 비교는 대표적인 데이터 모델과 워크로드 선택 기준까지만 다루고 제품별 기능·벤치마크 비교로 확장하지 않는다.

### 다루지 않는 것

- 정규형 계산, 함수 종속성 증명과 데이터 모델링 방법론 전체
- SQL 문법, ORM·마이그레이션 도구와 특정 제품의 설정값·명령어 카탈로그
- 공간 인덱스 등 모든 세부 인덱스 종류의 백과사전식 나열
- Raft·Paxos의 알고리즘 상세, CAP와 분산 일관성 모델의 이론적 재설명
- Saga·outbox·보상 트랜잭션과 서비스 워크플로 설계
- 백업·복원 운영 절차, 제품별 고가용성 설정, 관리형 DB·클라우드 제품 선택
- 데이터 웨어하우스·레이크·스트림 처리와 검색 랭킹·데이터 마이닝의 상세 이론

## 5. 문서별 설계

### `00-introduction.md` — 한 SQL 요청의 경로

- 주문 조회가 데이터 증가 후 느려지고, 인덱스 추가 뒤 쓰기 지연과 lock wait가 증가한 사례에서 시작한다.
- SQL 요청이 결과와 커밋에 도달하는 전체 경로를 한 장의 지도로 제시한다.
- `결과가 맞다`, `허용 자원 안에 끝난다`, `동시 실행에서 계약을 지킨다`, `성공 후 장애에도 남는다`를 서로 다른 보장으로 분리한다.
- “느린가, 기다리는가, 디스크에 남지 않았는가?”라는 세 진단 질문으로 본문 지도를 연결한다.

### `01-storage-layout-and-buffer-management.md` — 행에서 페이지 I/O까지

- 논리적 행이 레코드와 슬롯 페이지에 배치되고 RID로 식별되는 과정을 추적한다.
- 가변 길이 레코드, 행 폭, free space와 fragmentation이 페이지 밀도와 I/O를 바꾸는 이유를 설명한다.
- 버퍼 풀의 fetch/pin/unpin/dirty/evict/flush 상태 전이를 세우고 캐시 적중률만으로 충분하지 않은 조건을 다룬다.
- sequential/random access, read-ahead, DB 버퍼와 OS page cache의 중첩을 관찰 한계와 함께 설명한다.
- 실험에서는 접근 trace에 따른 page read, hit, dirty eviction 곡선을 예측·측정한다.

### `02-indexes-and-storage-engines.md` — 접근 경로가 만드는 교환 비용

- B+트리의 fan-out, 높이, leaf 순회와 page split이 point/range read와 쓰기 비용으로 이어지는 과정을 설명한다.
- hash index의 equality 장점과 range/order 경계를 B+트리와 비교한다.
- LSM의 memtable, immutable run/SSTable, compaction, Bloom filter를 B+트리와 read/write/space amplification 축에서 비교한다.
- 기본·보조와 clustered·unclustered를 제품별 이름이 아닌 데이터 배치와 추가 lookup 횟수로 정의한다.
- 복합 인덱스의 열 순서, 선택도, covering/index-only 접근을 “사용된다/안 된다”가 아니라 읽는 범위와 유지 비용으로 판단한다.
- 실험에서는 point/range/write 비율과 데이터 분포를 바꾸며 page read, split, compaction, amplification을 기록한다.

### `03-query-processing-and-execution-plans.md` — SQL에서 물리 연산자로

- parse, bind, rewrite가 각각 구문·이름·동치 변환을 책임지는 경계를 세운다.
- relational operator로 된 논리 계획과 scan·join·sort 구현을 고른 물리 계획을 구분한다.
- iterator 모델을 기본으로 설명하고 materialization·vectorized 실행은 호출·메모리 지역성 트레이드오프로 비교한다.
- nested-loop/hash/merge join, sort와 hash aggregate의 입력 조건, 메모리 사용과 spill 경계를 비교한다.
- 통계, 히스토그램, 상관관계, 선택도와 카디널리티 추정 오차가 access path·join order·join algorithm 선택으로 증폭되는 과정을 추적한다.
- 실행 계획은 추정값과 관측값을 분리하고, 노드별 rows×loops, page/buffer, spill, wait의 지배 비용을 찾아 읽는다.
- 실험에서는 균등/편향/상관 데이터와 인덱스 유무를 한 변수씩 바꾸어 계획을 예측하고 실제 출력과 대조한다.

### `04-transactions-and-concurrency-control.md` — 동시에 맞는 결과를 만드는 법

- ACID를 구호로 나열하지 않고 스케줄, 가시성, 충돌, 복구 구성 요소의 계약으로 분해한다.
- dirty read, non-repeatable read, phantom, lost update, write skew를 실행 순서와 불변식으로 구분한다.
- lock과 latch의 보호 대상·수명·교착 가능성을 구분하고 2PL과 wait-for graph를 연결한다.
- MVCC의 version, transaction/snapshot identifier와 visibility 판정을 단순 상태 모델로 설명한다.
- snapshot 유지가 읽기 대기를 줄이는 대신 오래된 버전, 정리 지연과 저장 공간 비용을 만드는 경계를 다룬다.
- 격리 수준 이름보다 허용되는 이상 현상과 엔진 문서의 실제 보장을 확인하는 절차를 우선한다.
- 실험에서는 두 트랜잭션의 interleaving을 통제해 결과·대기·교착·visibility timeline을 기록한다.

### `05-durability-and-recovery.md` — 커밋과 장애 사이

- data page보다 WAL record가 먼저 안정 저장되어야 하는 불변식을 세운다.
- steal/no-force 정책이 정상 경로의 I/O와 복구의 redo/undo 필요성을 어떻게 바꾸는지 설명한다.
- commit record, fsync, group commit과 성공 응답 시점의 관계를 분리한다.
- checkpoint가 모든 dirty page를 즉시 안전하게 만드는 단순한 저장점이 아니라 복구 탐색 범위를 줄이는 메커니즘임을 설명한다.
- crash와 media corruption을 구분하고 WAL이 해결하지 못하는 손상·백업 경계를 명시한다.
- 실험에서는 WAL append, page flush, commit 사이에 crash point를 주입하고 복구 전 예측과 redo/undo 결과를 대조한다.

### `06-database-replication.md` — 한 로그를 여러 노드의 상태로 만드는 법

- 복제의 단위를 page/WAL 같은 물리 기록과 row/change 같은 논리 기록으로 나누고, 호환성·대역폭·재생 비용을 비교한다.
- 새 replica가 snapshot/bootstrap으로 기준 상태를 만든 뒤 log position부터 따라잡는 과정을 추적한다.
- primary/replica 구조에서 local append, replica receive, durable write, apply와 client acknowledgment 시점을 분리한다.
- 동기·비동기 commit acknowledgment가 commit latency, 데이터 손실 한계와 가용성을 어떻게 바꾸는지 설명한다.
- replication lag가 stale read를 만드는 과정을 log position과 apply position으로 설명하고 read-your-writes·monotonic read가 필요한 조건을 구분한다.
- failover 때 새 primary 선택뿐 아니라 이전 primary를 fencing하지 못하면 split-brain과 중복 쓰기가 발생하는 이유를 다룬다.
- 진단에서는 commit latency, send/write/flush/apply lag, replica freshness와 topology·fencing 상태를 같은 timeline에 놓는다.
- Raft·Paxos의 상태 머신과 안전성 증명은 챕터 10에 위임하고, 여기서는 DB commit·failover 계약에 적용한다.

### `07-sharding-and-data-placement.md` — 데이터를 나누면 질의와 장애도 나뉜다

- 한 노드 안의 table partitioning과 여러 노드·실패 영역에 데이터를 배치하는 sharding을 구분한다.
- range·hash·directory routing이 범위 질의, 균등 분산, 배치 변경 비용에서 무엇을 얻고 포기하는지 비교한다.
- shard key를 cardinality만으로 고르지 않고 데이터 크기, QPS, 시간 편향, co-location과 주요 질의의 fan-out으로 판단한다.
- 주문·주문 항목·고객 데이터를 함께 배치하는 경우와 scatter-gather·cross-shard join/aggregation이 필요한 경우를 비교한다.
- local secondary index와 global secondary index가 query fan-out, 쓰기 조정, 일관성과 복구 비용을 어떻게 바꾸는지 설명한다.
- 전역 고유성·sequence가 단일 shard의 로컬 제약과 달리 조정 또는 key-space 분할을 요구하는 이유를 다룬다.
- resharding을 copy/backfill, catch-up, routing 전환, 검증·정리 단계로 나누고 이중 상태에서 누락·중복을 막아야 할 불변식을 세운다.
- 진단에서는 shard별 bytes·row count·QPS·latency, query fan-out, cross-shard bytes와 migration lag를 함께 본다.

### `08-distributed-transactions.md` — 여러 shard가 하나의 결정을 내리기까지

- local transaction, distributed concurrency control과 atomic commit을 서로 다른 책임으로 구분한다.
- 2PC의 coordinator와 participant가 `active → prepared → committed/aborted`로 전이하는 과정과 prepare/vote/decision log의 내구성 조건을 추적한다.
- participant가 `prepared`를 기록한 뒤 독자적으로 abort할 수 없는 이유와 coordinator 장애가 in-doubt·blocking을 만드는 과정을 설명한다.
- timeout은 상대의 실패나 전역 결정을 확정하지 못하므로 durable decision을 조회·복구해야 한다는 경계를 세운다.
- 중복 prepare·commit·abort 메시지를 안전하게 처리하려면 transaction identifier와 멱등 상태 전이가 필요함을 설명한다.
- cross-shard snapshot·version/timestamp 조정, distributed lock wait와 deadlock detection이 atomic commit과 별개의 격리 문제임을 다룬다.
- consensus-backed coordinator·transaction record처럼 결정 가용성을 높이는 설계가 2PC participant의 local durability와 어떻게 결합되는지 비교한다.
- 진단에서는 coordinator state, participant prepared state, decision log, 보유 lock/version과 단계별 latency를 하나의 transaction timeline에 대응시킨다.
- Saga·outbox·보상 트랜잭션은 다루지 않고 DBMS가 제공하는 원자성·격리·복구 경계에 집중한다.

### `09-database-paradigms-and-selection.md` — 데이터와 질의에 맞는 DBMS 고르기

이 문서는 앞선 세션에서 학습한 저장·인덱스·질의·트랜잭션·분산 비용을 DBMS 선택 문제에 적용하는 종합 개요다. “어떤 제품이 더 좋은가”가 아니라 다음 비교 축에서 워크로드와 보장에 맞는 패러다임을 판단한다.

- 주된 데이터 관계: 독립 key, aggregate document, table relation, sparse column, graph edge, 검색 대상 text
- 주된 접근 패턴: exact lookup, range/partition scan, ad-hoc query·join, multi-hop traversal, relevance search
- 필요한 계약: schema·constraint, transaction 범위, 읽기 일관성, 내구성과 장애 허용
- 규모와 비용: working set, write/read 비율, 수평 확장, fan-out, 메모리·저장 공간과 운영 복잡성

| 패러다임 | 대표 제품·구조 | 적합한 조건 | 실무 사례 | 주요 비용·경계 |
|---|---|---|---|---|
| Key-Value | Redis, Memcached; key 기반 lookup, 메모리 중심 저장·TTL·eviction | key를 정확히 알고 단순 get/set이 지배하며 매우 낮은 지연이 필요한 경우 | 캐시, session, rate limit, 임시 계산 결과 | RAM 비용, eviction, 제한된 ad-hoc query·관계 탐색, 제품별 persistence·durability 차이 |
| Wide Column | Cassandra, HBase; partition key와 정렬된 row/column family, 분산 LSM 계열 | 대량 쓰기와 수평 확장, partition/range 중심 질의가 지배하는 경우 | 시계열·IoT event, 로그·activity feed, 대규모 sparse data | query-first schema, hot partition, cross-partition 질의 비용, 제품별 일관성 보장 차이 |
| Document | MongoDB, Firestore; JSON 유사 document와 유연한 schema(`schema-less`로 불림) | 한 aggregate를 함께 읽고 쓰며 필드 구조가 빠르게 진화하는 범용 서비스 | 사용자 profile, 상품 catalog, CMS, mobile/web backend | schema가 없는 것이 아니라 application/validation으로 이동함, cross-document join·constraint·transaction 비용 |
| Relational | MySQL, PostgreSQL; table·SQL·constraint·index·ACID transaction | 관계와 무결성이 중요하고 ad-hoc query·join 및 다중 행 transaction이 필요한 경우 | 주문·재고·결제·원장, SaaS core data, 업무 시스템 | 정규화된 join 비용, write contention, 수평 분할과 cross-shard transaction의 운영 복잡성 |
| Graph | Neo4j; node·edge·property와 adjacency traversal | 관계 자체가 핵심 데이터이고 깊이·경로가 가변적인 multi-hop 질의가 지배하는 경우 | 추천, 사기 탐지, social/knowledge graph, dependency·권한 분석 | supernode·고차수 traversal, 전체 집계·단순 key lookup 비용, graph partitioning의 어려움 |
| Search Engine | Elasticsearch, Lucene; analyzer·token과 역색인(inverted index), relevance ranking | full-text·부분 일치·관련성 순위·facet이 필요한 경우 | 상품·문서 검색, 로그 탐색, observability와 support knowledge 검색 | refresh 지연과 eventual visibility, index 크기·재색인 비용, transaction·constraint를 요구하는 source of truth에는 부적합 |

- Redis·Memcached가 흔히 목표로 하는 sub-millisecond~millisecond 지연은 key-value 모델 자체의 보장이 아니라 RAM 배치, 네트워크, persistence 설정과 workload의 결과임을 명시한다.
- document DB의 “schema-less”는 데이터에 schema가 없다는 뜻이 아니라 schema 검증과 migration 책임이 DB 밖으로 이동하거나 완화된다는 뜻으로 교정한다.
- Cassandra와 HBase, MongoDB와 Firestore처럼 같은 계열의 제품도 transaction·일관성·query 기능이 다르므로 표의 공통 특성을 제품별 보장으로 단정하지 않는다.
- 한 시스템이 여러 저장소를 함께 쓰는 polyglot persistence는 독립된 데이터 소유권과 동기화 비용까지 정당화될 때만 선택하며 기본값으로 권하지 않는다.
- 확인 문제에서는 주어진 workload에 대해 후보 두 개를 고르고, 적합성뿐 아니라 포기하는 query·consistency·운영 능력을 비교하게 한다.

## 6. 통합 실습

`exercises/ch-11/`의 산출물은 **미니 스토리지 실험과 실행 계획·동시성 진단 리포트**다. 하나의 미니 DBMS를 완성하지 않고, 같은 주문 워크로드를 독립적인 다섯 실험에 통과시킨다.

### 공통 규율

- 기준 런타임은 저장소와 동일한 Node.js 24다. 내부 상태가 보이는 deterministic simulator를 직접 작성한다.
- 실제 실행 계획의 최소 관찰 경로는 SQLite CLI로 둔다. SQLite를 보편 구현으로 취급하지 않고 관찰하지 못한 지표는 `N/A`로 기록한다.
- 다른 DBMS 한 종류를 같은 관찰 표로 비교하는 것은 심화 트랙이다.
- 각 실험은 `사전 예측 → 단일 변수 변경 → 내부 카운터·계획 관찰 → 반증 여부 → 선택의 대가` 순서를 따른다.
- 절대 성능 수치보다 page read, hit ratio, amplification, cardinality error, wait, redo/undo 같은 원인에 가까운 계측값을 우선한다.
- 06~09 세션은 필수 구현 실험에 추가하지 않는다. 06~08은 각 본문의 상태 추적 예제와 확인 문제로 log position, shard routing과 2PC 상태를 판독하고, 09는 workload 선택표로 DBMS 패러다임의 적합성과 포기 비용을 비교한다.

### Part A — 페이지와 버퍼

- 고정 크기 page, record, buffer frame과 pin/dirty/evict/flush 상태를 구현한다.
- 행 폭, 순차/무작위 접근, 버퍼 크기와 교체 정책을 바꾸어 page read, hit ratio, dirty eviction을 기록한다.
- 같은 hit ratio라도 dirty flush와 접근 지역성에 따라 비용이 달라지는 사례를 포함한다.

### Part B — 인덱스와 스토리지 구조

- 교육용 B+트리형 정렬 page와 단순 LSM run을 구현하거나 계측 가능한 simulator로 제공한다.
- point/range/write 비율, key 순서와 buffer 크기를 바꾸어 read/write/space amplification, split, compaction을 비교한다.
- 복합 인덱스의 열 순서와 covering 여부가 읽는 범위와 추가 lookup을 바꾸는 실험을 포함한다.

### Part C — 실행 계획

- 주문 데이터 생성기는 데이터 양, 상태 분포, 고객군과 상태의 상관관계를 매개변수로 받는다.
- full scan/index access와 join 전략의 예상 비용을 측정 전에 작성한다.
- SQLite의 계획과 실행 통계를 가능한 범위에서 수집하고 공통 표로 변환한다. 지원하지 않는 estimate/actual/buffer 지표는 만들어 내지 않는다.
- 통계 수집 전후, 균등/편향 분포, 인덱스 유무를 비교해 계획 변화와 남는 오판을 설명한다.

### Part D — 동시성

- 두 트랜잭션의 연산 순서를 barrier로 통제하고 최소 두 가지 격리 계약에서 결과를 비교한다.
- 이상 현상, lock wait, deadlock과 MVCC visibility를 하나의 timeline에 대응시킨다.
- 최종 값만 보고 격리를 판정하지 않고 각 read가 본 version과 대기 이유를 기록한다.

### Part E — 복구

- WAL append, data-page flush, commit record와 fsync 사이에 crash point를 주입한다.
- 각 crash point에서 durable해야 할 transaction과 redo/undo 대상을 먼저 예측한다.
- 복구 후 committed transaction 보존, uncommitted effect 제거와 구조 불변식을 자동 테스트로 검증한다.

### 리포트 요구사항

1. 워크로드, 데이터 분포, 페이지·버퍼 크기, 실행 환경과 관찰 가능한 지표를 명시한다.
2. 각 실험의 측정 전 예측, 반증 조건, 결과와 내부 계측 근거를 대응시킨다.
3. 느린 쿼리에 대해 cardinality error, access path, join, spill, buffer, wait 중 경쟁 가설을 세우고 가장 싼 증거부터 판별한다.
4. 인덱스 변경의 읽기 이득과 쓰기·공간·유지보수 비용을 함께 제시한다.
5. 논리 결과의 정확성, 격리 보장과 내구성 보장을 서로 다른 계약으로 결론에 기록한다.

## 7. ROADMAP 반영안

전체 커리큘럼 구조의 챕터 이름과 산출물은 다음으로 변경한다.

| 챕터 | 주제 | 핵심 산출물 |
|---|---|---|
| 11 | 데이터베이스 시스템 내부와 실행 진단 | 미니 스토리지 실험과 실행 계획·동시성 진단 리포트 |

챕터 상세에는 §3의 학습 목표 5개와 문서 10편을 반영하고, 실습 과제는 다음 문장으로 요약한다.

> 페이지·버퍼, 인덱스·스토리지 구조, 실행 계획, 동시성, WAL 복구를 계층별 미니 실험으로 재현한다. 같은 합성 워크로드에서 예측과 내부 계측을 대조하고, 실행 계획과 대기 증거를 근거로 성능·격리·내구성 문제를 진단하는 리포트를 작성한다.

ROADMAP 변경과 동시에 PROGRESS의 챕터 11 이름과 문서 수를 `10`으로 맞춘다. 기존 `01-storage-engines.md`, `02-transactions-and-concurrency.md` 경로를 가리키는 참조는 새 문서 경계에 맞게 정리한다. 이 기획 문서를 확장하는 현재 작업에서는 ROADMAP과 PROGRESS를 수정하지 않는다.

## 8. 집필·검증 기준

- 독자는 5년차 이상 개발자로 가정하고 SQL 기초나 CRUD 사용법을 반복하지 않는다.
- 모든 주요 주제에 동작 모델, 설계 배경, 경계 조건과 직접 확인할 관찰 절차를 포함한다.
- B+트리/LSM, join 전략, 격리 수준처럼 선택지가 있는 주제는 워크로드와 불변식이 주어지기 전에는 단일 정답으로 서술하지 않는다.
- DBMS 표준의 보장, 이론 모델과 특정 엔진 구현을 명시적으로 구분한다.
- 실행 계획 예시는 데이터 정의·분포·통계 상태·엔진 버전과 실행 옵션을 함께 기록한다.
- 특정 엔진에서 얻은 수치와 동작을 다른 엔진에 일반화하지 않는다. 제품 차이는 공식 문서와 공식 소스로 검증한다.
- 재현 예제는 측정 전에 예상 결과와 반증 조건을 쓰고, 실제 실행 결과를 확인한 뒤 문서에 반영한다.
- 확인 문제는 용어 회상이 아니라 주어진 계획·카운터·transaction timeline·crash point에서 원인과 다음 조치를 판단하게 한다.
- 참고 자료는 DBMS 공식 문서·공식 소스와 원 논문을 우선한다. 기준 후보는 System R의 access path selection, ARIES 복구, LSM-tree 원 논문, *Architecture of a Database System*과 Spanner의 분산 transaction 설계이며 집필 시 원문과 현재 공식 문서를 다시 확인한다.
- `pnpm docs:build`, 내비게이션 문서 수·순서, 내부 링크, 실습 자동 테스트와 `git diff --check`를 완료 조건으로 삼는다.

## 9. 기획 자체 점검

- placeholder나 미정 파일명 없이 인트로 1편과 본문 9편의 책임을 확정했다.
- 제품 중립 원칙과 SQLite 최소 관찰 경로는 모순되지 않는다. SQLite는 실행 도구일 뿐 기준 구현이 아니며 미제공 지표를 `N/A`로 남긴다.
- 데이터 모델링은 물리 비용의 입력으로만 연결하고 정규화 과정으로 범위를 넓히지 않았다.
- 로컬 DBMS가 없어도 Node simulator로 내부 실험을 수행할 수 있고, 실행 계획 관찰에는 SQLite CLI를 별도 전제로 명시했다.
- 챕터 10은 부분 실패·합의·일관성 이론을 담당하고, 챕터 11은 이를 DB 로그 복제·shard 배치·atomic commit에 적용한다.
- 복제·샤딩·분산 트랜잭션과 DBMS 패러다임 비교는 본문 상태 추적·선택표와 확인 문제로 검증하며 필수 통합 실습 Part A~E에는 추가하지 않았다.
- Saga·outbox·보상 트랜잭션을 제외해 DBMS 내부 transaction 경계를 서비스 워크플로로 넓히지 않았다.
- DBMS 패러다임은 대표 제품의 공통 경향만 비교하고 제품별 기능·일관성·지연을 일반화하지 않았다.
- 문서별 주제, 로컬 통합 실습의 관찰 항목과 ROADMAP 산출물이 같은 학습 목표를 가리킨다.
