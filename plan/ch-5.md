# 챕터 5 기획 — 컴파일러와 인터프리터

[ROADMAP.md](../ROADMAP.md)의 챕터 5(`docs/ch-5/`, 인트로 1편과 본문 4편)을 집필하기 위한 상세 기획이다.
범위·경로가 ROADMAP과 어긋나면 ROADMAP을 우선한다.

## 1. 챕터의 관점

독자는 5년차 이상 개발자다. 컴파일 오류와 린터 경고를 고치고, 번들러가 만든 코드를 배포하며, JIT 워밍업이나 역최적화가 성능에 영향을 준다는 말도 들어 봤다. 그러나 소스 코드가 실행 가능한 형태로 바뀌는 동안 어떤 중간 산출물이 만들어지는지, 성능 문제나 오진이 어느 단계의 책임인지 구분하지 못할 수 있다.
이 챕터가 그 위에 더하는 것은 세 가지다.

1. **변환 파이프라인의 책임 추적** — 소스가 토큰·AST·IR·바이트코드 또는 기계어로 바뀌는 동안 각 단계가 어떤 정보를 보존하고 버리는지 설명하고, 오류가 생겼을 때 관찰할 산출물을 고른다.
2. **실행 전략의 비용 위치 판단** — 트리 워커, 바이트코드 VM, JIT가 준비 시간·실행 시간·메모리 중 어디에 비용을 지불하는지 비교하고, 워크로드 수명과 안정성에 맞는 전략을 판단한다.
3. **분석 결과를 근사로 해석하기** — 정적 분석이 실제 실행을 대신해 프로그램 상태를 근사하는 구조를 이해하고, 린터·타입 체커 경고를 건전성·완전성 및 false positive·false negative의 교환 위에서 해석한다.

각 본문은 하나의 독립적인 핵심 질문에 답하도록 구성한다. 공통 Toy 표현식 언어는 설명을 잇는 사례로 재사용하되, 앞 문서의 구현 세부를 읽어야만 다음 문서를 이해할 수 있는 선형 의존은 만들지 않는다. 모든 문서는 콘텐츠 집필 지침의 세 층위(동작 모델 → 설계 배경 → 경계 조건)를 따른다.

## 2. 범위 결정

### 다루는 것

- 토큰화와 구문 분석의 책임, 재귀 하강과 LR 계열 파서의 선택 기준, concrete syntax와 AST의 차이, 소스 위치와 오류 복구의 목적
- AST보다 낮은 수준의 중간 표현, 기본 블록과 제어 흐름 그래프(CFG), SSA의 불변식과 phi 함수, 대표 최적화 패스와 코드 생성의 책임
- 트리 워커, 스택 기반 바이트코드 VM, JIT 계층화(tiering), 프로파일 기반 추측 최적화, inline cache와 역최적화
- CFG 기반 데이터 흐름 분석, lattice·전달 함수·고정점의 직관, 도달 정의·활성 변수·상수 전파, 경로·흐름 민감도의 비용
- 컴파일러 진단, 린터, 타입 체커가 공유하는 분석 구조와 건전성·완전성, false positive·false negative의 트레이드오프
- Toy 표현식 언어의 트리 워커와 바이트코드 컴파일러·VM 구현, 두 실행 전략의 의미 동등성 검증과 성능 교차점 측정

### 다루지 않기로 결정한 것

- **파서 생성기 구현과 LR 테이블 구성 절차는 다루지 않는다.** 재귀 하강과 LR의 입력 문법·오류 메시지·도구화 트레이드오프를 비교하되 shift/reduce 테이블을 손으로 만드는 법은 제외한다.
- **산업용 백엔드나 기계어 인코더를 구현하지 않는다.** instruction selection, register allocation, instruction scheduling은 코드 생성의 책임으로 설명하되 LLVM 백엔드나 특정 ISA의 명령 인코딩은 범위 밖이다.
- **Toy VM에 JIT를 구현하지 않는다.** JIT의 원리는 V8 관찰 예제로 검증하고 실습은 트리 워커와 바이트코드 VM의 비교에 집중한다.
- **V8 내부 구성 요소를 카탈로그처럼 나열하지 않는다.** tier 이름과 진단 플래그는 버전 의존 사례로만 사용하고, 계층화·추측·역최적화라는 일반 모델을 중심에 둔다.
- **추상 해석의 수학적 전개는 별도 절로 두지 않는다.** lattice와 고정점은 데이터 흐름 분석을 이해하는 데 필요한 직관과 작은 예제까지만 다룬다.
- **소스 맵·디버거·증분 컴파일·매크로 시스템은 상세히 다루지 않는다.** 소스 위치 보존과 도구 관찰 지점에서 필요성을 언급하는 수준으로 제한한다.

## 3. 문서별 상세 기획

본문 4편은 콘텐츠 집필 지침의 기본 구조(학습 목표 → 배경 → 핵심 개념 → 실무 관점 → 더 깊이 → 정리 → 확인 문제 → 참고 자료)를 따르고, 45~90분 분량으로 작성한다. `00-introduction.md`는 같은 학습 흐름을 유지하되 10~15분 분량의 오리엔테이션으로 구성한다.

### `00-introduction.md` — 소스 코드는 어떻게 실행 가능한 형태가 되는가

- **핵심 질문**: 한 줄의 소스 코드가 실행 결과나 도구의 경고로 돌아오기까지 어떤 단계와 산출물을 거치는가?
- **도입 사례**: 같은 JavaScript 함수가 첫 호출과 반복 호출에서 다른 성능을 보이고, 사소한 값 형태 변화 뒤 다시 느려지는 상황으로 시작한다. “언어가 느리다”가 아니라 어느 단계가 무엇을 관찰하고 결정했는지 묻도록 유도한다.
- 컴파일과 인터프리트를 이분법으로 나누지 않고, 현대 런타임이 파싱·바이트코드 컴파일·해석·JIT를 조합한다는 전체 지도를 제시한다.
- 번역(01) → 표현과 최적화(02) → 실행 전략(03) → 실행하지 않고 추론하기(04)의 네 질문을 소개하고 내부 동작은 각 본문에 위임한다.
- AST, IR, 바이트코드 같은 용어를 미리 자세히 가르치지 않는다. 각 산출물이 “다음 단계가 필요로 하는 정보를 담은 인터페이스”라는 관점만 세운다.

### `01-lexing-and-parsing.md` — 문자열에서 프로그램 구조를 복원하기

- **핵심 질문**: 컴파일러는 문자 나열을 어떻게 의미 분석과 변환에 적합한 구조로 바꾸며, 좋은 AST는 무엇을 보존해야 하는가?
- **핵심 개념 뼈대**
  - 렉싱의 책임: 문자 스트림을 토큰 종류·값·소스 범위로 묶기, 공백·주석 처리, maximal munch와 모호성
  - 파싱의 책임: 토큰열이 문법에 맞는지 확인하고 계층 구조를 복원하기, precedence와 associativity가 AST 모양을 바꾸는 이유
  - concrete syntax tree와 AST: 괄호·구분자 같은 표면 구문을 제거하면서도 의미 분석·진단·변환에 필요한 정보를 보존하는 기준
  - 재귀 하강: 문법과 코드의 대응, 직접 제어와 좋은 오류 메시지, 좌재귀 제거 필요성
  - LR 계열: 더 넓은 문법을 체계적으로 처리하고 생성기로 자동화하는 이점, 문법 충돌과 오류 메시지의 비용
  - 오류 처리: fail-fast와 panic-mode 복구의 목적, 파일 전체 진단을 위해 동기화 지점을 고르는 법, line/column만 저장하기보다 절대 offset·source span을 보존해야 하는 이유
- **공통 관찰 예제**: `let result = add(1, 2 * 3);`를 토큰열과 AST로 바꾸고, 연산자 우선순위와 호출 인자 구조가 어떻게 표현되는지 단계별로 보인다. 잘못된 닫는 괄호에서는 진단 범위와 복구 지점을 관찰한다.
- **실무 연결**: 포매터가 concrete syntax와 주석을 더 많이 보존해야 하는 이유, IDE가 부분적으로 잘못된 코드에서도 복구 가능한 구문 트리를 필요로 하는 이유, codemod가 문자열 치환보다 AST 변환을 택하는 이유를 비교한다.
- **경계 조건**: 문법이 모호할 때 파서 구현만으로 의미를 정할 수 없는 경우, 컨텍스트 의존적 토큰화(JavaScript의 `/` 등), 오류 복구가 후속 가짜 오류를 늘리는 경우

### `02-ir-and-optimization.md` — 프로그램을 분석하고 바꾸기 좋은 형태

- **핵심 질문**: AST만으로 최적화와 코드 생성을 수행하기 어려운 이유는 무엇이며, IR과 SSA는 어떤 복잡성을 줄이는가?
- **핵심 개념 뼈대**
  - IR의 목적: 소스 문법과 대상 기계 사이를 분리하고 여러 분석·변환 패스가 공유하는 계약 제공
  - IR 수준: AST에 가까운 high-level IR, 제어 흐름과 연산이 명시된 mid-level IR, 기계 명령에 가까운 low-level IR의 정보·이식성 교환
  - 기본 블록과 CFG: 분기·합류·루프를 그래프로 만들고 지배 관계와 도달 가능성을 분석하는 좌표계
  - SSA: 각 값이 한 번만 정의된다는 불변식, 분기 합류의 phi 함수, use-def chain이 분석과 최적화를 단순하게 만드는 이유
  - 대표 패스: constant folding·propagation, dead code elimination, common subexpression elimination, inlining을 “필요한 분석 → 변환 → 지켜야 할 의미” 순서로 설명
  - 패스 순서: 한 최적화가 다른 최적화의 기회를 만들거나 없애는 상호작용, 고정된 단일 최적 순서가 없는 이유
  - 코드 생성의 책임: instruction selection, register allocation, scheduling을 개념적으로 분리하고 대상별 비용 모델이 개입하는 지점
  - 정확성 계약: 관찰 가능한 동작을 보존해야 한다는 기준과 overflow·부동소수점·aliasing·부작용이 최적화를 막는 사례
- **공통 관찰 예제**: 조건식과 반복 계산이 포함된 작은 함수를 AST → 기본 블록 → SSA 형태로 낮춘 뒤, 상수 전파와 죽은 코드 제거 전후를 비교한다. 실제 LLVM 문법을 그대로 가르치기보다 독립적인 축약 IR을 사용한다.
- **실무 연결**: 소스 코드상 사라질 것 같은 분기가 최종 코드에 남는 이유, 인라이닝이 호출 비용을 줄이면서 코드 크기와 instruction cache 압력을 키우는 이유, 최적화 빌드의 스택 트레이스가 소스와 어긋나는 이유
- **경계 조건**: 부작용과 aliasing 때문에 표현식 제거가 안전하지 않은 경우, IEEE 754 규칙이 대수적 재배치를 제한하는 경우, aggressive inlining이 성능을 악화시키는 경우, undefined behavior를 전제로 한 최적화가 개발자의 직관과 충돌하는 경우

### `03-interpreters-and-jit.md` — 실행 비용을 언제 지불할 것인가

- **핵심 질문**: 트리 워커, 바이트코드 VM, JIT는 같은 프로그램을 실행하면서 준비 시간·실행 시간·메모리 비용을 어떻게 다르게 지불하는가?
- **핵심 개념 뼈대**
  - 트리 워커: AST 노드를 직접 순회하는 단순성, 노드 디스패치와 포인터 추적 비용, 빠른 시작과 구현 용이성
  - 바이트코드 VM: compact instruction stream, opcode dispatch, operand stack과 call frame, 파싱·의미 확인 비용을 반복 실행에서 재사용하는 구조
  - stack VM과 register VM: 명령 수·디코딩·코드 크기·컴파일 복잡성의 교환. 실습은 stack VM으로 고정한다
  - ahead-of-time과 JIT: 실행 전 최적화와 실행 중 프로파일 활용의 차이, 시작 시간·peak 성능·배포 복잡성의 비용 위치
  - tiering: 빠르게 생성되는 저계층 코드와 비싸게 최적화한 상위 계층을 조합하는 이유, hotness 판단과 OSR(on-stack replacement)의 목적
  - 추측 최적화: 관찰된 타입·객체 형태·호출 대상을 가드로 고정해 일반성을 속도로 바꾸는 구조
  - inline cache: 반복되는 프로퍼티 접근·호출의 탐색을 캐시하는 원리, monomorphic→polymorphic→megamorphic 전이가 최적화 가능성에 미치는 영향
  - 역최적화: 가드가 깨질 때 안전한 일반 코드로 돌아가기 위해 최적화 코드가 소스/바이트코드 상태 복원 정보를 보존해야 하는 이유
- **관찰 예제**: Toy 언어의 동일 프로그램을 트리 워커와 바이트코드 VM으로 실행해 실행 경로를 비교한다. 별도 최소 JavaScript 함수는 안정된 숫자 입력으로 워밍업한 뒤 다른 형태의 값을 전달해 최적화·역최적화 로그를 관찰한다. V8 플래그와 tier 이름은 집필 시 Node.js 24에 포함된 V8 버전에서 재확인한다.
- **실무 연결**: 서버리스·CLI처럼 짧은 프로세스와 장수명 서버에서 최적 전략이 달라지는 이유, 마이크로벤치마크가 워밍업 상태를 통제하지 않으면 거짓 결론을 내는 이유, 실제 요청 데이터의 형태 불안정성이 p99 지연에 영향을 주는 구조
- **경계 조건**: 실행 시간이 짧아 JIT 비용을 회수하지 못하는 경우, 프로파일이 대표성을 잃는 phase change, 드문 입력이 역최적화를 반복시키는 경우, `eval`·동적 속성 변경처럼 가정을 어렵게 하는 언어 기능

### `04-static-analysis.md` — 실행하지 않고 프로그램을 추론하기

- **핵심 질문**: 정적 분석기는 모든 실행 경로를 실제로 돌지 않고 어떻게 사실을 추론하며, 경고를 어디까지 믿을 수 있는가?
- **핵심 개념 뼈대**
  - 분석의 입력: AST만으로 가능한 구문 검사와 CFG·IR이 필요한 흐름 분석의 차이
  - 추상 상태: 구체적인 모든 값 대신 “상수/비상수”, “초기화됨/안 됨”, 가능한 타입 집합처럼 관심 속성만 남기는 근사
  - lattice와 join의 직관: 분기별 정보를 합칠 때 더 안전하지만 덜 정밀한 상태로 올라가는 이유
  - 전달 함수와 고정점: 기본 블록을 지날 때 상태를 갱신하고, 루프가 있는 CFG를 변화가 멈출 때까지 반복하는 worklist 알고리즘
  - 대표 분석: reaching definitions, live variables, constant propagation, definite assignment를 같은 프레임으로 비교
  - 민감도 선택: flow-sensitive, path-sensitive, context-sensitive 분석이 정밀도를 얻는 대신 시간·메모리와 구현 복잡성을 지불하는 구조
  - 건전성과 완전성: 놓치지 않기 위해 오진을 허용하거나, 소음을 줄이기 위해 일부 문제를 놓치는 도구별 선택. ch-4 타입 시스템과 ch-2 결정 불가능성으로 연결
  - 진단 정책: severity, suppression, confidence가 분석 결과와 제품 정책을 분리하는 계층이라는 점
- **공통 관찰 예제**: 조건 분기와 루프가 있는 작은 함수의 definite assignment와 constant propagation을 worklist로 추적한다. 항상 참인 것처럼 보이지만 경로 정보가 없으면 확정할 수 없는 사례로 join에서 정밀도가 사라지는 지점을 보인다.
- **실무 연결**: ESLint의 구문·스코프 규칙, TypeScript의 흐름 기반 타입 좁히기, 컴파일러의 미초기화 변수 경고를 분석 정밀도와 정책의 조합으로 해석한다. 도구별 실제 동작은 공식 문서와 재현 예제로 검증한다.
- **경계 조건**: 동적 호출·reflection·외부 I/O가 분석을 불완전하게 만드는 경우, 경로 폭발로 path-sensitive 분석이 제한되는 경우, suppression이 실제 결함을 숨기는 경우, 분석기가 조용해도 안전을 보장하지 않는 경우

## 4. 문서 간 관계

```text
00 전체 지도 ──▶ 01 문자열→AST ──▶ 02 AST→IR·최적화 ──▶ 03 실행 전략
                   │                   │                    │
                   └───────────────────┴──────▶ 04 정적 분석
```

- 각 본문은 독립적인 핵심 질문과 최소 예제를 가져 선택적으로 읽을 수 있게 한다. 다만 00은 네 단계의 책임과 용어 지도를 제공하므로 권장 진입점이다.
- 01의 AST는 02의 lowering 입력이자 04의 구문 분석 입력이다. 두 문서에서 필요한 AST 조각을 다시 제시하되 렉싱·파싱 과정을 반복 설명하지 않는다.
- 02의 기본 블록·CFG·SSA는 03의 JIT 최적화와 04의 데이터 흐름 분석이 공유하는 좌표계다. 03과 04에서는 필요한 불변식만 요약하고 상세는 02로 연결한다.
- 03의 바이트코드 VM은 ch-6의 런타임·메모리 주제와 맞닿지만, 이 챕터에서는 명령 디스패치와 실행 프레임까지만 다룬다. 객체 할당과 GC를 VM 실습에 추가하지 않는다.
- 공통 Toy 언어는 사례의 어휘와 의미를 맞추는 장치다. 문서가 이전 산출물 파일이나 구현 완료를 전제로 하지 않도록 각 예제를 자체 완결적으로 제시한다.

## 5. 실습 과제 기획 (`exercises/ch-5/`)

ROADMAP 산출물: “챕터 4의 표현식 언어를 바이트코드로 컴파일하는 컴파일러와 VM을 구현하고, 트리 워커 인터프리터와 실행 성능을 비교 측정한다.”

### 환경과 언어 범위

- **환경**: TypeScript + Node.js 24, `exercises/ch-5/` 아래 pnpm 워크스페이스 패키지. 테스트는 `node:test`를 사용해 별도 테스트 프레임워크 의존성을 추가하지 않는다.
- **언어 기능**: 숫자·불리언 리터럴, 산술(`+ - * /`)과 비교(`== < <=`), `let` 지역 변수, 블록, `if/else` 조건식, 이름으로 호출하는 단순 함수와 `return`으로 제한한다.
- **명시적 제외**: 문자열·객체·배열, 클로저, 일급 함수, 가변 캡처, 예외, 모듈, 비동기 실행은 구현하지 않는다. 값 표현은 JavaScript `number`와 `boolean`을 사용하고 사용자 객체와 GC 문제를 배제한다.
- **의미 기준**: 챕터 4 실습이 먼저 존재하면 동일한 AST·평가 의미를 재사용한다. 아직 없거나 인터페이스가 다르면 위 최소 언어를 독립 패키지로 구현하되, 이후 ch-4와 연결할 수 있도록 lexer/parser/AST/evaluator 경계를 분리한다.

### 구현 단위와 인터페이스

1. **프런트엔드**
   - lexer는 `{ kind, lexeme, start, end }` 토큰을 만들고 JavaScript 문자열의 UTF-16 code-unit 절대 offset 기준 source span을 보존한다. line/column은 진단을 출력할 때 원문에서 계산한다.
   - parser는 discriminated union AST를 만들며 연산자 우선순위와 결합 법칙을 명시한다.
   - 오류는 `LexError`, `ParseError`로 구분하고 source span과 사람이 읽을 메시지를 포함한다. 실습에서는 첫 오류에서 중단하며 복구 파싱은 구현하지 않는다.
2. **트리 워커 기준 실행기**
   - lexical environment 체인을 사용해 변수와 함수 호출을 평가한다.
   - 언어 의미의 기준 구현으로 삼아 컴파일러·VM 결과를 비교한다.
3. **바이트코드 컴파일러**
   - AST를 상수 풀과 opcode/operand 명령열로 변환한다.
   - 최소 opcode는 `CONST`, `LOAD_LOCAL`, `STORE_LOCAL`, 산술·비교 연산, `JUMP`, `JUMP_IF_FALSE`, `CALL`, `RETURN`, `POP`으로 고정한다.
   - 분기 offset patching, 함수별 local slot 할당, stack effect 계산을 컴파일 단계에서 수행한다. 잘못된 opcode 피연산자나 스택 불균형은 `CompileError`로 보고한다.
4. **스택 기반 VM**
   - operand stack, instruction pointer, 함수별 call frame(base slot·return address)을 명시적으로 관리한다.
   - 잘못된 연산 타입, 0으로 나누기 정책, 정의되지 않은 함수, 스택 underflow를 `RuntimeError`로 구분한다. 나눗셈 0은 JavaScript의 `Infinity`가 아니라 오류로 고정해 Toy 언어 의미를 단순하게 유지한다.
5. **디스어셈블러와 계측**
   - 상수·offset·opcode를 사람이 읽을 수 있게 출력해 테스트 실패와 리포트에서 실행 경로를 설명할 수 있게 한다.
   - 실행기별 파싱·컴파일 시간, 실행 시간, 실행 명령/방문 노드 수를 기록한다.

### 검증 시나리오

- lexer: 공백·주석·다자리 숫자·키워드·잘못된 문자와 source span
- parser: 우선순위, 중첩 조건식, 함수 호출, 누락된 괄호·세미콜론의 오류 위치
- compiler: 상수 풀 중복 처리, local slot, forward jump patching, 함수 호출·반환, 예상 바이트코드 디스어셈블 결과
- VM: 산술·비교, 분기 양쪽, 중첩 호출, 재귀, 런타임 오류와 스택 정리
- 의미 동등성: 정상 프로그램 모음을 트리 워커와 VM으로 실행해 값이 같음을 확인하고, 0으로 나누기·정의되지 않은 이름처럼 두 실행기가 공유하는 런타임 오류는 같은 `RuntimeError` 종류가 나는지 확인한다. 바이트코드 검증에서만 생기는 `CompileError`는 컴파일러 단독 테스트로 분리한다.
- 경계: 깊은 재귀에서 call frame 한도를 명시적으로 검사하고 무한 재귀가 호스트 Node 프로세스의 스택을 소진하기 전에 Toy VM 오류로 중단되게 한다.

### 성능 비교와 완료 기준

- **워크로드**: 짧은 산술식, 분기 중심 함수, 재귀 피보나치(디스패치 비용 강조), 반복 호출이 많은 계산 함수를 사용한다. 동일 입력과 동일 의미를 두 실행기에 제공한다.
- **구간 분리**: (1) parse/compile을 포함한 end-to-end 시간과 (2) 준비된 AST/바이트코드를 반복 실행하는 steady-state 시간을 별도로 측정한다.
- **측정 규칙**: Node/V8·OS·CPU 버전, 워밍업 횟수, 측정 반복 수, 입력 크기를 기록한다. 여러 프로세스 반복의 중앙값과 분산을 제시하고, 한 프로세스 안의 단일 측정값만으로 결론 내리지 않는다.
- **분석 목표**: 한 번 실행할 때 트리 워커의 준비 비용 이점, 반복 실행에서 바이트코드 컴파일 비용을 회수하는 교차점, 프로그램 종류에 따른 opcode dispatch와 AST 방문 비용의 차이를 설명한다. “바이트코드가 항상 빠르다”를 전제로 하지 않는다.
- **V8 보조 관찰**: Toy VM 벤치와 분리된 최소 JavaScript 예제로 안정된 입력과 형태가 바뀌는 입력을 비교한다. Node.js 24에서 지원되는 공식 또는 진단 플래그로 최적화·역최적화를 관찰하되 그 수치를 Toy VM과 직접 비교하지 않는다.
- **완료 기준**: 모든 단위·동등성 테스트 통과, 한 명령으로 재현되는 벤치마크, 바이트코드 디스어셈블 예시, 실행 전략별 시간·연산 수 비교, “비용을 언제 지불하고 언제 회수했는가”를 설명하는 리포트를 포함한다.

집필 시 아래 자료를 우선 확인하고, V8·Node·도구 버전에 의존하는 명칭과 동작은 인용 시점에 재검증한다.

- **렉싱·파싱**: Aho et al. *Compilers: Principles, Techniques, and Tools* 및 Cooper & Torczon *Engineering a Compiler*를 개념 뼈대로 사용하고, ECMAScript Lexical Grammar·Syntactic Grammar와 실제 파서 공식 문서를 사례 검증에 사용한다.
- **IR·SSA**: Cytron et al. “Efficiently Computing Static Single Assignment Form and the Control Dependence Graph”, LLVM Language Reference와 공식 pass 문서, LLVM Kaleidoscope 튜토리얼의 IR·코드 생성 절
- **최적화 정확성**: LLVM의 undefined behavior·poison/freeze 규정, ECMAScript의 Number·연산 의미처럼 언어 및 IR 명세가 허용하는 변환 범위
- **인터프리터·VM**: CPython `ceval`과 bytecode 공식 문서, Lua VM 설계 자료를 보조 비교군으로 사용하되 본문은 특정 VM 구현 비교가 아니라 stack/register·dispatch 비용 모델에 집중한다.
- **V8·JIT**: V8 공식 블로그와 문서의 Ignition, inline cache, speculative optimization, deoptimization 설명 및 Node.js CLI 공식 문서. 현재 tier 구성과 플래그는 집필 시 설치된 Node.js 24의 V8 버전으로 확인한다.
- **정적 분석**: Kildall “A Unified Approach to Global Program Optimization”, Nielson et al. *Principles of Program Analysis*, TypeScript 공식 narrowing/control flow analysis 문서, ESLint 공식 rule·architecture 문서
- **벤치마크**: Node.js `perf_hooks` 공식 문서와 V8 벤치마크 주의사항. ch-1 `01-complexity-analysis.md`의 워밍업·반복·중앙값·환경 명시 규칙을 재사용한다.

### 통념 검증 목록 (본문에서 정면으로 다룰 것)

- “컴파일 언어는 인터프리터 언어보다 항상 빠르다” → 언어와 실행 전략은 일대일 대응이 아니며, 시작 시간·프로파일 정보·실행 수명에 따라 비용 위치가 달라진다.
- “JIT는 워밍업만 끝나면 계속 빨라진다” → 입력 형태와 호출 대상이 바뀌면 가드가 깨지고 역최적화나 재컴파일이 발생할 수 있다.
- “AST에 최적화 패스를 바로 붙이면 IR이 필요 없다” → 제어 흐름과 값 정의가 암묵적이면 분석·변환마다 소스 문법의 복잡성을 다시 처리해야 한다.
- “SSA는 소스 변수에 번호만 붙인 표기법이다” → 실제 목적은 정의를 하나로 만들고 use-def 관계와 합류를 명시해 분석·최적화를 단순화하는 데 있다.
- “최적화는 의미를 바꾸지 않으니 언제나 켜도 안전하다” → 언어의 관찰 가능 동작, 부동소수점, overflow, aliasing, undefined behavior 규칙이 허용 범위를 결정한다.
- “파서는 문법만 맞는지 확인하면 끝이다” → AST 소비자, 오류 복구, source span, 주석·표면 구문 보존 요구가 파서와 트리 설계를 바꾼다.
- “린터나 타입 체커가 조용하면 프로그램은 안전하다” → 분석은 선택한 추상화와 민감도 안에서만 결론을 내리며, 도구는 소음과 누락 사이의 정책을 가진다.
- “false positive는 분석기의 버그다” → 높은 건전성을 목표로 한 분석에서는 정보 부족을 위험으로 근사한 의도적 결과일 수 있다.

## 7. 작성 순서와 검증 계획

1. 독자 권장 순서는 `00 → 01 → 02 → 03 → 04`다. 다만 각 본문은 핵심 질문, 최소 AST/IR 예제, 용어 연결을 자체적으로 제공해 선택적으로 읽을 수 있게 한다.
2. 공통 Toy 프로그램의 구문과 의미를 먼저 고정한 뒤, 각 문서에서는 필요한 변환 단계만 보여 준다. 문서 사이에서 같은 예제의 AST나 IR 모양이 달라지지 않도록 집필 전 공통 fixture를 만든다.
3. 01의 parser 예제는 문법·우선순위·오류 span을 테스트로 검증하고, 02의 최적화 예제는 변환 전후 결과가 동일한지 실행 또는 기준 해석기로 확인한다.
4. 03의 모든 성능 수치는 Node.js 24에서 직접 반복 측정한 뒤 실행 환경과 분산을 함께 기재한다. V8 tier·플래그·로그 형식은 현재 버전의 공식 문서와 실제 출력 양쪽으로 확인한다.
5. 04의 도구 사례는 TypeScript·ESLint의 현재 공식 문서와 최소 재현 코드로 경고 발생 여부를 확인한다. 일반 데이터 흐름 모델과 특정 도구의 휴리스틱·정책을 구분해 서술한다.
6. 확인 문제는 용어 회상이 아니라 산출물 판독(AST/CFG/SSA), 안전하지 않은 최적화 식별, 워크로드별 실행 전략 선택, 정적 분석 경고의 누락·오진 원인 추론으로 구성한다.
7. `docs/ch-5/`을 처음 만들 때 `docs/.vitepress/navigation.ts`의 `PHASE_LABELS`에 챕터 5 레이블을 등록하고 빌드로 nav·sidebar 노출을 확인한다(저장소 운영 지침 적용).
8. 문서 수나 완료 상태가 실제로 바뀔 때만 `PROGRESS.md`를 갱신한다. 이번 `plan/ch-5.md` 작성만으로는 PROGRESS, VitePress 내비게이션, `exercises/ch-5/`를 변경하지 않는다.
9. 완료 전 ROADMAP의 5편 구성·파일명·학습 목표·실습 산출물과 대조하고, ch-2·ch-4·ch-6·ch-7로의 위임이 각 챕터 기획과 충돌하지 않는지 확인한다. 구현자가 추가로 선택해야 하는 도구·문법 결정이 없어야 한다.
