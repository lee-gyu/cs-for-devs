# 챕터 1 실습 — 자료구조 구현과 복잡도 예측·실측 대조

[ROADMAP](../../ROADMAP.md) 챕터 1의 산출물인 **"핵심 자료구조를 직접 구현하고, 점근 복잡도 예측과 실측(캐시 효과 포함)의 차이를 벤치마크 리포트로 정리한다"**를 수행하는 실습이다. [plan/ch-1.md](../../plan/ch-1.md) §5 기획을 기준으로 하며, 1.0 인트로의 문제의식과 챕터 1 본문 4편([1.1](../../docs/ch-1/01-complexity-analysis.md), [1.2](../../docs/ch-1/02-data-structures-in-memory.md), [1.3](../../docs/ch-1/03-algorithm-design-paradigms.md), [1.4](../../docs/ch-1/04-graph-theory-and-algorithms.md))의 동작 모델을 전제로 한다. 인트로가 작성되기 전에는 [ROADMAP의 0번 기획](../../ROADMAP.md)을 기준으로 삼는다.

이 실습의 핵심 규율은 하나다. **측정하기 전에 예측을 글로 남긴다.** 각 시나리오마다 점근 복잡도와 예상 배율(방향과 자릿수)을 먼저 표로 쓰고, 실측이 예측과 갈린 지점을 구현 내부의 계측 카운터와 메모리 접근 패턴으로 설명한다. 예측 없이 측정만 하면 어떤 숫자가 나와도 "그렇구나"로 끝나고, 갈린 지점이야말로 [1.1](../../docs/ch-1/01-complexity-analysis.md)이 말하는 점근 표기가 버린 상수 인자의 실체다.

## 학습 목표

- 동적 배열·연결 리스트·해시 맵·이진 힙을 직접 구현하고, 내부 동작(재할당, 충돌, 리사이징)을 관찰할 수 있는 계측 카운터를 함께 설계한다.
- 같은 점근 복잡도의 연산이 실측에서 갈리는 것을 자기 구현으로 재현하고, 원인을 접근 패턴(순차 vs 포인터 추적)과 계측 수치로 설명한다.
- amortized O(1)의 개별 연산 스파이크(재할당, 리사이징)를 연산 단위 시계열로 포착하고, 집계 통계(중앙값)가 숨기는 것을 확인한다.
- [1.1의 벤치마크 방법론](../../docs/ch-1/01-complexity-analysis.md)(워밍업, DCE 방지, 반복·중앙값, GC 통제, 환경 명시)을 갖춘 측정을 도구로 수행하고, 예측·실측 대조 리포트를 작성한다.

## 실행 환경

**TypeScript + Node.js 24**로 작성한다(저장소 도구 체인과 동일, [mise.toml](../../mise.toml) 기준). 실습 코드는 `exercises/ch-1/`을 pnpm 워크스페이스 패키지로 만들어 그 안에 둔다 — 저장소 루트의 `pnpm-workspace.yaml`에 `packages` 목록으로 `exercises/ch-1`을 등록하고, 이 디렉터리에 `private: true`인 `package.json`을 만든다. Node.js 24는 타입 스트리핑으로 `.ts` 파일을 직접 실행하므로 빌드 단계 없이 `node`로 테스트·벤치를 돌린다(enum·namespace 같은 비삭제(non-erasable) 문법은 쓰지 않는다).

### 도구 선정

plan §5에서 미결이던 러너를 다음과 같이 확정한다.

- **테스트 러너: `node:test`** (Node.js 내장). 자료구조 단위 테스트에 필요한 것은 assert와 테스트 구조화가 전부이고, 내장 러너는 의존성 없이 `.test.ts`를 직접 실행한다. 저장소가 유지하는 의존성을 늘리지 않는 선택이다.
- **벤치마크 러너: mitata** (devDependency로 추가). 워밍업, 반복·통계 처리(중앙값·백분위), 실행 환경(런타임·CPU) 출력을 갖춰 [1.1의 방법론](../../docs/ch-1/01-complexity-analysis.md) 다섯 항목과 어긋나지 않는다. 1.1이 전제한 대로 "러너가 함정을 어떻게 다루는지 이해하고 쓴다" — 워밍업이 JIT 티어 승격을 유도하는 과정과 DCE 방지를 위해 결과를 소비해야 한다는 점은 러너를 써도 사용자의 책임으로 남는다.
- **연산 단위 계측은 직접 작성한다.** mitata 같은 집계 러너는 반복 전체의 통계를 내므로, 개별 push가 리사이징에 걸려 튀는 순간은 통계에 뭉개진다. 스파이크 관찰(S3·S4)은 `performance.now()` 또는 `process.hrtime.bigint()`로 연산마다 시간을 기록하는 루프를 직접 쓴다. 집계 비교는 mitata, 시계열·스파이크는 자체 계측 — 이 분업 자체가 [1.1의 amortized와 꼬리 지연 논의](../../docs/ch-1/01-complexity-analysis.md)의 실습이다.

### 측정 규율

[1.1의 벤치마크 방법론](../../docs/ch-1/01-complexity-analysis.md)을 그대로 적용한다.

- **환경 명시**: 리포트에 CPU 모델, OS, Node.js 버전(`node --version`), 전원 상태(노트북이면 어댑터 연결 여부)를 기록한다. 결론은 절대값이 아니라 **배율과 곡선의 모양**으로 서술한다.
- **노이즈 통제**: 측정 중 같은 머신에서 다른 부하(빌드, 브라우저)를 돌리지 않는다. 반복 측정 후 중앙값을 취하고 min/max로 잡음 크기를 확인한다.
- **GC 통제**: 측정 루프가 측정 대상 외의 할당을 하지 않게 설계하고, `node --expose-gc`로 실행해 벤치 케이스 사이에 GC를 유도한다. 할당 자체가 측정 대상인 시나리오(S2 삽입, S4 성장)는 그 사실을 리포트에 명시한다.
- **DCE 방지**: 순회·조회 결과를 반드시 소비한다(합계를 외부로 반환하거나 mitata의 최적화 방어 장치를 사용).

## Part A — 구현과 계측

목표: 측정 대상을 내부가 보이는 상태로 만든다. 내장 자료구조는 동작을 관찰할 수 없으므로, 직접 구현하고 **계측 카운터를 공개 API로 노출한다**. 이 카운터가 Part B에서 "왜 갈렸는가"의 근거가 된다.

각 구조는 `node:test` 단위 테스트로 검증한다. 기본 동작뿐 아니라 경계(빈 구조, 용량 경계 직전·직후, 중복 키, 삭제 후 재삽입)를 포함하고, 무작위 연산 시퀀스를 내장 구조(`Array`, `Map`, 정렬 후 비교)와 대조하는 테스트를 하나 이상 둔다.

### A-1. 동적 배열

- `push`, `get`, `set`, `insertAt`, `removeAt`. 백킹 저장소는 고정 크기로 할당하고 가득 차면 재할당한다(내장 배열의 자동 성장에 얹으면 측정 대상이 사라진다 — `capacity`를 지정한 `new Array(capacity)` 또는 TypedArray를 백킹으로 쓰고 `length`를 직접 관리한다).
- **성장 계수를 생성자 인자로** 받는다(1.5와 2를 실험한다).
- 계측: 재할당 횟수, 복사된 원소 총수, 현재 `capacity`/`length`.

### A-2. 단일 연결 리스트

- `pushBack`, `insertAfter(node, value)`, 순회 이터레이터. S1·S2에서 배열의 비교 대상이 된다.
- **노드 배치를 통제하는 생성 모드 두 가지**를 만든다: 순회 순서대로 노드를 할당하는 순차 생성과, 노드를 모두 만든 뒤 무작위 순서로 연결하는 무작위 생성. [1.1 실측 1](../../docs/ch-1/01-complexity-analysis.md)이 보인 대로 할당 순서가 실측을 가르므로, 두 모드의 차이 자체가 관찰 대상이다.

### A-3. 해시 맵 — 체이닝과 오픈 어드레싱

- 같은 인터페이스(`get`, `set`, `delete`, `size`)로 **체이닝 버전과 오픈 어드레싱(선형 탐사) 버전**을 각각 구현한다. 오픈 어드레싱은 삭제 처리(tombstone)를 포함한다.
- **load factor 리사이징 임계값을 생성자 인자로** 받는다. 문자열 키에 대한 해시 함수(FNV-1a 수준)는 직접 작성한다.
- 계측: 충돌 횟수(체이닝은 체인 길이 분포, 오픈 어드레싱은 탐사 길이 분포), 리사이징 횟수와 발생 시점(몇 번째 삽입이었는지).

### A-4. 이진 힙

- 배열 표현 최소 힙([1.2의 힙 절](../../docs/ch-1/02-data-structures-in-memory.md) 전제). `insert`, `extractMin`, `peek`.
- 비교 대상으로 **정렬 유지 배열 우선순위 큐**(삽입 시 이진 탐색 + splice, 추출은 끝에서 pop)를 같은 인터페이스로 구현한다.

### A-5. (선택) 그래프

- 인접 리스트 그래프에 위상 정렬(사이클 감지 포함)과 다익스트라를 구현한다. 다익스트라의 우선순위 큐는 **A-4의 힙을 그대로 사용**한다([1.4](../../docs/ch-1/04-graph-theory-and-algorithms.md)의 알고리즘, [1.3](../../docs/ch-1/03-algorithm-design-paradigms.md)의 그리디 성립 조건 전제).
- 측정보다 구현·검증 중심이다: 사이클이 있는 입력, 음수 간선 입력(다익스트라가 틀린 답을 내는 것을 테스트로 확인)을 포함한다.

## Part B — 측정 시나리오

각 시나리오는 같은 절차를 따른다: **① 예측 표 작성**(점근 복잡도, 예상 승자와 배율의 자릿수, 그렇게 예상한 근거 한 줄) → **② 측정** → **③ 대조**(예측이 맞은 것/갈린 것, 갈린 원인을 계측 카운터·접근 패턴으로 설명). 예측 표는 측정 전에 커밋하거나 리포트에 "측정 전 작성"임을 명시한다.

### S1. 순차 순회 — 배열 vs 연결 리스트

- 같은 N(10⁵ ~ 10⁶)으로 동적 배열 순회, 연결 리스트(순차 생성) 순회, 연결 리스트(무작위 생성) 순회의 합계 계산을 mitata로 비교한다.
- 셋 다 Θ(n)이다. [1.1 실측 1](../../docs/ch-1/01-complexity-analysis.md)의 구도(순차 ≫ 포인터 추적, 할당 순서 개입)가 **자기 구현에서도** 재현되는지, 배율은 문서의 수치와 어떻게 다른지 기록한다.

### S2. 임의 위치 삽입 — 배열 vs 연결 리스트

- 무작위 위치에 원소를 삽입하는 작업을 N을 키우며(10³ ~ 10⁵) 측정한다. 연결 리스트는 삽입 위치까지 탐색하는 비용을 포함한다(위치를 미리 알고 있다는 가정은 실무에서 성립하지 않는다 — [1.2의 통념 검증](../../docs/ch-1/02-data-structures-in-memory.md) 전제).
- Stroustrup의 "연결 리스트를 피하라" 실험의 재현이다. 예측 표에는 "삽입 자체는 리스트가 O(1), 배열이 O(n)"과 "탐색 포함 총비용" 두 줄을 나눠 쓰고, 실측이 어느 쪽 예측을 따라가는지 본다.

### S3. 해시 맵 — load factor 곡선과 리사이징 스파이크

- **곡선**: 리사이징 임계값을 0.5 / 0.75 / 0.9 등으로 바꿔 가며, 고정 크기 맵에 대한 조회·삽입 단가를 체이닝·오픈 어드레싱 각각 mitata로 측정한다. 계측이 남긴 체인·탐사 길이 분포로 곡선의 모양을 설명한다. load factor가 1에 접근할 때 두 방식의 퇴화 양상 차이를 기록한다.
- **스파이크**: N회 연속 삽입에서 **삽입마다** 소요 시간을 자체 계측으로 기록해 시계열로 그린다(로그 스케일 권장). 스파이크 발생 지점이 계측 카운터의 리사이징 시점과 일치하는지 교차 확인한다. 같은 데이터로 중앙값·평균·p99·최대값을 함께 내서, 집계 통계가 스파이크를 어떻게 숨기는지([1.1의 amortized와 꼬리 지연](../../docs/ch-1/01-complexity-analysis.md)) 수치로 보인다.
- **대조군**: 내장 `Map`에 같은 삽입 시퀀스를 넣어 스파이크 시계열을 비교한다. 내부 계측은 없지만 스파이크 간격의 패턴(기하급수적으로 벌어지는지)으로 리사이징 정책을 추정해 본다.

### S4. 동적 배열 — 성장 계수 1.5 vs 2

- 같은 N(10⁶ 수준)을 push할 때 성장 계수별로 재할당 횟수, 복사된 원소 총수, 총 소요 시간, 종료 시점의 `capacity`/`length` 비율(공간 낭비)을 계측 카운터로 비교한다.
- 예측 표에는 [1.1의 amortized 분석](../../docs/ch-1/01-complexity-analysis.md)에서 계산한 총 복사량의 근사치를 먼저 적고 계측값과 대조한다. "둘 다 amortized O(1)인데 무엇이 달라지는가"를 시간·공간 트레이드오프로 서술한다.

### S5. 우선순위 큐 — 이진 힙 vs 정렬 배열

- 삽입·추출이 섞인 워크로드(예: 삽입 후 전부 추출, 삽입·추출 교차)로 두 구현을 mitata로 비교한다. N을 10² ~ 10⁵로 바꿔 가며 교차점이 있는지 본다.
- 예측 표: 힙은 삽입·추출 모두 O(log n), 정렬 배열은 삽입 O(n)(원소 이동)·추출 O(1). 작은 N에서 연속 메모리(정렬 배열)가 이기는 구간이 있는지가 관찰 포인트다.

1. **실행 환경**: CPU, OS, Node.js 버전, 실행 옵션(`--expose-gc` 등), 측정 시 머신 상태.
2. **시나리오별 예측·실측 대조**: 각 시나리오의 예측 표(측정 전 작성분), 실측 표·곡선, 대조 서술. S3의 스파이크 시계열과 집계 통계 비교를 포함한다.
3. **갈린 지점의 원인 분석**: 예측과 실측이 갈린 지점마다 계측 카운터(재할당·복사량·충돌·리사이징 시점)와 접근 패턴(순차 vs 포인터 추적)을 근거로 원인을 서술한다. "캐시 때문"이라고 쓸 때는 어떤 관찰이 그 결론을 지지하는지 함께 쓴다(하드웨어 메커니즘의 상세 규명은 챕터 7에서 다루므로, 여기서는 현상과 근거의 연결까지만).
4. **결론**: "이 데이터 크기, 이 접근 패턴, 이 런타임에서 무엇을 선택하는가"를 [1.1의 분석·측정 분업](../../docs/ch-1/01-complexity-analysis.md) 형식으로 진술한다.

## 완료 기준

- [ ] A-1 ~ A-4 구현이 `node --test` 단위 테스트를 통과하고, 경계 케이스와 내장 구조 대조 테스트가 포함되어 있다.
- [ ] 각 구현이 계측 카운터(재할당·복사량·충돌/탐사 길이·리사이징 시점)를 노출하고, 리포트의 원인 분석이 이 수치를 인용한다.
- [ ] S1 ~ S5 각각에 측정 전 작성된 예측 표가 있고, 리포트에서 실측과 대조되어 있다.
- [ ] S3의 리사이징 스파이크가 연산 단위 시계열로 제시되고, 집계 통계(중앙값·p99·최대값)와의 괴리가 수치로 서술되어 있다.
- [ ] S4의 성장 계수 비교가 계측 카운터(재할당 횟수·총 복사량·공간 낭비)로 제시되어 있다.
- [ ] 테스트는 한 명령(`pnpm test` 또는 `node --test`), 전체 측정은 한 명령(`run-bench.sh`)으로 재현된다.
- [ ] 리포트에 실행 환경이 명시되고, 결론이 절대값이 아니라 배율·곡선·교차점으로 서술되어 있다.
- [ ] 측정이 1.1의 방법론(워밍업, DCE 방지, 반복·중앙값, GC 통제)을 지켰고, 지키지 못한 부분이 있다면 그 한계가 리포트에 적혀 있다.
