# 챕터 13 실습 — 요구사항에서 ADR까지 설계 결정 재구성

[ROADMAP](../../ROADMAP.md) 챕터 13의 산출물인 **검증 가능한 요구사항, 설계 대안 비교와 ADR을 포함한 설계 리뷰 패킷**을 작성한다. [챕터 13 기획](../../plan/ch-13.md)과 본문 4편의 판단 모델을 전제로 한다.

## 학습 목표

- 실제 시스템의 요청을 이해관계자 목표·제약·가정으로 분해한다.
- 기능 요구사항과 측정 가능한 품질 속성 시나리오를 명세한다.
- 모듈 또는 아키텍처 대안 두 개 이상을 같은 변경·품질 시나리오에서 비교한다.
- 요구사항에서 설계 근거와 ADR까지 양방향으로 추적한다.

## 대상 선택과 안전

본인이 구조와 운영 맥락을 설명할 수 있는 실제 시스템 하나를 선택한다. 신규 시스템을 상상하기보다 이미 겪은 장애, 반복 변경 또는 설계 논쟁 하나가 있는 시스템이 좋다.

회사명, 고객·직원 식별 정보, 실제 거래량, 보안 통제와 내부 URL은 익명화한다. 수치를 비율이나 규모 구간으로 바꾸더라도 설계 판단에 필요한 상대 관계와 경계 조건은 보존한다. 공개하면 안 되는 코드·다이어그램·로그를 저장소에 복사하지 않는다.

## 필수 산출물

하나의 Markdown 또는 PDF 보고서에 다음을 포함한다.

1. 문제 프레임과 시스템 컨텍스트
2. 요구사항 명세와 품질 속성 시나리오
3. 모듈·아키텍처 대안 비교
4. 위험·민감점·트레이드오프 분석
5. ADR 1~2개
6. 요구사항–설계–검증 추적표

완성된 제품 구현은 요구하지 않는다. 불확실성을 판정하는 데 필요한 작은 프로토타입, 로그 분석이나 측정은 근거로 첨부할 수 있다.

## Part A — 문제와 경계 재구성

중요한 변경 하나를 한 문장으로 적는다. “마이크로서비스로 전환한다”처럼 해법으로 시작하지 말고 관찰된 문제나 목표로 시작한다.

```text
나쁜 시작: 주문 처리를 이벤트 기반으로 바꾼다.
좋은 시작: 외부 결제사 지연 때 취소 API timeout과 중복 환불이 함께 증가한다.
```

### A-1. 이해관계자와 목표

최소 네 종류의 이해관계자를 식별하고 다음 표를 채운다.

| 이해관계자 | 원하는 결과 | 피하려는 실패 | 결정 권한·제약 | 근거 |
|---|---|---|---|---|
| 예: 고객지원 | 취소 상태를 설명 | 수동 확인에 30분 소요 | 예외 처리 권한은 제한됨 | 티켓 분류·인터뷰 |

근거는 인터뷰, 운영 데이터, 사고 기록, 계약·규정, 코드·문서 중 무엇인지 밝힌다. 단순한 추측은 가정으로 표시한다.

### A-2. 시스템 컨텍스트

대상 시스템, 사람·외부 시스템과 입출력 경계를 그린다. 표기법은 자유지만 각 연결에는 입력, 출력, 성공·실패 의미와 기록의 원천을 설명한다.

### A-3. 사실·제약·가정·질문

최소 두 개의 가정을 포함해 다음 표를 작성한다.

| 항목 | 분류 | 틀렸을 때 영향 | 확인 방법·소유자 |
|---|---|---|---|
| 결제사 응답의 99%는 2초 이내다 | 가정 | 동기 설계의 p99 목표 실패 | 30일 지연 분포 분석 / 결제 담당 |

## Part B — 검증 가능한 요구사항

### B-1. 기능 요구사항

핵심 기능 요구사항 3~5개를 작성한다. 각 요구사항에 식별자, 출처·목표, 우선순위와 검증 방법을 붙인다. 정상 흐름뿐 아니라 거절, 중복, 시간 초과 같은 경계 조건을 포함한다.

```text
FR-CAN-01
출처: 고객 목표 G1, 중복 환불 사고 INC-17
요구: 인증된 고객이 같은 취소 의도를 반복 제출하면 시스템은 하나의
      취소 요청만 기록하고 모든 호출에 같은 취소 식별자와 상태를 반환한다.
검증: 동시 중복 요청 시나리오의 상태·외부 호출·환불 원장 확인
```

### B-2. 품질 속성 시나리오

서로 다른 품질 속성을 포함한 시나리오를 최소 3개 작성한다. 각 시나리오는 다음 여섯 요소를 모두 가져야 한다.

| 요소 | 내용 |
|---|---|
| source | 자극을 만드는 사람·시스템·사건 |
| stimulus | 발생한 사건 |
| environment | 정상·과부하·장애·변경 등 조건 |
| artifact | 영향을 받는 시스템·데이터·프로세스 |
| response | 기대하는 시스템 반응 |
| response measure | 시간·비율·범위·완료 조건 |

숫자마다 근거 또는 `검증이 필요한 가정` 표시를 붙인다. 평균만으로 꼬리 지연을 숨기지 않고 측정 시작·종료와 모집단을 밝힌다.

### B-3. 요구사항 검토

각 요구사항을 필요성, 단일성, 명확성, 실현 가능성, 일관성, 검증 가능성, 해법 중립성으로 검토한다. 발견한 결함과 수정 전후를 최소 두 건 제시한다.

## Part C — 모듈과 아키텍처 대안

현재안과 개선안 또는 서로 다른 개선안 두 개 이상을 제시한다. 기술 이름만 나열하지 말고 각 안에 다음을 포함한다.

- 모듈·시스템 경계와 책임
- 각 경계가 숨기는 변경 가능한 결정
- 공개 계약, 데이터 소유권과 핵심 불변식
- 동기·비동기 상호작용과 실패 의미
- 배포·운영 단위가 논리 경계와 다른 부분

최소 세 개의 변경 시나리오로 모듈 경계를 평가한다.

```text
- 외부 공급자 하나 추가
- 국가·고객군별 정책 변경
- 데이터 보존·감사 제약 변경
```

각 변경이 수정할 경계, 동시 배포, 데이터 마이그레이션과 재검증 범위를 표로 비교한다.

## Part D — 품질 속성 기반 평가

Part B의 **같은 품질 속성 시나리오**를 모든 대안에 통과시킨다. 장단점의 일반론 대신 구조의 어떤 결정이 어떤 반응을 만드는지 적는다.

| 시나리오 | 대안 A의 반응·근거 | 대안 B의 반응·근거 | 미확인 가정 |
|---|---|---|---|
| QA-RESP-01 | 외부 지연이 요청 thread를 점유 | 접수와 실행 분리, queue 지속성 필요 | 장애 중 유실 여부 |

평가 결과에서 다음을 각각 하나 이상 식별한다.

- 위험: 요구를 충족하지 못할 수 있는 구체적 우려
- 민감점: 작은 값 변화가 품질 결과를 크게 바꾸는 결정
- 트레이드오프: 둘 이상의 품질 속성에 상반된 영향을 주는 결정
- 증거 과제: 가정을 판정할 로그 분석·실험·프로토타입·walkthrough

## Part E — ADR

선택한 결정 중 영향이 크거나 되돌리기 비싼 것 1~2개를 ADR로 기록한다.

```markdown
# ADR-NNN: 결정문

## 상태
Proposed | Accepted | Rejected | Superseded

## 맥락
연결된 목표·요구사항, 현재 제약, 확인된 증거와 중요한 가정

## 결정
무엇을 선택하고 어느 범위에 적용하는가

## 검토한 대안
실제로 검토한 대안과 선택하지 않은 이유

## 결과
긍정적 결과, 부정적 결과, 새 운영·마이그레이션 책임과 남은 위험

## 재검토 조건
결정을 다시 열 측정값, 요구 변경, 실패 또는 기한
```

제품·패턴 이름을 제목으로 쓰기보다 결정한 행위를 제목으로 쓴다. `Kafka 사용`보다 `주문 취소 접수와 실행을 내구성 queue로 분리한다`가 결정의 범위와 의도를 잘 드러낸다.

## Part F — 추적성과 최종 검토

요구사항에서 설계·증거로, 설계에서 원래 근거로 이동할 수 있는 표를 만든다.

| 요구사항 | 출처·목표 | 설계 요소·ADR | 검증·증거 | 상태 |
|---|---|---|---|---|
| QA-RESP-01 | G1, INC-17 | CancellationQueue, ADR-004 | 장애 주입 실험 EXP-03 | 가정 확인 필요 |

다음 질문으로 패킷을 검토한다.

1. 어떤 요구에도 연결되지 않는 구조적 복잡성이 있는가?
2. 설계·검증에 연결되지 않은 중요한 요구가 있는가?
3. 대안들이 같은 조건에서 비교됐는가?
4. 숫자의 측정 경계와 근거가 있는가?
5. ADR의 부정적 결과와 재검토 조건이 구체적인가?

## 선택 심화

### 동료 리뷰

동료에게 시스템 설명 없이 패킷만 제공하고 숨은 이해관계자, 모호한 요구, 반례와 미평가 위험을 찾게 한다. 지적을 수용·기각한 근거를 남긴다.

### 변경 주입

동료가 다음 중 하나를 선택해 새 요구로 제시한다.

- 부분 취소 또는 다중 통화 지원
- 외부 공급자의 지연·rate limit 급변
- 개인정보 삭제·감사 보존 제약의 충돌
- 트래픽 10배 증가 또는 운영팀 축소

추적표로 영향을 받는 요구·모듈·ADR·검증을 찾고, 기존 결정을 유지할지 새 ADR로 대체할지 판단한다.

## 완료 기준

- [ ] 이해관계자 네 종류 이상과 시스템 경계, 목표·제약·가정을 근거와 함께 기록했다.
- [ ] 기능 요구사항 3~5개와 서로 다른 품질 속성 시나리오 3개 이상을 검증 가능하게 작성했다.
- [ ] 각 품질 시나리오에 source, stimulus, environment, artifact, response, response measure가 있다.
- [ ] 모듈 또는 아키텍처 대안 두 개 이상을 같은 변경·품질 시나리오에서 비교했다.
- [ ] 데이터 소유권, 불변식, 실패 의미와 운영 비용을 대안별로 설명했다.
- [ ] 위험·민감점·트레이드오프와 증거 과제를 각각 식별했다.
- [ ] ADR 1~2개에 대안, 부정적 결과와 재검토 조건을 포함했다.
- [ ] 요구사항에서 설계·검증으로, 설계에서 목표·요구사항으로 추적할 수 있다.
