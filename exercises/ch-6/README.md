# 챕터 6 실습 — mark-sweep GC 구현과 V8 힙 프로파일링

[ROADMAP](../../ROADMAP.md) 챕터 6의 산출물인 **"mark-sweep GC를 직접 구현하고, 실제 언어 런타임의 힙을 프로파일링해 GC 정책이 지연 시간에 미치는 영향을 리포트로 정리한다"**를 수행하는 실습이다. [plan/ch-6.md](../../plan/ch-6.md) §5 기획을 기준으로 하며, 챕터 6 본문 — [6.1의 스택·root 모델](../../docs/ch-6/01-runtime-systems.md), [6.2의 할당자·단편화 모델](../../docs/ch-6/02-memory-management.md), [6.3의 GC 비용 모델](../../docs/ch-6/03-garbage-collection.md) — 을 전제로 한다.

이 실습의 핵심 규율은 **같은 원리를 두 층에서 확인하는 것**이다. Part A에서 toy GC를 직접 만들어 계측하고, Part B에서 V8이라는 실전 GC를 로그·스냅샷으로 관찰한 뒤, 리포트에서 두 층의 관찰이 같은 비용 구조([6.3의 트레이드오프 지도](../../docs/ch-6/03-garbage-collection.md))를 가리키는지 대조한다. toy에서 본 것이 실전에서 재현되지 않으면, 어느 쪽 모델이 틀렸는지 설명해야 한다.

## 학습 목표

- 도달성 기반 회수(mark-sweep)를 직접 구현하고, mark·sweep 각각의 비용이 무엇에 비례하는지 자기 계측으로 입증한다.
- 순환 참조 회수(참조 계수와의 차이), root 보존, 힙 가득 참 → 수집 → 재할당의 정확성을 테스트로 검증한다.
- free list 조각 수·최대 구멍 크기를 계측해 [6.2의 외부 단편화](../../docs/ch-6/02-memory-management.md)를 수치로 관찰한다.
- V8의 `--trace-gc` 로그와 힙 설정 변경 실험으로 할당 패턴·힙 크기가 일시정지(p50/p99)와 처리량에 미치는 영향을 곡선으로 기록한다.
- 의도적 leak을 심고 힙 스냅샷 2장 비교로 retainer 경로를 찾는 절차를 수행한다.

## 실행 환경

**TypeScript + Node.js 24**로 작성한다(저장소 도구 체인과 동일, [mise.toml](../../mise.toml) 기준). `exercises/ch-6/`을 pnpm 워크스페이스 패키지로 만든다 — 저장소 루트의 `pnpm-workspace.yaml`에 `packages` 목록으로 `exercises/ch-6`을 등록하고, 이 디렉터리에 `private: true`인 `package.json`을 만든다. Node.js 24는 타입 스트리핑으로 `.ts`를 직접 실행하므로 빌드 없이 `node`로 테스트·벤치를 돌린다(enum·namespace 같은 비삭제 문법 금지). **외부 런타임 의존성은 두지 않는다** — 테스트는 `node:test`, 시간 측정은 `node:perf_hooks`로 충분하다.

plan §5에서 미결이던 두 가지를 다음과 같이 확정한다.

- **지연 히스토그램: `perf_hooks`의 `createHistogram()`을 사용한다.** Node 내장 HDR 히스토그램으로 p50/p99/max를 바로 얻을 수 있고, 의존성이 없으며, 직접 구현한 고정 버킷보다 꼬리 해상도가 좋다. 요청마다 `performance.now()` 차이를 `histogram.record()`(정수 ns/µs 단위로 환산)에 기록한다. 이벤트 루프 지연의 교차 검증에는 `monitorEventLoopDelay()`를 쓴다.
- **힙 워드와 객체 표현: 32비트 워드(`Uint32Array`) 힙, 주소는 워드 인덱스.** 객체는 `[헤더 1워드][참조 슬롯 N워드]` — 헤더의 하위 24비트에 슬롯 수 N, 최상위 비트(bit 31)에 mark bit를 둔다. 슬롯 값은 다른 객체의 워드 인덱스이고 0은 null이다(인덱스 0은 할당에 쓰지 않는다). 페이로드(숫자·문자열)와 타입 태그는 두지 않는다 — 도달성 계산에 필요한 최소 구조만 남겨 GC 자체에 집중한다.

## Part A — mark-sweep GC 구현

목표: [6.3의 일반 모델](../../docs/ch-6/03-garbage-collection.md)을 실행 가능한 최소 구현으로 만들고, 비용이 어디에 비례하는지 계측이 보이게 한다.

### A-1. toy 힙과 할당

- `ArrayBuffer` 위의 `Uint32Array`를 힙으로 쓰는 `Heap` 클래스를 만든다. 생성자 인자: 힙 크기(워드 수). root set은 명시적인 root 배열(가상의 스택·전역, [6.1의 root 모델](../../docs/ch-6/01-runtime-systems.md)에 대응)로 표현하고 `addRoot`/`removeRoot`로 조작한다.
- `allocate(slotCount): address` — **free list 기반**으로 구현한다(first-fit, [6.2의 splitting·coalescing](../../docs/ch-6/02-memory-management.md) 포함). 공간이 부족하면 `collect()`를 트리거하고, 수집 후에도 부족하면 명시적 오류를 던진다.
- 슬롯 읽기/쓰기 API(`getSlot`, `setSlot`)를 제공한다. 이후 확장(write barrier 실험)을 위해 참조 쓰기가 한 함수를 지나게 설계한다.

### A-2. mark — 명시적 스택 기반 탐색

- root 배열에서 출발해 참조 슬롯을 따라가며 헤더의 mark bit를 세운다. **재귀 금지** — 명시적 스택(배열)을 쓴다. 이유는 두 가지이고 둘 다 본문에서 다뤘다: 깊은 참조 사슬에서 [6.1의 스택 오버플로](../../docs/ch-6/01-runtime-systems.md)를 피하는 것, 그리고 mark가 [1.4의 그래프 탐색](../../docs/ch-1/04-graph-theory-and-algorithms.md) 그 자체임을 구현으로 확인하는 것.
- 이미 마크된 객체는 다시 넣지 않는다(방문 표시가 mark bit 그 자체다). 순환 그래프에서 종료함을 테스트로 확인한다.

### A-3. sweep — 힙 전체 순회

- 힙을 처음부터 끝까지 블록 단위로 걸으며, 마크 안 된 블록을 free list에 반환하고(인접 free 블록과 병합), 마크된 블록의 bit를 지운다.
- sweep이 끝난 힙에서 free list가 정합적인지(블록 겹침 없음, 전체 워드 수 보존) 검사하는 불변식 체크 함수를 만들어 테스트에서 매 수집 후 호출한다.

### A-4. 계측 인터페이스

각 `collect()`마다 다음을 기록해 반환한다. 이 수치들이 Part B와의 대조 및 리포트의 원자료다.

| 항목 | 입증하는 것 |
|------|-------------|
| mark 시간, 방문 객체 수 | mark 비용 ∝ 살아 있는 객체·참조 수 |
| sweep 시간 | sweep 비용 ∝ 힙 크기(생존자 수와 무관) |
| 생존 객체 수·워드, 회수 워드 | 워크로드의 생존율 |
| free list 조각 수, 최대 구멍 크기 | 외부 단편화([6.2 관찰 1](../../docs/ch-6/02-memory-management.md)의 지표) |

계측 검증 워크로드 두 개를 `bench/part-a-workloads.ts`에 만든다: ① 생존자 고정·힙 크기 증가 시리즈(sweep 시간만 늘어나는가), ② 힙 크기 고정·생존자 증가 시리즈(mark 시간만 늘어나는가). 결과 표를 리포트에 싣는다.

### A-5. 검증 시나리오 (`node:test`)

전부 단위 테스트로 작성한다.

1. **도달 불가 회수**: root에서 끊긴 객체가 수집 후 free list로 돌아온다.
2. **순환 참조 회수**: 서로 가리키는 두 객체(및 자기 참조)가 root에서 끊기면 회수된다 — 참조 계수와의 차이를 입증하는 테스트임을 주석으로 명시한다.
3. **도달 가능 보존**: 깊은 사슬·다이아몬드·순환을 포함한 그래프에서 root 도달 객체는 수집 후에도 슬롯 값이 보존된다.
4. **힙 가득 참 → collect → 재할당 성공**: 가득 채우고 일부 root를 끊은 뒤 재할당이 성공한다. 전부 도달 가능하면 명시적 오류가 난다.
5. **불변식**: 모든 시나리오에서 A-3의 free list 정합성 검사를 통과한다.

### A-6. (선택 확장) 두 번째 회수 전략

mark-compact(산 객체를 앞으로 밀고 참조 갱신) 또는 2-공간 copying(Cheney) 중 하나를 같은 인터페이스로 추가하고, A-4의 ①·② 시리즈와 단편화 지표를 mark-sweep과 비교한다 — copying/compact에서 "최대 구멍" 지표가 왜 무의미해지는지([6.2 확인 문제 2](../../docs/ch-6/02-memory-management.md)의 답)를 수치로 확인할 수 있다.

## Part B — V8/Node.js 힙 프로파일링

목표: Part A에서 만든 비용 모델이 실전 런타임에서 재현되는지 확인하고, 힙 설정과 할당 패턴이 지연 분포에 미치는 영향을 곡선으로 기록한다. 측정 규율은 [챕터 1 실습](../ch-1/README.md)과 같다 — 실행 환경(CPU·OS·Node 버전)을 리포트에 명시하고, 결론은 절대값이 아니라 배율과 곡선의 모양으로 서술하며, 측정 중 다른 부하를 돌리지 않는다.

### B-1. 워크로드 — 가짜 서버

`bench/part-b-server.ts`에 네트워크 없는 인메모리 요청 처리 루프를 만든다. "요청" 하나는 객체 그래프 생성 + 가공 + 응답 객체 반환이고, 요청당 지연을 `perf_hooks` 히스토그램에 기록한다. 할당 패턴을 두 축의 파라미터로 조절할 수 있어야 한다.

- **단수명 비율**: 요청 안에서 만들고 버리는 임시 객체 수.
- **장수명 유지 집합**: 전역 캐시에 유지하는 객체 수(고정 크기로 순환 교체 — 중간 수명 객체를 만드는 축이기도 하다).

주의: 임시 객체가 escape analysis로 제거되지 않도록 [6.3 관찰 1](../../docs/ch-6/03-garbage-collection.md)의 링 버퍼 기법을 쓰고, 응답 값을 소비(합산)해 DCE를 막는다.

### B-2. 측정 시나리오

각 시나리오는 `run-bench.sh`에서 한 명령으로 재현되어야 하고, 결과(히스토그램 백분위 + `--trace-gc` 요약)를 표로 리포트에 싣는다.

1. **세대 가설 확인**: 단수명 위주 vs 장수명 유지 위주 두 설정을 `--trace-gc`로 실행해 Scavenge/Mark-Compact의 빈도·일시정지 분포를 비교한다. 기대 모양은 [6.3 관찰 1](../../docs/ch-6/03-garbage-collection.md)의 표와 같다 — 다르게 나오면 원인을 로그의 생존량으로 추적한다.
2. **힙 설정 트레이드오프 곡선**: young 세대 크기(`--min-semi-space-size`를 4/16/64MB 등으로)와 old 상한(`--max-old-space-size`)을 바꿔 가며 p50/p99 지연·처리량(req/s)·GC 총 시간을 기록하고, "빈도 vs 회당 비용 vs 메모리 점유"의 교환 곡선을 그린다. 본문 실측에서 확인했듯 `--max-semi-space-size`(상한)만 키우는 것은 효과가 없을 수 있다 — 효과가 없는 설정도 "왜 없는지"와 함께 기록한다(플래그가 묶는 자원이 병목인지 판별하는 연습이다).
3. **leak 탐지 절차**: `bench/part-b-leak.ts`에 의도적 leak(전역 캐시에 요청 잔여물 누적)을 심고, `v8.writeHeapSnapshot()`으로 부하 전후 스냅샷 2장을 떠 DevTools 비교 뷰에서 증가 생성자 → retainer 경로 → 원인 코드 행까지 도달하는 과정을 스크린샷 또는 경로 서술로 리포트에 남긴다([6.3 실무 관점](../../docs/ch-6/03-garbage-collection.md)의 절차).

## 리포트 (`report.md`)

다음 구성을 갖춘다.

1. **실행 환경** — CPU, OS, Node 버전, 전원 상태.
2. **Part A 계측 표** — A-4의 두 시리즈: mark ∝ 생존자, sweep ∝ 힙 크기가 수치로 성립하는가. 단편화 지표의 추이.
3. **Part B 결과** — 시나리오별 표·곡선과 해석. 할당 패턴·힙 설정이 일시정지에 미친 영향을 "무엇에 비례했는가"로 서술한다.
4. **두 층의 대조** — toy GC 계측과 V8 로그에서 **공통으로 확인된 원리**(예: 수집 비용의 비례 대상, 생존율이 일시정지를 지배)와, toy에는 없어서 V8에서만 보인 것(세대, 동시 marking, 승격)을 구분해 정리한다.
