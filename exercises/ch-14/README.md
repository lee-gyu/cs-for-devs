# 챕터 14 실습 — 테스트 하네스 구축과 SLO·에러 버짓 설계

[ROADMAP](../../ROADMAP.md) 챕터 14의 산출물인 **테스트 하네스 구축 기록과 SLO·에러 버짓 설계 리포트**를 작성한다. [챕터 14 기획](../../plan/ch-14.md)과 본문 4편의 판단 모델을 전제로 한다.

두 부로 구성된다. Part A는 저장소가 제공하는 레거시 주문 확정 서비스에 특성화 테스트 → seam → 테스트 대역 → 속성 기반 테스트 순으로 하네스를 구축하고, 각 단계에서 **어떤 위험에 대한 증거가 생겼고 무엇이 여전히 검증되지 않았는지** 기록한다. Part B는 제공된 스크립트로 30일치 요청 로그를 생성해 SLI 계산 → SLO·에러 버짓 정의 → burn rate 경보 계산 → 가상 배포의 진행/중단 판단까지 수행한다.

## 학습 목표

- 테스트 없는 코드에 특성화 테스트와 seam으로 안전한 변경 경로를 구축한다.
- 각 테스트 단계가 사는 증거와 남는 위험의 경계를 기록한다.
- 예제 기반 테스트가 놓치는 결함을 속성 기반 테스트의 불변식으로 표현한다.
- 요청 로그에서 SLI·에러 버짓·burn rate를 계산하고 경보 발화 시점을 판정한다.
- 에러 버짓 정책을 정의하고 가상 배포 시나리오의 진행/중단을 근거와 함께 판단한다.

## 환경

- Node 24 이상 (TypeScript 파일을 타입 제거 방식으로 직접 실행한다)
- 테스트 러너는 의존성이 가장 적은 `node:test`를 기본으로 한다. 속성 기반 테스트 단계에서만 fast-check를 설치한다.
- 외부 계측 도구 없이 로컬에서 전부 재현 가능하다.

## 산출물

하나의 Markdown 보고서(또는 두 개의 파트별 문서)에 다음을 포함한다.

1. Part A: 단계별 테스트 코드와 **증거·경계 기록표** (단계 / 생긴 증거 / 여전히 검증되지 않는 것 / 발견한 동작·결함)
2. Part B: SLO 문서(SLI 정의, 목표, 윈도우, 측정 범위 밖 명시)와 판단 리포트(예산 소진 분석, 경보 발화 판정, 배포 진행/중단 결정과 근거)

## Part A — 레거시 서비스에 테스트 하네스 구축

시작점은 `legacy-order-service/`다. 시간(심야 할증), 전역 상태(재고·주문 목록), 외부 호출(결제 API)이 의도적으로 얽혀 있다.

```bash
cd legacy-order-service
node src/payment-stub.ts          # 터미널 1: 결제 스텁 서버
node src/demo.ts                  # 터미널 2: 현재 동작 관찰
```

### A-0. 실행 관찰

demo의 출력을 그대로 기록한다. 빈 주문과 음수 수량 주문의 결과, 재고 부족으로 거절된 주문 전후의 재고 상태를 특히 주의해서 본다. 이 관찰이 특성화 테스트의 재료다.

### A-1. 특성화 테스트

`confirmOrder`의 **현재 동작**을 기록하는 테스트를 작성한다. 이 단계에서 제품 코드는 수정하지 않는다. 결제 스텁이 떠 있어야 테스트가 도는 상태, 테스트 간 재고 공유, 실행 시각에 따라 결과가 달라지는 문제를 전부 그대로 겪는 것이 의도다.

- 정상 주문, 빈 주문, 음수 수량, 재고 부족(단일 품목), 재고 부족(두 번째 품목에서 실패) 시나리오를 포함한다.
- 이상해 보이는 동작도 판단 없이 기록한다. "고치고 싶은 것"은 테스트 이름이나 주석에 표시만 해 둔다.
- 기록: 이 테스트들이 시간대·실행 순서에 따라 흔들리는지 확인하고([14.1](../../docs/ch-14/01-testing.md)의 flaky 구조 세 가지 중 무엇에 해당하는지) 남긴다.

### A-2. seam 도입

시계·결제 호출·재고 저장소를 주입 가능한 의존성으로 끌어올린다. 기본값이 기존 동작을 보존해야 하며, **A-1의 특성화 테스트가 전부 그대로 통과하는 것**이 이 단계의 완료 조건이다. seam 도입에 리팩터링 욕심을 섞지 않는다.

### A-3. 테스트 대역 교체

주입 지점에 가짜 시계, 페이크 결제, 격리된 재고를 넣어 테스트를 결정적으로 만든다.

- 심야 할증을 고정 시각으로 검증한다 (21:59 / 22:00 / 05:59 / 06:00 경계).
- 결제 실패(402) 시의 동작을 관찰한다. 재고가 복구되는가? 이 발견을 기록한다.
- 결제 스텁 서버 없이 전체 스위트가 돌고, 반복 실행해도 같은 판정이 나오는지 확인한다.
- 기록: 대역 교체로 **잃은** 증거는 무엇인가? (실제 결제 API와의 계약은 이제 무엇이 검증하는가?)

### A-4. 안전한 변경과 속성 기반 테스트

하네스 위에서 첫 실제 변경을 가한다. **멱등성 키 지원** — 같은 `idempotencyKey`로 재호출하면 새 주문을 만들지 않고 같은 결과를 반환한다.

```bash
npm install --save-dev fast-check   # 또는 pnpm add -D fast-check
```

- 변경 전에 실패하는 테스트를 먼저 쓴다 (같은 키 두 번 → 주문 1건).
- fast-check로 최소 두 개의 속성을 작성한다. 예: "임의 품목 배열에서 총액은 부분합의 합이다", "같은 키의 반복 호출 횟수와 무관하게 확정 주문 수는 1이고 재고는 1회분만 차감된다".
- 속성이 반례를 찾으면 shrinking이 제시한 최소 반례를 기록한다.
- 기록: 속성 테스트가 잡은 것과, 속성으로 표현하지 않아 여전히 미검증인 것.

### A-5. (선택 심화) 검증 강도 감사

`node --test --experimental-test-coverage`로 커버리지를 확인한 뒤, 제품 코드에 수동 뮤턴트(`+`→`-`, `<`→`<=` 등)를 하나씩 심어 스위트가 죽이는지 확인한다. 살아남은 뮤턴트가 곧 단언 보강 지점이다. Stryker 자동화까지 확장해도 좋다. 자신의 실무 코드로 A-1~A-4를 반복하는 것도 심화 트랙으로 허용된다.

## Part B — SLO·에러 버짓 시뮬레이션

시작점은 `slo-simulation/`이다. 제공된 생성기가 주문 확정 API의 30일치 분 단위 요청 로그를 만든다. 시드가 고정되어 있어 모두 같은 로그를 얻는다.

```bash
cd slo-simulation
node generate-logs.ts             # requests.ndjson 생성 (분당 1줄, 43,200줄)
```

각 줄은 1분 집계다: `requests`(유효 요청 수), `errors`(5xx 응답 수), `slow`(성공했지만 800ms 초과 응답 수). 분석 스크립트는 직접 작성한다(언어 자유, Node 권장). 생성기 안의 시나리오 정의를 먼저 읽으면 아래 B-2가 무의미해지므로 분석을 끝낸 뒤에 답을 맞춰 보길 권한다.

### B-1. SLI 정의와 실측

가용성 SLI(오류가 아닌 응답 비율)와 지연 SLI(800ms 안에 응답한 비율)를 좋은 이벤트/유효 이벤트 형태로 정의하고, 30일 전체와 일별 실측값을 계산한다. 이 로그가 측정하지 못하는 것(클라이언트 측 실패, 서버 미도달)을 SLI 문서에 명시한다.

### B-2. 장애 시나리오 식별

일별·시간별 SLI를 훑어 이상 구간을 모두 찾고, 각각의 시작·종료 시각, 성격(급격한 전면 장애 / 지연 저하 / 저강도 지속 오류), 소진한 예산량을 기록한다. 로그에는 성격이 다른 이상 구간이 네 개 주입되어 있다.

### B-3. SLO와 에러 버짓 정의

실측을 근거로 두 SLI 각각의 SLO(목표율, 30일 윈도우)를 선택하고 근거를 적는다. "장애가 없던 기간의 실측은 여유 있게 달성하지만, 주입된 장애를 포함하면 위반이거나 아슬아슬한 수준"의 목표가 판단 연습에 적당하다. 선택한 SLO로 에러 버짓을 요청 수와 시간 양쪽 단위로 환산한다.

### B-4. burn rate 경보 판정

[14.3](../../docs/ch-14/03-reliability-engineering.md)의 경보 설계(14.4배/1시간, 6배/6시간 → 즉시 호출, 1배/3일 → 티켓, 짧은 보조 윈도우는 긴 윈도우의 1/12)를 로그에 적용해, 30일 동안 각 경보가 **언제 발화하고 언제 해소되는지** 계산한다.

- 각 이상 구간을 어떤 경보가 얼마나 빨리 잡았는가?
- 즉시 호출 경보에 안 걸리고 예산만 태운 구간이 있는가? 그 구간은 어떻게 발견되어야 했는가?
- 보조 윈도우가 없을 때 경보가 얼마나 오래 잔향으로 울리는지 비교한다.

### B-5. 배포 판단 리포트

에러 버짓 정책을 정의하고(잔여 예산 구간별 행동, 동결 해제 권한 포함), 다음 가상 시나리오를 판정한다.

1. 로그 마지막 날(30일차) 기준 잔여 예산을 계산하라. 정책상 어느 구간인가?
2. 26일차의 카나리 배포 구간을 분석하라. 카나리가 아니라 전체 배포였다면 같은 결함이 소진했을 예산을 계산해 카나리의 효과를 수치로 보여라.
3. 31일차에 "주문 확정 경로를 바꾸는 큰 배포"가 계획되어 있다. 잔여 예산과 정책을 근거로 진행/조건부 진행/중단을 결정하고, 조건부라면 조건(카나리 단계, 관찰 시간, 자동 롤백 기준)을 구체적으로 적어라.
